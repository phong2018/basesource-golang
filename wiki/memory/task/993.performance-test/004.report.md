## 4. Bottleneck Analysis

### Primary Bottleneck: MySQL Connection Pool Exhaustion

| MySQL Stat                          | Value   |
|-------------------------------------|---------|
| Max connections ever used           | **152** |
| MySQL default max_connections       | 151     |
| Connection errors (max connections) | **770,392** |

MySQL hit its default `max_connections=151` limit during the write-heavy and stress tests. Any request arriving when all 151 connections are occupied gets an immediate connection error — this explains the high error rates even at 500 VUs, well before the API pods themselves were overwhelmed.

The Go app's connection pool (sqlx default) likely opened too many connections under concurrent load. Each API pod opens its own pool; with 7 pods × ~25 connections = 175 — over the MySQL limit.

### Secondary Bottleneck: Outbox Dual-Write Transaction Cost

Every POST/PUT/DELETE writes 3 rows in a single MySQL transaction:
1. `todos` table
2. `outbox_events` table  
3. `outbox_deliveries` table (one row per destination: kafka + rabbitmq = 2 rows)

This is 3–4× heavier than a plain single-table write. Under high write load this lengthens transaction hold time, increasing connection contention.

### API Pods: Not the Bottleneck

| Pod                | CPU after stress test |
|--------------------|----------------------|
| basesource-api (active) | 502m / 500m limit |
| mysql              | **511m** (high)      |
| Other API pods     | ~1m (idle after test) |

API pod CPU was within limits. MySQL CPU was equally high. The connection error count (770k) confirms MySQL connections — not CPU — was the wall.

### Worker Pods: Unaffected

Worker pods (KafkaOutboxRelay, RabbitMQOutboxRelay, consumers) stayed at 5–7m CPU throughout. The outbox relay processes are intentionally async and separate from the API request path — they absorbed the backlog from the outbox tables after the test without issue.

## 6. Recommendations

### Fix 1 — Increase MySQL max_connections (immediate, low risk)

```yaml
# k8s/infra/mysql.yaml — add env var
env:
  - name: MYSQL_MAX_CONNECTIONS
    value: "500"
```

Or via MySQL config:
```sql
SET GLOBAL max_connections = 500;
```

Expected impact: removes the hard connection ceiling, allows more concurrent queries.

### Fix 2 — Cap the Go DB connection pool per pod (important)

In the application's DB init code, set explicit pool limits so N pods × pool_size stays under MySQL max:

```go
db.SetMaxOpenConns(20)      // max 20 connections per pod
db.SetMaxIdleConns(5)
db.SetConnMaxLifetime(5 * time.Minute)
```

With 10 API pods × 20 conns = 200 — safely under MySQL's 500 if Fix 1 is applied.

### Fix 3 — Add a DB index on outbox_deliveries (performance)

The relay polls `outbox_deliveries WHERE status='pending'` frequently. Ensure there is an index:

```sql
CREATE INDEX idx_outbox_deliveries_status ON outbox_deliveries(status, destination);
```

### Fix 4 — Tune HPA for faster response

The current HPA reacts in ~30s by default. For spike tolerance, reduce the stabilization window:

```yaml
behavior:
  scaleUp:
    stabilizationWindowSeconds: 0     # scale up immediately
    policies:
      - type: Pods
        value: 2
        periodSeconds: 15
```

## 7. Conclusion

**The bottleneck is MySQL connection pool saturation, not API CPU.**

The basesource API handles ~350 RPS cleanly (P95 < 400ms, 0% errors) on a local kind cluster with 2–7 API pods. Beyond ~300 VUs, MySQL connection exhaustion (default limit: 151) causes cascading failures for both reads and writes.

The outbox dual-write pattern (3–4 rows per write request) is the multiplier — it makes write operations ~3× heavier on DB connections than plain CRUD, and the relay polling adds background read pressure on the same `outbox_deliveries` table.

With `max_connections` raised and connection pool caps applied per pod, the system should handle 2–3× current load before hitting the next bottleneck (likely MySQL CPU or disk I/O).
