# Performance Testing Guide

## Overview

This document explains:

* Step-by-step performance testing process
* Tools used during testing
* Important performance testing concepts
* Infrastructure monitoring and bottleneck analysis
* Real-world API load testing workflow

---

# 1. What is Performance Testing?

Performance testing is the process of evaluating:

* how fast a system responds
* how many requests it can handle
* how stable it is under heavy traffic
* where bottlenecks exist

The goal is to verify:

* scalability
* stability
* reliability
* response time
* system limits

---

# 2. Types of Performance Testing

| Type                | Purpose                                |
| ------------------- | -------------------------------------- |
| Load Testing        | Verify behavior under expected traffic |
| Stress Testing      | Push system beyond normal capacity     |
| Spike Testing       | Test sudden traffic increase           |
| Scalability Testing | Verify scaling capability              |
| Endurance Testing   | Test long-running stability            |

---

# 3. Step-by-Step Performance Testing Process

## Step 1 — Define Testing Goals

First, define:

* API or service to test
* expected traffic
* acceptable response time
* success criteria

Example:

* Endpoint:
  * `POST /api/v1/ec1/member-card`
* Target traffic:
  * 10,000 req/min
  * 20,000 req/min
  * 30,000 req/min
  * 50,000 req/min
* Success criteria:
  * response time < 3 seconds
  * success rate > 95%
  * no major timeout

---

## Step 2 — Prepare Testing Environment

Prepare infrastructure similar to production.

Example:

* AWS ECS
* Multiple ECS tasks
* External API integration
* CloudWatch monitoring

Infrastructure setup:

| Component     | Purpose                    |
| ------------- | -------------------------- |
| ECS           | Run application containers |
| CloudWatch    | Monitor infrastructure     |
| Logs Insights | Analyze application logs   |
| External API  | Simulate real dependencies |

---

## Step 3 — Prepare Load Testing Tool

Initially, a simple bash script may be used.

Example:

```bash
curl -X POST https://api.example.com/member-card
```

However, bash scripts have limitations:

* connection overhead
* port exhaustion
* no HTTP Keep-Alive optimization

Therefore, professional tools such as k6 are preferred.

---

## Step 4 — Configure k6 Load Testing

Example k6 script:

```javascript
import http from 'k6/http';
import { sleep } from 'k6';

export default function () {
  http.post('https://api.example.com/member-card');
  sleep(1);
}
```

Test parameters:

| Parameter | Meaning                |
| --------- | ---------------------- |
| VUs       | Virtual users          |
| Duration  | Test duration          |
| Sleep     | Delay between requests |
| Threshold | Pass/fail criteria     |

Example:

* 100 VUs
* 300 VUs
* 3-minute duration

---

## Step 5 — Execute Load Tests

Run tests gradually.

| Stage    | Load           |
| -------- | -------------- |
| Baseline | 10,000 req/min |
| Medium   | 20,000 req/min |
| High     | 30,000 req/min |
| Stress   | 50,000 req/min |

Purpose:

* identify safe operating range
* detect system limits
* analyze scaling behavior

---

## Step 6 — Collect Metrics

Metrics are collected from:

* client side
* server side
* infrastructure
* application logs

---

### Client-side Metrics (k6)

k6 measures:

* response time
* requests per second (RPS)
* success rate
* timeout rate
* P95/P99 latency

Example:

| Metric       | Value       |
| ------------ | ----------- |
| RPS          | 406 req/sec |
| Avg response | 113ms       |
| P95          | 348ms       |
| Success rate | 100%        |

---

### Infrastructure Metrics (CloudWatch)

CloudWatch monitors:

* CPU usage
* memory usage
* network usage
* ECS scaling

Example:

| Metric | Value |
| ------ | ----- |
| CPU    | 31%   |
| Memory | 1.2%  |

---

### Application Logs

CloudWatch Logs Insights is used to analyze:

* request count
* processing duration
* API latency
* trace IDs
* external API timing

Example query:

```sql
fields @timestamp, duration_ms
| stats avg(duration_ms)
```

---

## Step 7 — Compare Client vs Server Metrics

Example:

| Metric       | Client | Server |
| ------------ | ------ | ------ |
| Avg response | 113ms  | 4ms    |

This shows:

* server processing is fast
* network latency dominates total response time

---

## Step 8 — Analyze Bottlenecks

A bottleneck is the slowest part of the system.

Possible bottlenecks:

* CPU
* memory
* database
* network
* external APIs

Example findings:

| Component    | Status  |
| ------------ | ------- |
| ECS CPU      | healthy |
| ECS Memory   | healthy |
| External API | slow    |

Conclusion:

* external API became the primary bottleneck

---

## Step 9 — Capacity Planning

Determine:

* maximum safe traffic
* scaling strategy
* infrastructure requirements

Example:

| Load        | Result           |
| ----------- | ---------------- |
| 30k req/min | stable           |
| 50k req/min | timeout failures |

Safe operating range:

* approximately 30,000 req/min

---

## Step 10 — Optimization Recommendations

Possible improvements:

* enable HTTP Keep-Alive
* add caching
* optimize external API calls
* tune timeout values
* increase concurrency
* improve autoscaling

# 5. Important Concepts

## HTTP Keep-Alive

Reuses the same TCP connection for multiple requests.

Benefits:

* lower latency
* fewer TCP handshakes
* higher throughput

---

## RPS (Requests Per Second)

Number of requests processed per second.

Example:

```text
406 req/sec
```

---

## Throughput

Amount of traffic the system can process.

Higher throughput means:

* better scalability
* higher processing capacity

---

## Response Time

Time between sending request and receiving response.

Example:

```text
113ms
```

---

## P95 / P99 Latency

P95 means:

* 95% of requests complete within this time

P99 means:

* 99% of requests complete within this time

These metrics are more important than average latency.

---

## Success Rate

Percentage of successful requests.

Example:

```text
100%
```

---

## Timeout

Occurs when response exceeds allowed waiting time.

Example:

* timeout limit: 10 seconds

---

## Bottleneck

The slowest component limiting overall performance.

Example:

* external API latency
* database saturation
* network congestion

---

## Scalability

Ability to handle increasing traffic by adding resources.

Example:

* adding ECS tasks increases throughput

---

## Autoscaling

Automatically adjusts server count based on load.

Example:

* CPU > 70%
* automatically add ECS tasks

---

## Network Latency

Time spent transferring data over the network.

Example:

| Component         | Time  |
| ----------------- | ----- |
| Server processing | 4ms   |
| Total response    | 113ms |

Network latency:

* approximately 109ms

---

## Trace ID

Unique identifier for tracking requests across services.

Useful for:

* debugging
* distributed tracing
* log correlation

---
