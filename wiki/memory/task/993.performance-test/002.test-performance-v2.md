# Performance Testing Guide and Real Case Study

## Overview

This document explains:

1. What performance testing is
2. Step-by-step process for executing performance tests
3. Tools used during testing
4. Important performance testing concepts
5. Real-world performance testing case study
6. How to analyze bottlenecks and system limits

The examples in this document are based on an actual production-like performance test executed on the `member-card API` integrated with the external Visionary API.

---

# 1. What is Performance Testing?

Performance testing is the process of verifying:

* How many requests a system can process
* How fast the system responds
* Whether the system remains stable under high traffic
* What component becomes the bottleneck
* Whether auto scaling works correctly

Performance testing is important because a system that works correctly with 10 users may fail with 10,000 users.

---

# 2. Main Goals of Performance Testing

Typical goals include:

| Goal                      | Description                              |
| ------------------------- | ---------------------------------------- |
| Throughput                | How many requests the system can process |
| Response Time             | How fast the API responds                |
| Stability                 | Whether the system crashes under load    |
| Scalability               | Whether adding servers improves capacity |
| Bottleneck Detection      | Identify the slowest component           |
| Infrastructure Validation | Verify CPU, memory, network usage        |

# 3. Important Performance Testing Concepts

## 3.1 Request Per Second (RPS)

RPS means how many requests are processed every second.

Example:

```text
300 req/sec
```

means:

```text
The system processes 300 requests every second.
```

Formula:

```text
RPS = Total Requests / Test Duration
```

---

## 3.2 Requests Per Minute (req/min)

Another throughput metric.

Example:

```text
30,000 req/min
```

## 3.3 Response Time

Time between:

```text
Client sends request
→ Server processes request
→ Client receives response
```

Measured in:

```text
milliseconds (ms)
```

Example:

```text
100ms = very fast
3000ms = 3 seconds
10000ms = timeout risk
```

---

## 3.4 P50 / P95 / P99

These are percentile metrics.

| Metric | Meaning                                    |
| ------ | ------------------------------------------ |
| P50    | 50% of requests completed within this time |
| P95    | 95% completed within this time             |
| P99    | 99% completed within this time             |

Example:

```text
P95 = 300ms
```

means:

```text
95% of requests finished within 300ms
```

P95 is one of the most important metrics in performance testing.

---

## 3.5 Virtual Users (VU)

A virtual user simulates a real user.

Example:

```text
200 VUs
```

means:

```text
200 simulated users sending requests simultaneously
```

---

## 3.6 Concurrency

Concurrency means how many requests are running at the same time.

Higher concurrency increases system load.

---

## 3.7 Throughput

Throughput means how much work the system completes.

Usually measured using:

* RPS
* req/min

---

## 3.8 Bottleneck

A bottleneck is the slowest component limiting system performance.

Examples:

* Database
* External API
* CPU
* Network
* Disk I/O

Even if ECS CPU is low, an external API can still become the bottleneck.

---

## 3.9 Timeout

Timeout means:

```text
The request took too long and was forcibly stopped.
```

Example:

```text
10 second timeout
```

If response exceeds 10 seconds:

```text
Request fails
```

---

## 3.10 Keep-Alive

HTTP Keep-Alive reuses TCP connections.

Without Keep-Alive:

```text
Every request creates a new TCP connection
```

This causes:

* High overhead
* Port exhaustion
* Slow performance

With Keep-Alive:

```text
One TCP connection reused for many requests
```

Benefits:

* Lower latency
* Higher RPS
* Lower CPU/network overhead

# 4. Tools Used During Performance Testing

# 4.1 k6

k6 is a modern load testing tool.

Purpose:

* Generate high traffic
* Measure response times
* Measure throughput
* Validate success rates

Advantages:

| Feature              | Benefit                |
| -------------------- | ---------------------- |
| HTTP Keep-Alive      | Better performance     |
| JavaScript scripting | Flexible scenarios     |
| Lightweight          | Can generate high load |
| Metrics support      | Easy analysis          |
| Threshold support    | Automatic validation   |

Example k6 script:

```javascript
import http from 'k6/http';
import { sleep } from 'k6';

export const options = {
  vus: 300,
  duration: '3m',
};

export default function () {
  http.post('https://example.com/api/v1/member-card');
  sleep(0.6);
}
```

---

# 4.2 Bash Script

Initially, a bash script was used.

Example:

```bash
curl -X POST https://example.com/api
```

Problem:

* No Keep-Alive
* Port exhaustion
* Client-side bottleneck
* Poor concurrency handling

Result:

```text
Only 20 req/sec achieved
```

even though server CPU remained low.

---

# 4.3 AWS ECS

Amazon ECS (Elastic Container Service) runs containerized applications.

In this test:

* Multiple ECS tasks handled API requests
* ECS scaling capability was validated

Important metrics:

| Metric            | Meaning                      |
| ----------------- | ---------------------------- |
| CPUUtilization    | CPU usage percentage         |
| MemoryUtilization | Memory usage percentage      |
| Task Count        | Number of running containers |

---

# 4.4 Amazon CloudWatch

CloudWatch collects:

* Metrics
* Logs
* Monitoring data

Used for:

* CPU analysis
* Memory analysis
* Response time analysis
* Log searching

---

# 4.5 CloudWatch Logs Insights

Used to query application logs.

Example query:

```sql
fields @timestamp, trace_id, duration_ms
| filter endpoint = "/api/v1/ec1/member-card"
| stats avg(duration_ms)
```

Purpose:

* Analyze server-side processing time
* Compare client vs server latency
* Detect bottlenecks

# 4.6 Visionary External API

The system depends on an external API called Visionary.

This external dependency became the final bottleneck during high-load tests.

# 5. Step-by-Step Performance Testing Process

# Step 1 — Define Performance Requirements

Before testing, define:

| Requirement               | Value                  |
| ------------------------- | ---------------------- |
| Comfortable response time | < 3 seconds            |
| Technical timeout limit   | < 10 seconds           |
| Target throughput         | 10,000–50,000 req/min |
| Success rate              | > 95%                  |

---

# Step 2 — Prepare Environment

Environment used:

| Component    | Configuration                |
| ------------ | ---------------------------- |
| ECS Tasks    | 2–10                        |
| Endpoint     | POST /api/v1/ec1/member-card |
| Load Tool    | k6                           |
| Monitoring   | CloudWatch                   |
| Log Analysis | CloudWatch Logs Insights     |

---

# Step 3 — Start with Small Load

Initial testing:

| Load           | Purpose                     |
| -------------- | --------------------------- |
| 3,000 req/min  | Baseline validation         |
| 10,000 req/min | Medium load                 |
| 20,000 req/min | Stress increase             |
| 30,000 req/min | Production-scale validation |
| 50,000 req/min | Find system limit           |

---

# Step 4 — Execute Load Test

Example k6 execution:

```bash
k6 run script.js
```

During testing, measure:

* RPS
* Success rate
* P95 latency
* CPU usage
* Memory usage
* External API latency

---

# Step 5 — Monitor Infrastructure

During testing:

CloudWatch monitored:

* ECS CPU
* ECS memory
* Scaling behavior

Example:

| CPU | Meaning    |
| --- | ---------- |
| 30% | Healthy    |
| 80% | Heavy load |
| 95% | Near limit |

---

# Step 6 — Analyze Application Logs

Use CloudWatch Logs Insights.

Goal:

Separate:

```text
Network latency
vs
Application processing time
```

This is extremely important.

---

# Step 7 — Compare Client vs Server Metrics

Example:

| Source           | Average Response |
| ---------------- | ---------------- |
| k6 client side   | 113ms            |
| Application logs | 4ms              |

Conclusion:

```text
Most latency came from network overhead.
```

---

# Step 8 — Identify Bottleneck

At 50,000 req/min:

| Component     | Status  |
| ------------- | ------- |
| ECS CPU       | Healthy |
| ECS Memory    | Healthy |
| Visionary API | Slow    |

Conclusion:

```text
External API became the bottleneck.
```

---

# Step 9 — Determine Safe Operating Range

Final result:

| Load           | Result   |
| -------------- | -------- |
| 30,000 req/min | Stable   |
| 50,000 req/min | Unstable |

Therefore:

```text
30,000 req/min became the recommended safe operating range.
```

# 8. Important Lessons Learned

## Lesson 1 — Low CPU Does Not Mean System Is Fast

At 50,000 req/min:

| Metric       | Value |
| ------------ | ----- |
| ECS CPU      | 35%   |
| Success Rate | 84%   |

Problem was NOT CPU.

Problem was:

```text
External API latency
```

---

## Lesson 2 — Network Latency Can Dominate Response Time

Example:

| Component   | Time  |
| ----------- | ----- |
| Application | 4ms   |
| End-to-end  | 113ms |

Most delay came from network.

---

## Lesson 3 — Client-Side Load Generator Matters

Bad load generator:

* Produces incorrect results
* Creates fake bottlenecks
* Cannot achieve target RPS

Always use:

* k6
* Vegeta
* JMeter
* Locust

instead of simple bash scripts.

---

## Lesson 4 — External Dependencies Are Dangerous

Even if your own application is fast:

```text
A slow external API can break the whole system.
```

# 13. Key Takeaways

1. Use proper load testing tools like k6
2. Measure both client-side and server-side latency
3. Monitor infrastructure during testing
4. Always identify the real bottleneck
5. External APIs often become the scalability limit
6. CPU alone is not enough to judge system health
7. P95 latency is more important than average latency
8. Safe operating range is more important than theoretical maximum capacity
