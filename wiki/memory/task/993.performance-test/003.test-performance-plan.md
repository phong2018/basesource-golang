# Performance Test Plan — basesource-golang

## Context

This project is a Go REST API (Echo v4 + MySQL + RabbitMQ + Kafka) deployed on Kubernetes (kind for local, real cluster for staging/prod).
The endpoints under test are the Todo CRUD API — the canonical working example that exercises the full stack.

The methodology is taken from `003.test-performance.md` and adapted to this project's actual infrastructure.

---

## Endpoints Under Test

| Method | Path              | Description                                       |
|--------|-------------------|---------------------------------------------------|
| GET    | /health           | Health check (no DB hit)                          |
| POST   | /api/v1/todos     | Create todo — writes DB + outbox_deliveries (heavy) |
| GET    | /api/v1/todos     | List todos — DB read                              |
| GET    | /api/v1/todos/:id | Get by ID — DB read single row                    |
| PUT    | /api/v1/todos/:id | Update todo — writes DB + outbox_deliveries       |
| DELETE | /api/v1/todos/:id | Delete todo — writes DB + outbox_deliveries       |

Primary focus: **POST /api/v1/todos** — this touches the most components (MySQL write + outbox pattern).
Secondary focus: **GET /api/v1/todos** — pure read path, should be fastest.

---

## Goals and Success Criteria

| Stage    | Target RPS | Max P95 latency | Min success rate |
|----------|------------ |-----------------|------------------|
| Baseline | 50 RPS      | < 200ms         | 100%             |
| Medium   | 200 RPS     | < 500ms         | 99%              |
| High     | 500 RPS     | < 1s            | 95%              |
| Stress   | 1000 RPS    | measure only    | observe          |

These targets are calibrated for a local kind cluster (single-node, limited CPU/RAM). Adjust upward for staging.

## k6 Script Design

### Stage progression (load_test.js)

```javascript
export const options = {
  stages: [
    { duration: '30s', target: 10  },   // warm up
    { duration: '1m',  target: 50  },   // baseline
    { duration: '1m',  target: 200 },   // medium
    { duration: '1m',  target: 500 },   // high
    { duration: '30s', target: 0   },   // cool down
  ],
  thresholds: {
    http_req_duration: ['p(95)<1000'],   // P95 < 1s overall
    http_req_failed:   ['rate<0.05'],    // error rate < 5%
  },
};
```

### Scenarios

**Scenario A — Write-heavy (POST)**
- 100% POST /api/v1/todos with random title
- Tests: DB write throughput, outbox write, connection pool saturation

**Scenario B — Read/write mix (realistic)**
- 70% GET /api/v1/todos, 30% POST /api/v1/todos
- Tests: read vs write contention

**Scenario C — Read-only (GET)**
- 100% GET /api/v1/todos
- Tests: maximum read throughput baseline

### Server-side (Kubernetes)

```bash
# CPU / memory per pod
kubectl top pods -n basesource

# HPA scaling decisions
kubectl get hpa -n basesource --watch

# API pod logs (latency from slog)
kubectl -n basesource logs -l app=basesource-api --tail=200

# MySQL connections (via exec)
kubectl -n basesource exec $(kubectl -n basesource get pod -l app=mysql -o jsonpath='{.items[0].metadata.name}') \
  -- mysql -u appuser -papppass appdb -e "SHOW STATUS LIKE 'Threads_connected';"
```

### Application logs (slog structured output)

The middleware logs every request with `latency_ms` under message `"request completed"` (see `middleware/logger.go`).
Query locally with:
```bash
kubectl -n basesource logs -l app=basesource-api --tail=5000 | \
  grep '"msg":"request completed"' | jq '.latency_ms' | sort -n | \
  awk 'BEGIN{c=0;s=0} {c++;s+=$1} END{print "avg:", s/c, "ms  count:", c}'
```

## Steps

### Step 0 — Install metrics-server (kind only, one-time)

`kubectl top` and HPA both require metrics-server, which kind does not include by default.

```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
# kind uses self-signed kubelet certs — patch to allow insecure TLS
kubectl patch deployment metrics-server -n kube-system \
  --type=json -p='[{"op":"add","path":"/spec/template/spec/containers/0/args/-","value":"--kubelet-insecure-tls"}]'
# wait until ready
kubectl rollout status deployment/metrics-server -n kube-system
```

### Step 1 — Start cluster and port-forward
```bash
kubectl -n basesource get pods                  # verify all pods ready
kubectl -n basesource port-forward svc/basesource-api-svc 18080:80 &
curl http://localhost:18080/health              # confirm API up
```

### Step 2 — Smoke test (1 VU, 30s)
```bash
k6 run --vus 1 --duration 30s wiki/memory/task/993.performance-test/scripts/smoke_test.js
```
Expected: 100% success, P95 < 100ms. If this fails, stop — environment issue.

### Step 3 — Baseline load test (POST-heavy)
```bash
k6 run wiki/memory/task/993.performance-test/scripts/load_test.js
```
Watch in another terminal:
```bash
kubectl top pods -n basesource --watch
kubectl get hpa -n basesource --watch
```

### Step 4 — Stress test
```bash
k6 run wiki/memory/task/993.performance-test/scripts/stress_test.js
```
Purpose: find the breaking point — where error rate climbs or P95 spikes.

### Step 5 — Collect and record results

Fill in the results table after each run:

| Stage    | VUs | RPS achieved | P50  | P95   | P99   | Error rate |
|----------|-----|--------------|------|-------|-------|------------|
| Baseline | 10  |              |      |       |       |            |
| Medium   | 50  |              |      |       |       |            |
| High     | 200 |              |      |       |       |            |
| Stress   | 500 |              |      |       |       |            |

### Step 6 — Analyze bottlenecks

Check in order:
1. **MySQL** — if CPU spikes on DB pod, the write path (outbox_deliveries) is the bottleneck
2. **API pod CPU** — if API pods max out before DB, increase replicas or optimize handler
3. **HPA lag** — HPA reacts in ~30s; a spike test will expose this gap
4. **Connection pool** — MySQL `Threads_connected` saturation shows connection exhaustion
5. **Outbox relay** — relay polling may cause DB read contention under high write volume

---

## What Makes This Project Interesting to Test

1. **Dual-write outbox** — every POST/PUT/DELETE writes `outbox_events` + `outbox_deliveries` in the same transaction. Under load, this doubles DB write pressure compared to a simple CRUD API.

2. **HPA autoscaling** — the cluster will scale API pods from 2 → up to 10 when CPU > 70%. A well-shaped load test (gradual ramp) will show HPA responding during the test.

3. **Worker relay contention** — `KafkaOutboxRelay` and `RabbitMQOutboxRelay` both poll `outbox_deliveries` every few seconds. Under high write load, relay SELECT queries compete with INSERT transactions on the same table.

4. **Read vs write asymmetry** — GET /api/v1/todos is a simple SELECT; POST writes 3 rows in a transaction. A mixed scenario reveals how much the write path degrades read throughput.

---

## Comparison: Client vs Server latency

The guide (`003.test-performance.md`) shows server processing can be 4ms while client sees 113ms — network + TCP overhead. In this local kind setup, both client and server are on localhost so network overhead is near zero. This is worth noting: local numbers will be much better than production. The gap tells you how much is pure network.
