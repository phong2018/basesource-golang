# RabbitMQ Integration — Real Test Guide

This guide shows how to run the full RabbitMQ outbox flow end-to-end on a local machine.

---

## Prerequisites

- Docker & Docker Compose installed
- Go 1.21+
- `.env` file with RabbitMQ vars (see below)

---

## Step 1 — Set up `.env`

```bash
cp .env.example .env
```

Add RabbitMQ vars to `.env`:

```env
RABBITMQ_URL=amqp://guest:guest@localhost:5672/
RABBITMQ_EXCHANGE=todo.events
```

---

## Step 2 — Start infrastructure (DB + RabbitMQ)

```bash
docker compose -f docker/docker-compose.yaml up -d db rabbitmq
```

Wait for both to be healthy:

```bash
# DB ready check
until docker compose -f docker/docker-compose.yaml exec -T db \
  mysqladmin ping -h localhost -u root -proot --silent 2>/dev/null; do sleep 2; done
echo "DB ready"

# RabbitMQ ready check
until docker compose -f docker/docker-compose.yaml exec -T rabbitmq \
  rabbitmq-diagnostics ping 2>/dev/null; do sleep 2; done
echo "RabbitMQ ready"
```

## Step 7 — Run the scenarios

### Scenario 1: Create a todo

```bash
curl -s -X POST http://localhost:8080/api/v1/todos \
  -H "Content-Type: application/json" \
  -d '{"title":"RabbitMQ outbox test","description":"Integration test"}'
```

Expected response:

```json
{"id":10,"title":"RabbitMQ outbox test","description":"Integration test","done":false,...}
```

What happens internally:
- Usecase opens a DB transaction
- Inserts row into `todos`
- Inserts row into `audit_logs`
- Inserts row into `outbox_events` with `status='pending'` and `event_type='todo.created'`
- Transaction commits atomically

### Scenario 2: Update the todo

```bash
curl -s -X PUT http://localhost:8080/api/v1/todos/10 \
  -H "Content-Type: application/json" \
  -d '{"title":"Updated title","description":"Updated","done":true}'
```

What happens: inserts `outbox_events` row with `event_type='todo.updated'`

### Scenario 3: Delete the todo

```bash
curl -s -X DELETE http://localhost:8080/api/v1/todos/10
```

What happens: inserts `outbox_events` row with `event_type='todo.deleted'`

## Step 8 — Verify outbox events in DB

After the relay runs (every 2 seconds), all events should be marked `published`:

```bash
docker compose -f docker/docker-compose.yaml exec db \
  mysql -uappuser -papppass appdb \
  -e "SELECT id, event_type, aggregate_id, status, published_at FROM outbox_events ORDER BY id;" \
  2>/dev/null | grep -v Warning
```

Expected output:

```
id  event_type    aggregate_id  status     published_at
1   todo.created  10            published  2026-05-23 23:24:32
2   todo.updated  10            published  2026-05-23 23:24:40
3   todo.deleted  10            published  2026-05-23 23:24:40
```

**All 3 rows must show `status=published`** — this confirms the relay successfully read each event from DB and published it to RabbitMQ.

---

## Step 9 — Verify consumer received events (worker log)

In the worker terminal you should see:

```json
{"level":"INFO","msg":"event received","event_type":"todo.created","aggregate_id":"10"}
{"level":"INFO","msg":"event received","event_type":"todo.updated","aggregate_id":"10"}
{"level":"INFO","msg":"event received","event_type":"todo.deleted","aggregate_id":"10"}
```

> Note: if two worker instances are running simultaneously, events are distributed between them (RabbitMQ round-robin). Check both logs — total across both must be 3.

## How the outbox flow works (summary)

```
POST /todos
  └─ usecase.Create()
       └─ tx.WithinTransaction()
            ├─ INSERT todos
            ├─ INSERT audit_logs
            └─ INSERT outbox_events (status='pending')  ← same atomic transaction

worker: OutboxRelay (every 2s)
  └─ SELECT * FROM outbox_events WHERE status='pending'
       └─ for each event:
            ├─ publisher.Publish() → RabbitMQ exchange 'todo.events'
            └─ UPDATE outbox_events SET status='published'

worker: RabbitMQConsumer
  └─ queue 'todo.events.worker' bound to exchange 'todo.events' with routing key '#'
       └─ handler(body) → log / process event → d.Ack()
```

**Why this approach is safe:**
- The outbox INSERT is inside the same DB transaction as the business write. If the API crashes after commit, the relay will pick up the event on the next poll — no event is lost.
- Direct `publisher.Publish()` after `tx.Commit()` would lose the event if the process crashes in the gap between commit and publish.

**Trade-off:**
- ~2s delivery latency (configurable via `relayInterval` in `outbox_relay.go`)
- At-least-once delivery — consumers must be idempotent (check `event_id` for deduplication if needed)
