# Task 995 — Split `cmd/worker/cmd.go` into Clean Architecture Layers

## Problem

All worker logic (message handlers + bootstrap) lives in one file: `cmd/worker/cmd.go`.
This violates the project's layering rule — the `cmd/` directory is pure entrypoint/bootstrap,
not a place for business-logic-adjacent handlers.

---

## Current Structure

```
cmd/worker/cmd.go          ← bootstrap + handleDomainEvent + handleNotificationTask
```

Three goroutines are started:
1. **Outbox Relay** → polls `outbox_events` and publishes to Kafka
2. **Kafka Consumer** → reads domain events for analytics / audit log
3. **RabbitMQ Consumer** → reads notification tasks for email / push

Two handler functions that belong in the presentation layer:
- `handleDomainEvent(body []byte) error` — deserializes `OutboxEvent`, logs it
- `handleNotificationTask(body []byte) error` — deserializes `Notification`, logs it

---

## Target Structure

```
cmd/worker/
  cmd.go                                          ← bootstrap only (Cobra, signal, errgroup)

internal/presentation/worker/
  domain_event_handler.go                         ← HandleDomainEvent
  notification_handler.go                         ← HandleNotificationTask
```

### Why `internal/presentation/worker/`?

The existing HTTP presentation layer lives in `internal/presentation/http/`.
Worker handlers are the "presentation" of the worker process — they receive external input
(Kafka / RabbitMQ messages) and dispatch to usecases, exactly as HTTP handlers receive HTTP
requests and dispatch to usecases. Adding a `worker/` sibling to `http/` keeps the pattern
consistent across both transport types.

## Architecture Analysis

### Dependency Rule Compliance

| File | May import | Actual imports |
|---|---|---|
| `internal/presentation/worker/*.go` | `domain`, `usecase` interfaces | `domain/model`, `log/slog` |
| `cmd/worker/cmd.go` | anything (entrypoint) | `container`, `config`, `presentation/worker` |

Both pass the rule. No `infrastructure` leaks into `usecase` or `domain`.

### Handler Signature Contract

The Kafka consumer (`KafkaConsumer.Start`) and RabbitMQ consumer (`RabbitMQNotificationConsumer.Start`)
both accept `func([]byte) error`. The handlers keep the same signature after the move.

## Implementation Plan

### Step 1 — Create `internal/presentation/worker/domain_event_handler.go`

```go
package worker

import (
    "encoding/json"
    "log/slog"

    domainModel "github.com/yourname/go-clean-base/internal/domain/model"
)

func HandleDomainEvent(body []byte) error {
    var event domainModel.OutboxEvent
    if err := json.Unmarshal(body, &event); err != nil {
        return err
    }
    slog.Info("domain event received",
        "event_type", event.EventType,
        "aggregate_id", event.AggregateID,
        "event_id", event.EventID,
    )
    return nil
}
```

- Rename from unexported `handleDomainEvent` → exported `HandleDomainEvent`
- Package: `worker` under `internal/presentation/worker`

### Step 2 — Create `internal/presentation/worker/notification_handler.go`

```go
package worker

import (
    "encoding/json"
    "log/slog"

    domainModel "github.com/yourname/go-clean-base/internal/domain/model"
)

func HandleNotificationTask(body []byte) error {
    var n domainModel.Notification
    if err := json.Unmarshal(body, &n); err != nil {
        return err
    }
    slog.Info("notification task received", "to", n.To, "subject", n.Subject)
    return nil
}
```

### Step 3 — Slim down `cmd/worker/cmd.go`

Replace the two inline handler functions with imports of the new package.

```go
package worker

import (
    ...
    workerPresentation "github.com/yourname/go-clean-base/internal/presentation/worker"
    ...
)

// goroutine 2: pass workerPresentation.HandleDomainEvent
return c.KafkaConsumer.Start(gCtx, workerPresentation.HandleDomainEvent)

// goroutine 3: pass workerPresentation.HandleNotificationTask
c.RabbitMQNotificationConsumer.Start(gCtx, ..., workerPresentation.HandleNotificationTask)
```

Remove the `handleDomainEvent` and `handleNotificationTask` function bodies from `cmd.go`.

## Future Extension Points

Once the handlers live in the presentation layer, future tickets can:

1. **Inject usecases into handlers** — e.g., `NewDomainEventHandler(auditUsecase IAuditUsecase)` returns a `func([]byte) error` closure, calling real business logic instead of just logging.
2. **Add handler tests** — unit-test handlers with mocked usecases, isolated from Kafka/RabbitMQ infrastructure.
3. **Add a `HandlerRegistry`** — map event types to handler functions for a dispatch table pattern.
