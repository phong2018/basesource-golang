# 992 — Kubernetes Infra Fix Plan (v1)

## Context

The existing `k8s/infra/` manifests use `Deployment` for MySQL, Kafka, and RabbitMQ — all stateful services that require stable pod identity and persistent storage. This plan fixes those manifests and assesses impact on related tasks.

---

## Problems Found

| Service | Current | Correct | Risk of current state |
|---|---|---|---|
| MySQL | `Deployment` | `StatefulSet` | Pod reschedule = data loss (no PVC) |
| Kafka | `Deployment` | `StatefulSet` | `nodeId` is hardcoded; log dir is ephemeral |
| RabbitMQ | `Deployment` | `StatefulSet` | Durable queues lost on pod reschedule |

All three also have **credentials hardcoded** in the manifest instead of referencing `basesource-secret`.

---

## Fix Plan

### T1 — Convert MySQL to StatefulSet

File: `k8s/infra/mysql.yaml`

- Change `kind: Deployment` → `kind: StatefulSet`
- Add `serviceName: mysql-headless` (StatefulSet requires a headless Service)
- Add `volumeClaimTemplates` for `/var/lib/mysql` (10Gi, ReadWriteOnce)
- Remove hardcoded `MYSQL_ROOT_PASSWORD`, `MYSQL_USER`, `MYSQL_PASSWORD` — replace with `secretKeyRef` from `basesource-secret`
- Add headless Service (`clusterIP: None`) alongside the existing ClusterIP Service
- Add `livenessProbe` (same `mysqladmin ping` command already used in readinessProbe)

Key: the test scripts in `994.script-test-k8s-kafka` and `994.script-test-k8s-rabbitmq` use `kubectl exec` against the pod selected by `-l app=mysql`. StatefulSet pods still carry this label — the `lib.sh` `mysql_pod()` helper is unaffected.

---

### T2 — Convert Kafka to StatefulSet

File: `k8s/infra/kafka.yaml`

- Change `kind: Deployment` → `kind: StatefulSet`
- Add `serviceName: kafka-headless`
- Add `volumeClaimTemplates` for `/var/lib/kafka/data` (10Gi, ReadWriteOnce)
- Change `KAFKA_ADVERTISED_LISTENERS` from `kafka-service:9092` to `kafka-0.kafka-headless.basesource.svc.cluster.local:9092` (stable DNS name from StatefulSet)
- Keep existing ClusterIP Service `kafka-service` for consumers that connect by service name
- Add headless Service

The k8s Kafka test scripts (`994.script-test-k8s-kafka/lib.sh`) look up the pod by `-l app=kafka` and use `kafka-service:9092` as the broker address. Both still work after the change.

---

### T3 — Convert RabbitMQ to StatefulSet

File: `k8s/infra/rabbitmq.yaml`

- Change `kind: Deployment` → `kind: StatefulSet`
- Add `serviceName: rabbitmq-headless`
- Add `volumeClaimTemplates` for `/var/lib/rabbitmq` (5Gi, ReadWriteOnce)
- Remove hardcoded `RABBITMQ_DEFAULT_USER`/`RABBITMQ_DEFAULT_PASS` — reference `basesource-secret`
- Add `RABBITMQ_NODENAME: rabbit@rabbitmq-0.rabbitmq-headless.basesource.svc.cluster.local` for stable cluster identity
- Keep existing ClusterIP Service `rabbitmq-service` for AMQP connections
- Add headless Service

The RabbitMQ test scripts connect via `rabbitmq-service:5672` and the management API at `localhost:15672` (via port-forward). Both are unaffected.

---

### T4 — Wire secrets for infra credentials

File: `k8s/secret.yaml` (existing template)

Add keys for infra services:
```
MYSQL_ROOT_PASSWORD: CHANGE_ME
MYSQL_USER: appuser
MYSQL_PASSWORD: CHANGE_ME
RABBITMQ_DEFAULT_USER: CHANGE_ME
RABBITMQ_DEFAULT_PASS: CHANGE_ME
```

These are referenced via `secretKeyRef` in the StatefulSet env blocks (T1, T3).

## DaemonSet Additions

Two DaemonSets are added to `k8s/infra/` to demonstrate the third workload kind from the review.

### T5 — Fluent Bit (log collector)

File: `k8s/infra/fluent-bit.yaml`

**Why DaemonSet:** Container logs live at `/var/log/containers/` on the node's filesystem. A Deployment pod lands on one node and cannot read logs from other nodes. DaemonSet guarantees exactly one Fluent Bit pod per node so every node's logs are collected.

What it does:
- Tails `/var/log/containers/*basesource*.log` — filters to only this app's containers
- Parses Docker JSON log format (the format containerd/Docker writes to the node)
- Adds `cluster` and `env` fields via a record modifier filter
- Outputs to stdout as JSON lines (swap `[OUTPUT]` block to ship to Elasticsearch/Loki/CloudWatch in production)
- Mounts `/var/log` and `/var/lib/docker/containers` as read-only host paths

RBAC: needs `ClusterRole` to read pod/namespace metadata for Kubernetes metadata enrichment.

---

### T6 — Prometheus Node Exporter (per-node metrics)

File: `k8s/infra/node-exporter.yaml`

**Why DaemonSet:** Each node's hardware metrics (CPU, memory, disk I/O, network) are only visible from that node. A Deployment cannot collect metrics from nodes it is not scheduled on. DaemonSet places one exporter on every node.

What it does:
- Exposes per-node Prometheus metrics at `:9100/metrics`
- Uses `hostNetwork: true` and `hostPID: true` to read node-level stats
- Mounts the node's root filesystem read-only via `hostPath: /`
- Annotated with `prometheus.io/scrape: "true"` for auto-discovery
- Headless Service lets Prometheus scrape each pod individually

RBAC: needs `ClusterRole` for `/metrics` non-resource URL.

## Three Workload Kinds — Complete Picture

| Kind | Files | Use case in this project |
|---|---|---|
| `Deployment` | `k8s/api/`, `k8s/worker/` | Stateless Go API + worker pods — any pod is interchangeable |
| `StatefulSet` | `k8s/infra/mysql.yaml`, `kafka.yaml`, `rabbitmq.yaml` | Stateful services needing stable identity + persistent storage |
| `DaemonSet` | `k8s/infra/fluent-bit.yaml`, `node-exporter.yaml` | Node-level agents that must run on every node |
