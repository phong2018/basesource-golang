# Kubernetes Concepts — Plain-Language Guide

All examples are grounded in the `basesource-golang` project so you can see exactly where each concept appears in the real manifests.

---

## Mental model first

Think of Kubernetes like a **smart data centre OS**:

```
You write:  "I want 2 copies of my API running at all times"
K8s does:   finds 2 healthy machines, starts the containers,
            restarts them if they crash, replaces them if a machine dies
```

You declare **what you want** (desired state) in YAML files.  
K8s continuously reconciles **what actually exists** toward your desired state.

---

## 1. Cluster, Node, Pod

### Cluster
A group of machines (physical or virtual) managed together by Kubernetes.

```
Cluster
├── Node 1  (machine / VM)
├── Node 2  (machine / VM)
└── Node 3  (machine / VM)
```

### Node
A single machine in the cluster. Kubernetes decides which node runs which workload.

### Pod
The smallest deployable unit. A pod wraps one or more containers that always run together on the same node and share the same network (IP address + ports).

```
Pod: basesource-api-7d9f6b-xk2p
├── container: api  (runs /usr/local/bin/app api)
└── (shared localhost between containers if there were more)
```

**Analogy:** A node is a physical server rack. A pod is a single process slot on that rack.

You almost never create pods directly — you use a Deployment which manages pods for you.

---

## 2. Namespace

A namespace is a **virtual cluster inside the cluster** — a way to isolate resources by team, environment, or project.

```yaml
# k8s/namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: basesource
```

Without a namespace, everything lands in `default`. Creating `basesource` means:
- `kubectl get pods -n basesource` shows only our pods
- Names only need to be unique within the namespace, not the whole cluster
- You can set resource quotas per namespace

**Analogy:** Namespaces are like separate folders on a shared computer. Different teams work in their own folder and don't see each other's files unless they look explicitly.

## 3. Deployment

A Deployment tells Kubernetes: *"keep N copies of this pod alive and roll out updates safely"*.

```yaml
# k8s/api/deployment.yaml (simplified)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: basesource-api
spec:
  replicas: 2                  # ← keep 2 pods alive
  selector:
    matchLabels:
      app: basesource-api      # ← manages pods that have this label
  template:                    # ← blueprint for the pods it creates
    metadata:
      labels:
        app: basesource-api
    spec:
      containers:
        - name: api
          image: your-registry/basesource:latest
          args: ["api"]
```

**What it does automatically:**
- If a pod crashes → Deployment starts a replacement
- If a node dies → Deployment reschedules the pods on a healthy node
- If you update the image tag → Deployment does a rolling update (see section 5)

**Analogy:** A Deployment is like a job posting. You say "I need 2 senior Go developers working at all times." If one quits, the recruiter immediately finds a replacement. The job description (pod template) says exactly what skills are required.

## 4. Service

A Service gives your pods a **stable network address** regardless of which pods are currently alive.

Pods are ephemeral — they get new IP addresses every time they restart. A Service watches for pods matching a label and load-balances traffic across them.

```yaml
# k8s/api/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: basesource-api-svc
spec:
  selector:
    app: basesource-api        # ← forward traffic to pods with this label
  ports:
    - port: 80                 # ← Service listens on port 80 (inside cluster)
      targetPort: 8080         # ← forwards to container port 8080
  type: ClusterIP              # ← only reachable inside the cluster
```

**Traffic flow inside the cluster:**
```
Any pod → basesource-api-svc:80 → [load balancer] → api-pod-1:8080
                                                   → api-pod-2:8080
```

**Three Service types:**
| Type | Reachable from | Use case |
|---|---|---|
| `ClusterIP` | Inside cluster only | Internal services (DB, worker) |
| `NodePort` | Outside via `<node-ip>:<port>` | Dev/testing without ingress |
| `LoadBalancer` | Outside via cloud LB | Direct cloud exposure (costs money) |

In our plan we use `ClusterIP` and expose externally only via Ingress (section 7).

**Analogy:** A Service is like a reception desk with a fixed phone number. Employees (pods) come and go, but callers always dial the same number and the desk routes them to whoever is available.

## 5. Rolling Update (zero-downtime deploy)

When you push a new image, Kubernetes replaces pods one batch at a time so there is always capacity to serve requests.

```yaml
# inside api/deployment.yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 0    # never go below 2 running pods
    maxSurge: 1          # allow 1 extra pod (temporary 3 pods during rollout)
```

**Step-by-step with `replicas: 2`:**

```
Start:      [api-v1]  [api-v1]          ← 2 pods serving traffic

Step 1:     [api-v1]  [api-v1]  [api-v2-starting]
            maxSurge=1 allows a 3rd pod to start

Step 2:     [api-v1]  [api-v1]  [api-v2-ready]
            new pod passes readiness probe → joins Service endpoints

Step 3:     [api-v1-terminating]  [api-v1]  [api-v2]
            old pod removed; maxUnavailable=0 so we still have 2 ready

Step 4:     [api-v1]  [api-v2]
            repeat for remaining old pod

End:        [api-v2]  [api-v2]          ← rollout complete, no downtime
```

`maxUnavailable: 0` is the key — it ensures the Service always has at least `replicas` ready pods serving traffic.

**Analogy:** Renovating a hotel floor by floor. Guests always have rooms (maxUnavailable=0), but during renovation there might be one extra floor open temporarily (maxSurge=1).

## 6. Resource Requests and Limits

```yaml
resources:
  requests:
    cpu: 100m       # minimum CPU guaranteed (100 millicores = 0.1 CPU core)
    memory: 128Mi   # minimum RAM guaranteed
  limits:
    cpu: 500m       # maximum CPU allowed (burst up to 0.5 core)
    memory: 256Mi   # maximum RAM — if exceeded, container is OOM-killed
```

**Requests vs Limits:**
- **Request** = what Kubernetes uses for **scheduling**. It guarantees this much is available on the node.
- **Limit** = the **hard ceiling**. The container is throttled (CPU) or killed (memory) if it exceeds this.

**Why both matter:**
```
Node has 2 CPU cores total

Pod A requests 100m → scheduled on node
Pod B requests 100m → scheduled on node
Pod C requests 100m → scheduled on node
... up to 2000m total

If Pod A suddenly spikes to 500m (its limit), it can burst
but won't starve the node for other pods beyond its limit.
```

Without limits, one runaway pod can starve all others on the same node ("noisy neighbour" problem).

**Analogy:**
- `requests` = the seat you reserved at a restaurant (guaranteed)
- `limits` = the maximum table space you're allowed to use even if the restaurant is empty

## 7. Ingress

An Ingress routes **external HTTP/HTTPS traffic** into the cluster. It acts as a reverse proxy sitting in front of one or many Services.

```yaml
# k8s/api/ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: basesource-ingress
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"   # auto-issue TLS cert
spec:
  ingressClassName: nginx
  tls:
    - hosts: [api.example.com]
      secretName: basesource-tls    # cert-manager stores the cert here
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: basesource-api-svc
                port:
                  number: 80
```

**Traffic flow:**
```
Internet
  → DNS: api.example.com → cluster IP
  → nginx-ingress-controller (handles TLS termination)
  → basesource-api-svc (ClusterIP)
  → api pods
```

**Deployment vs Service vs Ingress — one-line summary:**
```
Deployment  = manages pod lifecycle (how many, which image, how to update)
Service     = stable internal address for pods (load balances within cluster)
Ingress     = routes external traffic in, handles TLS (the front door)
```

**Analogy:**
- Deployment = the kitchen staff
- Service = the internal intercom system between departments
- Ingress = the front door and reception that takes calls from outside

## 8. HorizontalPodAutoscaler (HPA)

HPA automatically adjusts the number of pod replicas based on a metric (usually CPU or memory).

```yaml
# k8s/api/hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef:
    kind: Deployment
    name: basesource-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # scale out if avg CPU > 70%
```

**How HPA decides:**

```
Current state: 2 pods, avg CPU = 85%
Target:        70%
Desired pods:  ceil(2 × 85/70) = ceil(2.43) = 3  → scale to 3 pods

Later: avg CPU drops to 40%
Desired pods:  ceil(3 × 40/70) = ceil(1.71) = 2  → scale back to 2 pods
```

HPA checks metrics every 15 seconds (default) but waits several minutes before scaling down to avoid flapping.

**Why minReplicas: 2?**
A single replica means one pod failure = total outage. Two replicas ensures the Service always has a backup.

**Why worker maxReplicas: 6 (not 10)?**
The Kafka topic has 1 partition. Kafka assigns at most 1 consumer per partition. Having 10 worker pods does not help Kafka throughput — 9 pods would just sit idle for the Kafka consumer goroutine. Setting max to 6 still helps the RabbitMQ consumer and outbox relay goroutines while avoiding waste.

**Analogy:** HPA is like a call centre manager who watches hold times. If the queue grows (high CPU = busy), they call in extra agents. When it quiets down, they let some agents go home.

## 9. Readiness Probe vs Liveness Probe

Both probes are health checks — but they trigger **different actions**.

```yaml
readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5     # wait 5s before first check
  periodSeconds: 10          # check every 10s
  failureThreshold: 3        # fail 3 times before acting

livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 15    # give it more time to fully start
  periodSeconds: 20
  failureThreshold: 3
```

| | Readiness | Liveness |
|---|---|---|
| **Fails → action** | Remove from Service endpoints (stop sending traffic) | Restart the container |
| **Passes → action** | Add to Service endpoints (resume traffic) | Nothing (all good) |
| **Purpose** | "Is this pod ready to accept requests?" | "Is this pod alive and not deadlocked?" |
| **Use case** | Pod is still starting; pod is temporarily overloaded | Pod is stuck in infinite loop or deadlock |

**Real scenario:**
```
Deploy new version → pod starts
├── Readiness FAILS (app still loading DB connections)
│   → Service stops routing traffic here
│   → Old pod keeps serving until new one is ready
│
├── Readiness PASSES (app fully initialised)
│   → Service adds this pod to its endpoints
│   → Traffic flows in
│
└── Later: deadlock in goroutine → /health starts timing out
    ├── Readiness FAILS → Service stops routing traffic
    └── Liveness FAILS × 3 → Kubernetes restarts container
```

**Analogy:**
- Readiness = "Are you available to take a call?" — if no, calls go to someone else
- Liveness = "Are you still conscious?" — if no, manager taps you on the shoulder (restart)

## 10. ConfigMap vs Secret

Both inject configuration into pods as environment variables or files.

```yaml
# ConfigMap — plain text, safe to commit to Git
apiVersion: v1
kind: ConfigMap
data:
  APP_PORT: "8080"
  KAFKA_TOPIC: "todo-events"
  RABBITMQ_EXCHANGE: "todo.events"

# Secret — base64-encoded, never commit real values to Git
apiVersion: v1
kind: Secret
type: Opaque
stringData:
  DATABASE_DSN: "appuser:secret@tcp(mysql:3306)/appdb"
  AWS_SECRET_ACCESS_KEY: "supersecret"
  RABBITMQ_URL: "amqp://user:secret@rabbitmq:5672/"
```

Both are referenced in the pod spec with `envFrom`:

```yaml
envFrom:
  - configMapRef:
      name: basesource-config    # → all keys become env vars
  - secretRef:
      name: basesource-secret    # → all keys become env vars
```

**Rule of thumb:**
| Belongs in ConfigMap | Belongs in Secret |
|---|---|
| Port numbers | Passwords |
| Topic/exchange names | API keys |
| Region names | Connection strings with credentials |
| Feature flags | TLS certificates |

**Secrets are not truly encrypted by default** in Kubernetes (just base64). For production use a secrets manager (AWS Secrets Manager, Vault, external-secrets-operator) that injects values at deploy time and keeps them out of Git entirely.

**Analogy:**
- ConfigMap = a public noticeboard with the office WiFi network name
- Secret = a sealed envelope with the WiFi password, handed out individually

## 11. Namespace (isolation detail)

Namespace resources that are **namespace-scoped** (most things):
- Pod, Deployment, Service, Ingress, HPA, ConfigMap, Secret, Job

Resources that are **cluster-scoped** (not namespaced):
- Namespace itself, Node, ClusterRole, PersistentVolume

When you run `kubectl get pods` you only see the `default` namespace.  
Always add `-n basesource` or `--all-namespaces` (`-A`) to see everything:

```bash
kubectl get pods -n basesource
kubectl get all -n basesource
kubectl get events -n basesource --sort-by='.lastTimestamp'
```

## 12. Job (one-shot task)

A Job runs a pod **to completion** and does not restart it unless it fails.

```yaml
# k8s/migrate-job.yaml
apiVersion: batch/v1
kind: Job
spec:
  backoffLimit: 3        # retry up to 3 times on failure
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          args: ["migrate"]   # runs: /usr/local/bin/app migrate
```

This is used for the database migration: run once per deploy, verify it completed, then roll out the api and worker Deployments.

```bash
kubectl apply -f k8s/migrate-job.yaml
kubectl wait --for=condition=complete job/basesource-migrate \
  -n basesource --timeout=120s
```

**Deployment vs Job:**
| | Deployment | Job |
|---|---|---|
| Lifecycle | Runs forever, restarts on crash | Runs until completion |
| Use case | API server, worker | DB migration, data import, report generation |
| Restart policy | Always | Never / OnFailure |

**Analogy:** A Deployment is a permanent employee. A Job is a contractor hired for one specific task — once done, they leave.

## 13. securityContext (non-root)

Running as root inside a container is a security risk — if the container is compromised, the attacker has root. Kubernetes can enforce non-root execution.

```yaml
# in pod spec
securityContext:
  runAsNonRoot: true
  runAsUser: 65532      # distroless nonroot uid

# in Dockerfile
FROM gcr.io/distroless/static-debian11:nonroot
USER nonroot:nonroot    # uid 65532
```

`runAsNonRoot: true` tells Kubernetes to reject the pod at admission if the container would run as root — it's a safety net even if the Dockerfile is wrong.

## 14. Full traffic flow — end to end

Putting it all together for a `POST /api/v1/todos` request:

```
1. Browser / client
   → DNS: api.example.com → cluster load balancer IP

2. nginx-ingress-controller (Ingress)
   → TLS termination (cert-manager cert stored in Secret basesource-tls)
   → routes to Service basesource-api-svc

3. Service basesource-api-svc (ClusterIP)
   → load-balances across ready api pods
   → only sends to pods where readinessProbe passes

4. Pod: basesource-api-xxx (Deployment)
   → container: app api
   → env vars from ConfigMap basesource-config + Secret basesource-secret
   → writes to MySQL (via DATABASE_DSN from Secret)
   → writes outbox_deliveries to MySQL

5. Pod: basesource-worker-yyy (Deployment — separate)
   → KafkaOutboxRelay polls DB → publishes to Kafka
   → RabbitMQOutboxRelay polls DB → publishes to RabbitMQ
   → KafkaConsumer receives domain events
   → RabbitMQConsumer receives notifications
```

## 15. Useful kubectl commands

```bash
# see everything in the namespace
kubectl get all -n basesource

# watch pods in real time
kubectl get pods -n basesource -w

# describe a pod (events, probe status, image)
kubectl describe pod <pod-name> -n basesource

# tail logs from the api deployment
kubectl logs -f deployment/basesource-api -n basesource

# tail logs from all worker pods simultaneously
kubectl logs -f -l app=basesource-worker -n basesource

# check HPA status (current replicas, CPU %)
kubectl get hpa -n basesource

# exec into a running pod
kubectl exec -it <pod-name> -n basesource -- /bin/sh

# force a rolling restart (picks up new ConfigMap values)
kubectl rollout restart deployment/basesource-api -n basesource

# check rollout status
kubectl rollout status deployment/basesource-api -n basesource

# roll back to previous version
kubectl rollout undo deployment/basesource-api -n basesource
```
