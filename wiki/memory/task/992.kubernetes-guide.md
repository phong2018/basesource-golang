# Kubernetes Hands-On Learning Guide

Cluster: `kind` (Kubernetes-in-Docker), name `basesource`  
All commands assume you are in the repo root.

---

## Prerequisites

```bash
# Verify cluster is running
kubectl cluster-info --context kind-basesource

# Verify all pods are up
kubectl -n basesource get pods
```

Expected: 8 pods — api (×2), worker (×2), mysql, rabbitmq, kafka, migrate (Completed).

---

## Lab 1 — Observe the Running Cluster

**Goal:** understand what is deployed and how the pieces connect.

```bash
# List everything in the namespace
kubectl -n basesource get all

# Describe a deployment — shows replicas, strategy, image, envFrom
kubectl -n basesource describe deployment basesource-api

# Describe a pod — shows node, IP, containers, readiness probe, events
kubectl -n basesource describe pod -l app=basesource-api

# Show the ConfigMap and Secret (values are base64 in Secret)
kubectl -n basesource get configmap basesource-config -o yaml
kubectl -n basesource get secret basesource-secret -o yaml
```

**What to notice:**
- `envFrom` in the deployment pulls all keys from ConfigMap + Secret into env vars.
- The pod IP is internal (10.x.x.x) — unreachable from your laptop directly.
- `ClusterIP` services give pods a stable DNS name inside the cluster.

## Lab 2 — Access the API via Port-Forward

**Goal:** understand how traffic reaches pods from outside the cluster.

```bash
# Forward local port 18080 → service port 80 → pod port 8080
kubectl -n basesource port-forward svc/basesource-api-svc 18080:80
```

In another terminal:

```bash
# Health check
curl http://localhost:18080/health

# Create a todo
curl -s -X POST http://localhost:18080/api/v1/todos \
  -H "Content-Type: application/json" \
  -d '{"title":"Learn Kubernetes"}' | jq

# List todos
curl -s http://localhost:18080/api/v1/todos | jq
```

**What to notice:**
- port-forward is a tunnel for development only — not for production.
- In production, traffic flows: Internet → Ingress (nginx) → Service → Pod.
- The Service round-robins between the 2 API pods.

## Lab 3 — Read Logs

**Goal:** understand how to debug running workloads.

```bash
# Logs from all API pods (label selector)
kubectl -n basesource logs -l app=basesource-api

# Follow logs in real time (-f like tail -f)
kubectl -n basesource logs -l app=basesource-api -f

# Worker logs — you can see outbox relay ticking every 2s
kubectl -n basesource logs -l app=basesource-worker -f

# Logs from a specific pod (copy the name from get pods)
kubectl -n basesource logs basesource-api-77644885b8-8x5gs

# Previous container logs (after a crash)
kubectl -n basesource logs basesource-api-77644885b8-8x5gs --previous
```

**What to notice:**
- Worker logs show `outbox_kafka_relay` and `outbox_rabbitmq_relay` polling.
- Every POST /todos triggers outbox events that the worker picks up and publishes.
- `request_id` flows through all log lines — this is the `ctx` propagation.

## Lab 4 — Kill a Pod, Watch Kubernetes Restart It

**Goal:** understand self-healing.

```bash
# Terminal 1 — watch pods
kubectl -n basesource get pods -w

# Terminal 2 — delete one API pod
kubectl -n basesource delete pod -l app=basesource-api --field-selector=status.phase=Running

# Or delete a specific pod by name
kubectl -n basesource delete pod basesource-api-77644885b8-8x5gs
```

**What to observe:**
1. Pod enters `Terminating` state.
2. Deployment sees `READY = 1/2`, immediately starts a new pod.
3. New pod starts, passes readiness probe (`/health`), enters `Running`.
4. Total downtime: ~5 seconds (readiness probe: `initialDelaySeconds: 5`).

**Why it works:** The Deployment controller continuously reconciles `desired replicas = 2` vs `actual ready = N`. A deleted pod drops N below 2, so a new one is created.

## Lab 5 — Rolling Update (Zero Downtime Deploy)

**Goal:** understand how a new image version is deployed without dropping requests.

```bash
# Terminal 1 — watch pods during rollout
kubectl -n basesource get pods -w

# Terminal 2 — trigger a rollout by updating the image tag
# (we'll just force a restart since the image didn't change)
kubectl -n basesource rollout restart deployment basesource-api
```

**What to observe:**
1. `maxSurge: 1` → a 3rd pod is started first.
2. Once the new pod is Ready, one old pod is terminated.
3. Then the 4th pod starts, the last old pod is terminated.
4. At no point are there fewer than 2 ready pods.

```bash
# Check rollout status
kubectl -n basesource rollout status deployment basesource-api

# See rollout history
kubectl -n basesource rollout history deployment basesource-api

# Roll back to previous version
kubectl -n basesource rollout undo deployment basesource-api
```

## Lab 6 — Manual Scaling

**Goal:** understand how to scale a deployment up and down.

```bash
# Scale API to 4 replicas
kubectl -n basesource scale deployment basesource-api --replicas=4

# Watch new pods come up
kubectl -n basesource get pods -w

# Scale back down to 2
kubectl -n basesource scale deployment basesource-api --replicas=2
```

**What to notice:**
- When scaling down, pods are terminated gracefully (SIGTERM → 30s → SIGKILL).
- The Service automatically updates its endpoint list to include/exclude pods based on readiness.

## Lab 7 — HPA (Horizontal Pod Autoscaler)

**Goal:** understand how Kubernetes auto-scales based on CPU.

```bash
# Current HPA state
kubectl -n basesource get hpa

# Watch HPA decisions in real time
kubectl -n basesource describe hpa basesource-api-hpa
```

**How HPA works:**
- Every 15s the HPA reads average CPU across all API pods.
- If CPU > 70%: scale up (add pods, max 10).
- If CPU < 70% for 5 minutes: scale down (remove pods, min 2).
- `cpu: <unknown>` means the metrics-server isn't installed in kind (normal for local dev).

To install metrics-server in kind:
```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
kubectl patch -n kube-system deployment metrics-server \
  --type=json \
  -p='[{"op":"add","path":"/spec/template/spec/containers/0/args/-","value":"--kubelet-insecure-tls"}]'

# Then after ~30s
kubectl -n basesource get hpa
kubectl -n basesource top pods
```

## Lab 8 — Exec Into a Pod

**Goal:** inspect a running container from inside.

```bash

# Check todos table
SELECT * FROM todos;
SELECT * FROM outbox_events;
SELECT * FROM outbox_deliveries;
```

## Lab 9 — Watch the Outbox Flow End-to-End

**Goal:** see the full journey: HTTP → DB → outbox → Kafka/RabbitMQ → consumer.

```bash
# Terminal 1: watch worker logs
kubectl -n basesource logs -l app=basesource-worker -f

# Terminal 2: port-forward and create a todo
kubectl -n basesource port-forward svc/basesource-api-svc 18080:80
curl -s -X POST http://localhost:18080/api/v1/todos \
  -H "Content-Type: application/json" \
  -d '{"title":"Watch the outbox"}' | jq
```

**What to observe in worker logs:**
1. `outbox_kafka_relay: published event` — delivery sent to Kafka.
2. `outbox_rabbitmq_relay: published event` — delivery sent to RabbitMQ.
3. `notification_consumer: received message` — RabbitMQ consumer picks it up.

## Lab 10 — ConfigMap and Secret Hot-Reload Pattern

**Goal:** understand the difference between ConfigMap and Secret, and how to update them.

```bash
# Edit the ConfigMap (e.g. change KAFKA_TOPIC)
kubectl -n basesource edit configmap basesource-config

# Changes to ConfigMap don't auto-restart pods — you must rollout restart
kubectl -n basesource rollout restart deployment basesource-api
kubectl -n basesource rollout restart deployment basesource-worker
```

**Why pods don't auto-restart on ConfigMap change:**
- envFrom loads values at container start time only.
- To pick up new values, the container must restart.
- This is intentional — it prevents accidental config changes from disrupting production.

## Lab 11 — Tear Down and Rebuild

**Goal:** practice deploying from scratch.

```bash
# Delete everything in the namespace
kubectl delete namespace basesource

# Recreate from scratch
kubectl apply -f k8s/namespace.yaml

kubectl -n basesource wait --for=condition=ready pod -l app=rabbitmq --timeout=120s

kubectl apply -f k8s/api/ -f k8s/worker/
kubectl -n basesource wait --for=condition=ready pod -l app=basesource-api --timeout=60s
kubectl -n basesource get pods
```

## Lab 12 — Rebuild Image After Code Change

**Goal:** practice the full local dev loop with kind.

```bash
# 1. Make a code change (e.g. edit a handler)
# 2. Rebuild the image
docker build -f docker/Dockerfile -t basesource:local .

# 3. Load the new image into kind (replaces the old one)
kind load docker-image basesource:local --name basesource

# 4. Force a rollout (pods still reference the same tag, so restart forces pull)
kubectl -n basesource rollout restart deployment basesource-api
kubectl -n basesource rollout restart deployment basesource-worker

# 5. Watch new pods come up
kubectl -n basesource get pods -w
```

**Why `imagePullPolicy: Never`:**
- kind nodes are Docker containers — they can't reach your local Docker registry.
- `kind load docker-image` copies the image tarball into the kind node.
- `imagePullPolicy: Never` tells kubelet to use the already-loaded image instead of pulling.
