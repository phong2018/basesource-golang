# Kubernetes Deployment Plan — basesource-golang

## Overview

Deploy the Go Clean Architecture app to Kubernetes with zero-downtime rolling updates, horizontal auto-scaling, health probes, and clean separation of config from secrets.

The app has two runtime modes managed by separate Kubernetes Deployments:
- **api** — HTTP server on port 8080
- **worker** — outbox relays + Kafka/RabbitMQ consumers (no HTTP port)

## 1. Dockerfile Improvements

Current Dockerfile is already multi-stage + distroless. Two changes needed:

```dockerfile
FROM golang:1.23 AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -a -o app .

FROM gcr.io/distroless/static-debian11:nonroot AS final   # ← nonroot variant
COPY --from=builder /src/app /usr/local/bin/app
COPY --from=builder /src/db/migrations /app/db/migrations
WORKDIR /app
USER nonroot:nonroot                                       # ← explicit non-root
ENTRYPOINT ["/usr/local/bin/app"]
CMD ["api"]
```

Changes from current:
- `distroless/static-debian11:latest` → `distroless/static-debian11:nonroot` — ships with uid 65532 (`nonroot`)
- Add `USER nonroot:nonroot` — required for Kubernetes `runAsNonRoot: true` admission check
- `HEALTHCHECK` is not needed in Kubernetes (k8s probes replace Docker HEALTHCHECK); skip to keep image minimal

## 5. migrate-job.yaml

Run once before deploying api/worker to bring the schema up to date.

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: basesource-migrate
  namespace: basesource
spec:
  backoffLimit: 3
  template:
    spec:
      restartPolicy: Never
      securityContext:
        runAsNonRoot: true
        runAsUser: 65532
      containers:
        - name: migrate
          image: your-registry/basesource:latest
          args: ["migrate"]
          envFrom:
            - configMapRef:
                name: basesource-config
            - secretRef:
                name: basesource-secret
```

Apply and wait:
```bash
kubectl apply -f k8s/migrate-job.yaml
kubectl wait --for=condition=complete job/basesource-migrate -n basesource --timeout=120s
```

## 6. api/deployment.yaml

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: basesource-api
  namespace: basesource
spec:
  replicas: 2
  selector:
    matchLabels:
      app: basesource-api
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0    # zero-downtime: never drop below desired replica count
      maxSurge: 1          # allow 1 extra pod during rollout
  template:
    metadata:
      labels:
        app: basesource-api
    spec:
      securityContext:
        runAsNonRoot: true
        runAsUser: 65532
      containers:
        - name: api
          image: your-registry/basesource:latest
          args: ["api"]
          ports:
            - containerPort: 8080
          envFrom:
            - configMapRef:
                name: basesource-config
            - secretRef:
                name: basesource-secret
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 256Mi
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
            failureThreshold: 3
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 20
            failureThreshold: 3
```

**Probe distinction:**
- **readinessProbe** — removes the pod from Service endpoints while it is starting up or temporarily overloaded; no traffic is routed to it
- **livenessProbe** — restarts the pod if it is stuck (e.g. deadlocked goroutines that never recover)

## 8. api/ingress.yaml

```yaml

kind: Ingress
metadata:
  name: basesource-ingress
  namespace: basesource
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - api.example.com
      secretName: basesource-tls
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: basesource-api-svc
                port:
                  number: 80
```

Prerequisites: nginx-ingress-controller + cert-manager installed in the cluster.

## 9. api/hpa.yaml

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: basesource-api-hpa
  namespace: basesource
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: basesource-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

Scales out when average CPU across pods exceeds 70%. Minimum 2 replicas maintains HA during normal load.

## 10. worker/deployment.yaml

The worker runs 4 background goroutines (Kafka relay, RabbitMQ relay, Kafka consumer, RabbitMQ consumer). No HTTP port.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: basesource-worker
  namespace: basesource
spec:
  replicas: 2
  selector:
    matchLabels:
      app: basesource-worker
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1    # workers are idempotent — 1 down during rollout is fine
      maxSurge: 1
  template:
    metadata:
      labels:
        app: basesource-worker
    spec:
      securityContext:
        runAsNonRoot: true
        runAsUser: 65532
      containers:
        - name: worker
          image: your-registry/basesource:latest
          args: ["worker"]
          envFrom:
            - configMapRef:
                name: basesource-config
            - secretRef:
                name: basesource-secret
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 256Mi
          # No liveness probe — distroless has no shell or procutils (pgrep unavailable).
          # The worker uses errgroup: any goroutine error propagates and exits the process.
          # Kubernetes restarts the pod automatically on process exit (restartPolicy: Always).
          # A liveness probe is only needed when the process stays alive but is deadlocked;
          # errgroup already handles that by returning the error to the main goroutine.
```

**Why no liveness probe on the worker:**
`gcr.io/distroless/static-debian11` contains only the Go binary and CA certificates — no shell, no `pgrep`, no `test`. An exec-based probe would fail immediately and trigger endless restarts.

The worker is safe without a probe because:
- `errgroup` propagates any goroutine error to `g.Wait()`, which causes `RunE` to return — the process exits and Kubernetes restarts it
- The four goroutines (Kafka relay, RabbitMQ relay, Kafka consumer, RabbitMQ consumer) all handle transient broker failures internally (reconnect loops, retry-on-next-tick) without exiting, so a deadlock restart is not needed in practice

If a liveness check is required in future, the options are:
1. Switch to `distroless/base-debian11` (has glibc + basic tools) — not needed since the binary is `CGO_ENABLED=0`
2. Add a minimal HTTP health endpoint to the worker and use an `httpGet` probe — cleanest long-term solution
3. Use a goroutine that writes a timestamp file periodically and probe with `execAction` against a sidecar — unnecessary complexity

**Note on Kafka consumer group:** Multiple worker pods share `KAFKA_GROUP_ID=todo-worker`. Kafka distributes partition ownership — each domain event is processed by exactly one pod. The RabbitMQ notification consumer is also competing-consumer safe (ACK/NACK per message). Both patterns allow safe horizontal scaling.

## 11. worker/hpa.yaml

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: basesource-worker-hpa
  namespace: basesource
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: basesource-worker
  minReplicas: 2
  maxReplicas: 6
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

Worker max is lower (6 vs 10) because Kafka partition count caps effective Kafka consumer parallelism. Extra replicas beyond the partition count still help with the RabbitMQ consumer and outbox relay load.

## What this demonstrates to the CTO interviewer

| Concept | Evidence in the manifests |
|---|---|
| Deployment vs Service vs Ingress roles | api/{deployment,service,ingress}.yaml |
| Rolling update / zero-downtime deploy | `maxUnavailable: 0, maxSurge: 1` in api deployment |
| Resource requests + limits | `resources:` block in both deployments |
| HPA — horizontal scaling by CPU | api/hpa.yaml, worker/hpa.yaml |
| Readiness probe (remove from LB) vs Liveness probe (restart if stuck) | probes in api deployment |
| ConfigMap vs Secret — no secrets in Git | configmap.yaml, secret.yaml (template) |
| Non-root container security | `securityContext + USER nonroot` in Dockerfile + pod spec |
| api vs worker as separate Deployments | separate Deployment per CLI command |
| Database migration as a one-shot Job | migrate-job.yaml |
