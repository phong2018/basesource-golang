# feature/996 — RabbitMQ + Kafka Dual-Broker Real Test Guide

## Architecture Recap

```
POST /api/v1/todos
        │
        ▼
  TodoUsecase.Create()
        │
        ├── INSERT todos               (same DB transaction)
        ├── INSERT outbox_events       (same DB transaction)
        └── PublishNotification()
                │
                ▼
          RabbitMQ queue: todo.notifications
                │
                ▼
        NotificationConsumer
        → logs "notification task received"   ← RabbitMQ path

[OutboxRelay polls DB every 2s]
        │
        ▼
  KafkaProducer.Publish()
        │
        ▼
  Kafka topic: todo-events
        │
        ▼
  KafkaConsumer
  → logs "domain event received"              ← Kafka path
```

**RabbitMQ** = notification task queue (fire-and-forget, consumed once)
**Kafka** = domain event log (durable, replayable, ordered per aggregate)

## Test Case 1: Create Todo — Full Dual-Broker Flow

### Action

```bash
curl -s -X POST http://localhost:8080/api/v1/todos \
  -H "Content-Type: application/json" \
  -d '{"title":"Buy milk"}' | jq .
```

### Wait for relay cycle

```bash
sleep 3
```

### Verify: both brokers fired

```bash
tail -20 /tmp/996-worker.log
```

Must contain **both** lines:

```
{"msg":"notification task received","to":"admin@example.com","subject":"New Todo Created"}
{"msg":"domain event received","event_type":"todo.created","aggregate_id":"...","event_id":"..."}
```

### Verify: outbox row marked published

```bash
docker compose -f docker/docker-compose.yaml exec db \
  mysql -u appuser -papppass appdb \
  -e "SELECT event_id, event_type, status FROM outbox_events ORDER BY created_at DESC LIMIT 5;"
```

Expected: `status=published` for the new row.

## Test Case 2: Update + Delete Todo — Kafka Only

Update and delete do **not** trigger a RabbitMQ notification, only a Kafka domain event.

### Action


```
{"msg":"domain event received","event_type":"todo.deleted","aggregate_id":"1","event_id":"..."}
```

No `notification task received` line — correct, notifications only fire on create.

## Test Case 3: Kafka Message Inspection

Read raw messages directly from the Kafka topic to confirm payload structure and partition routing.

```bash
docker compose -f docker/docker-compose.yaml exec kafka \
  /opt/kafka/bin/kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic todo-events \
  --from-beginning \
  --max-messages 20
```

Expected: JSON payload per message:

```json
{"event_id":"...","aggregate_id":"1","event_type":"todo.created","payload":"...","created_at":"..."}
```

### Check partition distribution

```bash
docker compose -f docker/docker-compose.yaml exec kafka \
  /opt/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group todo-worker \
  --describe
```

Shows which partition each message landed on and current consumer lag. Same `aggregate_id` always maps to the same partition (Hash balancer by key).

## Test Case 5: RabbitMQ Broker Down — Notification Degrades Gracefully

1. Stop RabbitMQ:

   ```bash
   docker compose -f docker/docker-compose.yaml stop rabbitmq
   ```
2. Create a todo:

   ```bash
   curl -s -X POST http://localhost:8080/api/v1/todos \
     -H "Content-Type: application/json" \
     -d '{"title":"RabbitMQ down test"}' | jq .
   ```
3. Check logs — `notification task received` will NOT appear, but the API still returns 201 and the outbox event is still inserted.
4. Kafka domain event still fires (RabbitMQ outage does not affect Kafka path):

   ```bash
   sleep 3 && tail -5 /tmp/996-worker.log
   # domain event received → still logged
   ```
5. Restart RabbitMQ:

   ```bash
   docker compose -f docker/docker-compose.yaml start rabbitmq
   ```

   ```bash

   docker compose -f docker/docker-compose.yaml exec db \
     mysql -u appuser -papppass appdb \
     -e "SELECT event_type, status FROM outbox_events ORDER BY created_at DESC LIMIT 3;"
   ```

## Test Case 7: Kafka Consumer Group Replay

Simulates a new downstream consumer or a reprocessing scenario.

```bash
# Kill worker
pkill -f basesource-996

# Reset consumer group offsets to earliest
docker compose -f docker/docker-compose.yaml exec kafka \
  /opt/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group todo-worker \
  --topic todo-events \
  --reset-offsets --to-earliest --execute
```

Expected:

```
GROUP        TOPIC        PARTITION  NEW-OFFSET
todo-worker  todo-events  0          0
todo-worker  todo-events  1          0
todo-worker  todo-events  2          0
```

```bash
# Restart worker
/tmp/basesource-996 worker > /tmp/996-worker.log 2>&1 &
sleep 5
cat /tmp/996-worker.log
```

Expected: all past `domain event received` entries replayed — one per historical outbox event.

## Troubleshooting

### `notification task received` never appears

- Check RabbitMQ is healthy: `docker compose ps rabbitmq`
- Check exchange exists: `rabbitmqctl list_exchanges`
- Check queue binding: Management UI → Queues → `todo.notifications` → Bindings

### `domain event received` never appears after first run

Consumer group already has committed offsets. Reset:

```bash
pkill -f basesource-996
docker compose -f docker/docker-compose.yaml exec kafka \
  /opt/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group todo-worker --topic todo-events \
  --reset-offsets --to-earliest --execute
```

### Offset reset fails — "group is PreparingRebalance"

Worker is still running. Kill first:

```bash
pkill -f basesource-996
# then run reset command
```

### outbox_events stuck at `status=pending`

Kafka is unreachable. The relay retries automatically every 2s when Kafka recovers.

```bash
docker compose -f docker/docker-compose.yaml ps kafka
docker compose -f docker/docker-compose.yaml restart kafka
```
