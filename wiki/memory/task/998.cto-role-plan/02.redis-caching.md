# Task 02 — Redis Caching Layer

## Requirement
CTO must demonstrate: Redis — TTL strategies, cache invalidation patterns.

## What the source currently has
- None. Every `GetByID` and `List` hits MySQL directly.

## What to add

### Pattern: Cache-Aside (Read-Through, Write-Invalidate)
```
GetByID(id):
  1. hit Redis — cache HIT → return
  2. cache MISS → query MySQL → SET in Redis with TTL → return

Create/Update/Delete:
  → write to MySQL (inside transaction)
  → DEL cache key (invalidation, not update)
```

### 1. Domain service interface
- `internal/domain/service/cache_client.go`
```go
type ICacheClient interface {
    Get(ctx context.Context, key string) (string, error)
    Set(ctx context.Context, key string, value string, ttl time.Duration) error
    Del(ctx context.Context, keys ...string) error
}
```

### 2. Infrastructure implementation
- `internal/infrastructure/cache/redis_client.go`
- Uses `redis/go-redis/v9`
- Config: `RedisAddr`, `RedisPassword`, `RedisDB` in `config.go`

### 3. Cache key convention
- `todo:<id>` — single item
- `todo:list:<hash-of-filter-and-page>` — list results (optional, higher complexity)

### 4. TTL strategy
- Single item: 5 minutes (low churn, high read)
- List: 30 seconds (invalidated on any write) or skip caching lists (simpler)
- Configurable via `CACHE_TODO_TTL` env

### 5. Usecase integration
- `todo_usecase_impl.go` receives `ICacheClient` as optional dep (nil-safe)
- `GetByID`: check cache first
- `Create`, `Update`, `Delete`: invalidate after successful tx commit

### 6. Wire in container
- `container/container.go` — init `redis_client`, pass to usecase

## Files to create / modify
- `internal/domain/service/cache_client.go`
- `internal/infrastructure/cache/redis_client.go`
- `internal/usecase/todo_usecase_impl.go` — cache-aside logic
- `config/config.go` — `RedisConfig`
- `.env.example` — `REDIS_ADDR`, `REDIS_PASSWORD`, `REDIS_DB`, `CACHE_TODO_TTL`
- `container/container.go` — wire cache client
- `docker/docker-compose.yaml` — add Redis service

## Demonstrates to CTO interviewer
- Cache-aside pattern (most common, safe with eventual consistency)
- TTL reasoning (not too short = no benefit, not too long = stale data)
- Invalidation on write (correct approach vs update-on-write)
- Clean Architecture respected: `ICacheClient` in domain/service, redis impl in infrastructure
