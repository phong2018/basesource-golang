# Task 09 — Trade-offs: Business vs Performance vs Technical Complexity

## Requirement
CTO must demonstrate: the ability to reason about and document architectural trade-offs — not just technical ones (consistency, latency, scaling) but also **business trade-offs** (customer requirements vs performance, customer requirements vs technical complexity). A CTO who can only talk about tech-vs-tech is an engineer. A CTO talks about tech-vs-business.

---

## Category 1: Business (Customer Requirement) vs Performance

The customer always wants everything: instant response, real-time data, zero downtime. The CTO's job is to explain what "instant" actually costs and propose the right balance.

### Trade-off 1A: Strong consistency vs response latency

**Business requirement:** "When I create a todo, I must see it immediately in the list."

| Approach | Consistency | p99 Latency | Cost |
|---|---|---|---|
| Read from primary DB (no cache) | Strong — always fresh | Higher under load | Always hits DB |
| Read from cache (TTL 5 min) | Eventual — up to 5 min stale | Low, consistent | Redis infra cost |
| Read from read replica | Eventual — replication lag ~ms | Lower than primary | RDS Multi-AZ cost |

**CTO decision for this product:**
- Todo list is low-stakes (stale by 30s is acceptable) → use cache for `List`
- `GetByID` after `Create` is called immediately by the same user → invalidate on write so they see their own change
- **Rule:** "read-your-own-writes" consistency matters; global strong consistency does not

**How to present to customer (non-technical):**
> "After you create a todo, you'll see it immediately. Other users' lists refresh within 30 seconds. This lets us handle 10× more traffic without extra server cost."

---

### Trade-off 1B: Real-time notification vs system reliability

**Business requirement:** "Users must receive a notification email immediately when a todo is assigned to them."

| Approach | Delivery guarantee | API latency impact | Reliability |
|---|---|---|---|
| Synchronous HTTP call in request | At-most-once (if API fails, no retry) | +external HTTP latency (50–500ms) | Notification service outage = API 500 |
| Fire-and-forget goroutine | At-most-once, no retry, lost on crash | Zero | Process crash = lost notification |
| Message broker + consumer | At-least-once, retry on failure | Zero | Survives crashes, broker outage |
| Outbox pattern + broker | Exactly-once (with idempotent consumer) | Zero | Strongest guarantee |

**CTO decision framework:**
- Is the notification **critical** (OTP, payment receipt)? → Outbox + broker (Task 03)
- Is it **important but not critical** (assignment notification)? → Broker + retry, accept occasional duplicate
- Is it **informational** (activity digest)? → Fire-and-forget, loss acceptable

**How to present to customer:**
> "We can guarantee every notification is delivered, but it adds 2 weeks of engineering and requires additional infrastructure. For your current scale, fire-and-forget covers 99.9% of cases. We can upgrade when you hit issues."

---

### Trade-off 1C: Pagination vs "show all" UX

**Business requirement:** "The customer wants to see all todos on one page, no pagination."

**Reality:** At 10,000 todos, `SELECT * FROM todos` = 10MB response, 2s render, browser freeze.

| Approach | UX | Performance | Risk |
|---|---|---|---|
| No pagination (show all) | Simple for user | Breaks at scale | OOM, timeout |
| Cursor-based pagination | Best UX for infinite scroll | O(1) per page, index-friendly | More complex to implement |
| Offset pagination (current) | Familiar UX | O(n) for deep pages | Slow at page 500+ |
| Virtual scroll (frontend) | Feels like "show all" | Load on demand | Requires frontend work |

**CTO decision:**
- Implement offset pagination now (simple, works until ~100k rows)
- Enforce max limit = 100 in usecase (prevents accidental unbounded query)
- Document migration path to cursor-based when needed
- Propose virtual scroll to frontend team as UX compromise

**How to present to customer:**
> "We'll add pagination with a 'load more' button. It looks like you're seeing all todos, but we only load 20 at a time. This keeps the app fast even with thousands of items."

## Category 2: Business (Customer Requirement) vs Technical Complexity

The customer wants features. The CTO decides when the simplest solution is correct and when complexity is justified. Premature complexity is as dangerous as insufficient complexity.

### Trade-off 2A: Simple CRUD vs Event-Driven Architecture

**Business requirement:** "We need to track all changes to todos for audit purposes."

| Approach | Complexity | Operational cost | When it fits |
|---|---|---|---|
| Write audit log in same transaction (current) | Low — 1 extra INSERT | Zero extra infra | ✅ Current scale |
| Event sourcing (store events, derive state) | High — new paradigm | High — event store infra | Regulatory audit, time-travel queries |
| CDC (Debezium reads MySQL binlog) | Medium — infra only | Medium — Kafka + Debezium | When app can't be modified |

**CTO decision:**
- Current requirement: audit trail for compliance → simple INSERT in same tx is correct
- Do NOT introduce event sourcing unless the business needs time-travel queries or event replay
- **Rule:** complexity must be justified by a specific business need, not technical elegance

**How to present to customer:**
> "We track every create/update/delete in an audit table. It's simple and reliable. If you later need to 'replay' history or undo actions, we can discuss event sourcing — that's a bigger project."

---

### Trade-off 2B: Microservices vs Monolith

**Business requirement:** "We want to scale the notification service independently because it's slower than the todo service."

| Approach | Team complexity | Deployment complexity | When it fits |
|---|---|---|---|
| Monolith (current) | Low | Low (single binary) | ✅ Team < 10 engineers |
| Modular monolith | Low-Medium | Low | Team 10–30, clear domain boundaries |
| Microservices | High (distributed tracing, network failures, versioning) | High (k8s, service mesh) | Team 30+, independent deploy cadence |

**CTO decision framework:**
- Current team size (15–20 engineers, 1 squad) → modular monolith is the right answer
- Extracting notification into a microservice adds: network latency, distributed tracing, versioned API contracts, separate CI/CD pipeline, on-call rotation
- The scaling problem (notification is slow) is solved cheaper by: async messaging (Task 03) within the monolith

**How to present to customer:**
> "Breaking out notification into a separate service adds 6–8 weeks of infrastructure work and ongoing operational complexity. Instead, we'll make it async — the API responds instantly and notification is sent in the background. Same result, simpler system."

---

### Trade-off 2C: Generic platform vs domain-specific optimization

**Business requirement:** "Build a todo app that can later become a project management platform."

| Approach | Flexibility | Complexity | Delivery speed |
|---|---|---|---|
| Build for current requirements only | Low | Low | Fast |
| Build generic task engine with plugins | High | Very high | Slow (3× estimate) |
| Build current + clean boundaries for extension | Medium | Medium | ✅ Correct balance |

**CTO decision:**
- Clean Architecture already gives extension points (add new usecase, new handler, new repo)
- Do NOT build a generic plugin system speculatively
- **Rule:** YAGNI (You Aren't Gonna Need It) — build for today's requirements, with architecture that doesn't block tomorrow's

**How to present to customer:**
> "We're building the todo features you need now. The code structure makes it straightforward to add project management features later — we're not locked in. But building a generic platform today would take 3× longer and delay your launch."

## Demonstrates to CTO interviewer
- **Business vs performance:** can translate technical constraints into customer-friendly language; knows when "good enough" is the right answer
- **Business vs complexity:** pushes back on over-engineering; applies YAGNI; knows the real cost of microservices
- **Decision documentation:** uses ADRs — shows leadership thinking, not just coding
- **Read-your-own-writes:** knows the difference between strict consistency and practical consistency that satisfies the user
- **Monolith-first:** can defend a monolith to stakeholders who want microservices, with concrete cost/benefit reasoning
