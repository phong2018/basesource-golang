## Concept 1: Pub/Sub Pattern

### What it is

Producer publishes events to a **topic/exchange**. Consumers subscribe independently — each gets their own copy of the event. Producer does not know who consumes or how many consumers exist.

```
Producer (todo usecase)
    │
    ▼
[Exchange: todo.events]  ← RabbitMQ fanout/topic exchange
    ├──▶ Queue: notification-service   → sends email
    ├──▶ Queue: search-indexer         → indexes todo for search
    └──▶ Queue: analytics-service      → records event for reporting
```

**vs Point-to-Point (queue only):** one message → one consumer. Pub/sub: one message → N independent consumers.

### Why use pub/sub over direct HTTP call

|                   | Direct HTTP                       | Pub/Sub                                        |
| ----------------- | --------------------------------- | ---------------------------------------------- |
| Coupling          | Producer must know consumer URL   | Producer knows only topic name                 |
| Failure isolation | Consumer down = producer gets 500 | Consumer down = messages queue up, retry later |
| Adding consumers  | Must modify producer              | Zero producer change                           |
| API latency       | Blocked by slowest consumer       | Producer returns immediately                   |

### Implementation in this source

- Binding keys: `todo.created`, `todo.updated`, `todo.deleted`
- Each downstream service binds its own durable queue to the exchange

## Concept 2: Consumer Groups

### What it is

Multiple instances of the **same consumer** sharing work from one queue. Each message is processed by **exactly one** instance in the group — load balancing, not broadcast.

```
Queue: notification-service (1000 pending messages)
    ├──▶ Worker pod 1  (processes message 1, 4, 7, ...)
    ├──▶ Worker pod 2  (processes message 2, 5, 8, ...)
    └──▶ Worker pod 3  (processes message 3, 6, 9, ...)
```

**RabbitMQ:** consumer groups = multiple consumers connected to the same queue. RabbitMQ round-robins messages across them.

**Kafka:** consumer groups are explicit — each group has a `group.id`. Each partition is assigned to exactly one consumer in the group. Adding consumers scales up to `num_partitions` maximum.

### Why it matters

- Horizontal scaling of consumers without producer change
- `prefetch_count = 1` in RabbitMQ ensures fair dispatch (don't send next message to a worker until it acks the current one)

```go
// infrastructure/messaging/rabbitmq_consumer.go
ch.Qos(
    1,     // prefetch count — one message at a time per consumer
    0,     // prefetch size — no limit
    false, // global
)
```

### Implementation in this source

- `cmd/worker/cmd.go` — starts N goroutines each consuming from the same queue
- Demonstrates: run 3 worker pods → throughput triples, each message processed once
- `ACK` on success, `NACK + requeue=false` on unrecoverable error (goes to dead-letter queue)

## Concept 3: Ordering Guarantees

### The problem

Events for the same entity must be processed in order. If `todo.updated` is processed before `todo.created`, the consumer crashes or produces wrong state.

### RabbitMQ ordering

- **Per-queue FIFO** — messages in a single queue are delivered in publish order
- **Breaks when:** multiple consumers on the same queue (consumer 1 takes msg 1, consumer 2 takes msg 2 — if consumer 2 finishes first, ordering is lost from the consumer's perspective)
- **Fix:** for strict ordering, use **one consumer per queue** OR partition messages by entity ID into separate queues

```
Queue: todos.updates.shard-0  → consumer 0 (todo IDs % 3 == 0)
Queue: todos.updates.shard-1  → consumer 1 (todo IDs % 3 == 1)
Queue: todos.updates.shard-2  → consumer 2 (todo IDs % 3 == 2)
```

### Kafka ordering

- **Per-partition ordering** — all messages with the same partition key land on the same partition, processed by the same consumer in the group
- **Producer sets partition key = entity ID:**

```go
producer.Produce(&kafka.Message{
    TopicPartition: kafka.TopicPartition{Topic: &topic, Partition: kafka.PartitionAny},
    Key:   []byte(strconv.Itoa(int(todoID))),  // same todo → same partition
    Value: payload,
})
```

- All events for `todo:42` go to the same partition → same consumer → in-order processing guaranteed

### RabbitMQ vs Kafka — when to choose which

| Factor                 | RabbitMQ                                        | Kafka                                             |
| ---------------------- | ----------------------------------------------- | ------------------------------------------------- |
| Throughput             | ~50k msg/s                                      | >1M msg/s                                         |
| Message retention      | Until consumed (or TTL)                         | Configurable retention (days/weeks) — can replay |
| Ordering               | Per-queue FIFO (breaks with multiple consumers) | Per-partition (strong, scales)                    |
| Exactly-once           | Not natively                                    | Exactly-once semantics (v0.11+, complex config)   |
| Operational complexity | Low                                             | High (ZooKeeper/KRaft, partition rebalancing)     |
| Best for               | Task queues, RPC, moderate scale                | Event streaming, audit log, analytics, replay     |
| Setup for this project | `docker run rabbitmq:management`              | Needs Kafka + ZooKeeper or Confluent              |

**Decision for this base source:** RabbitMQ — simpler ops, sufficient for the demonstrated scale. Document migration path to Kafka.

## Concept 4: Reliable Delivery — Outbox Pattern

Direct publish after DB commit has a dual-write problem:

```
tx.Commit()          ← success
broker.Publish()     ← CRASH here → event lost forever
```

### Outbox pattern (fix)

```
Inside transaction:
  1. INSERT todos
  2. INSERT audit_logs
  3. INSERT outbox_events (type, payload, status=pending)  ← same tx, atomic

Background relay worker (separate goroutine / cmd):
  → SELECT * FROM outbox_events WHERE status = 'pending' LIMIT 100
  → broker.Publish(event)
  → UPDATE outbox_events SET status = 'published'
```

- If app crashes between INSERT and relay → row stays `pending` → relay retries on restart
- At-least-once delivery (consumer must be idempotent — check `event_id` before processing)
