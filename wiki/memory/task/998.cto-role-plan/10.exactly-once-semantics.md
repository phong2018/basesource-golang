---

## The Three Delivery Guarantees

| Guarantee | How | Risk | Use when |
|---|---|---|---|
| At-most-once | Fire-and-forget, no retry | Message lost on crash | Logs, metrics — loss acceptable |
| At-least-once | Retry until ACK | Duplicate processing | Most business events — make consumer idempotent |
| Exactly-once | Idempotent producer + transactional consumer | Complex, performance cost | Payment, financial settlement, inventory deduction |

**RabbitMQ**: supports at-most-once and at-least-once only. No exactly-once.
**Kafka**: supports exactly-once via **idempotent producer** + **transactional API** (v0.11+).

---

## How Kafka Exactly-Once Works

### Step 1: Idempotent Producer
Each message gets a sequence number. Kafka broker deduplicates retries from the same producer session:
```go
producer, _ := kafka.NewProducer(&kafka.ConfigMap{
    "bootstrap.servers":     cfg.KafkaBrokers,
    "enable.idempotence":    true,           // dedup retries on broker side
    "acks":                  "all",          // wait for all replicas
    "max.in.flight.requests.per.connection": 5,
})
```
Prevents duplicate from: producer retries after timeout but broker actually committed the first write.

### Step 2: Transactional Producer
Atomic publish across multiple topics/partitions — all succeed or all fail:
```go
producer.InitTransactions(ctx)
producer.BeginTransaction()

producer.Produce(&kafka.Message{Topic: "todo.events", ...})
producer.Produce(&kafka.Message{Topic: "audit.events", ...})

if err != nil {
    producer.AbortTransaction(ctx)   // nothing committed
} else {
    producer.CommitTransaction(ctx)  // both committed atomically
}
```

### Step 3: Transactional Consumer (read-process-write atomic)
The hardest part: commit the offset AND the DB write atomically:
```go
// Consume message → process → write result to DB → commit offset
// If we commit offset before DB write: message "processed" but result lost
// If we commit offset after DB write: DB write OK but crash before offset commit → reprocessed

// Solution: write offset to DB in the same transaction as the business data
tx.Exec("INSERT INTO processed_events (event_id) VALUES (?)", msg.ID)  // idempotency key
tx.Exec("UPDATE accounts SET balance = balance - ? WHERE id = ?", amount, accountID)
tx.Commit()
// offset committed separately — if crash here, message reprocessed but DB insert deduplicates
```

## Exactly-Once in the Outbox Pattern (Practical)

For this base source (RabbitMQ, not Kafka), exactly-once is approximated via **idempotent consumer**:

```go
// consumer: check processed before acting
func (c *consumer) handle(ctx context.Context, msg Message) error {
    // idempotency check — skip if already processed
    exists, _ := c.processedRepo.Exists(ctx, msg.ID)
    if exists {
        return nil  // duplicate — ack and skip
    }

    return c.db.WithinTransaction(ctx, func(ctx context.Context) error {
        // business logic
        if err := c.todoRepo.UpdateStatus(ctx, msg.Payload.TodoID, "notified"); err != nil {
            return err
        }
        // mark as processed in same tx — atomic
        return c.processedRepo.Create(ctx, msg.ID)
    })
}
```

**Table:** `processed_events(event_id VARCHAR PRIMARY KEY, processed_at DATETIME)` — unique constraint prevents double-insert even under race condition.

## When to Use Each Approach

| Scenario | Approach |
|---|---|
| Todo notification email | At-least-once + idempotent consumer (deduplicate by event_id) |
| Payment deduction | Exactly-once (Kafka transactional) OR at-least-once + idempotency key in DB |
| Inventory reservation | Exactly-once OR optimistic locking with version check |
| Audit log append | At-least-once is fine — duplicate audit row is better than missing one |
| Analytics event count | At-most-once acceptable — approximate counts are fine |

---

## Trade-off: Exactly-Once vs Performance

| | At-least-once | Exactly-once |
|---|---|---|
| Throughput | High | ~30–50% lower (2-phase commit overhead) |
| Latency | Low | Higher (waits for all replicas + transaction coordinator) |
| Complexity | Low | High (idempotent producer + transactional consumer + offset management) |
| When justified | Most cases | Financial transactions, inventory, settlement |

**CTO decision:** default to at-least-once + idempotent consumer. Use Kafka exactly-once only for financial flows where duplicate processing has monetary consequence.
