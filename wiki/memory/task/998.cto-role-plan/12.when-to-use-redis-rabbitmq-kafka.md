# When to Use Redis vs RabbitMQ vs Kafka

## The Core Question

Before choosing a tool, ask two questions:

1. **Is the user waiting for the result?** → real-time or async?
2. **Can we afford to lose the message if something crashes?** → durability required?

---

## Quick Decision Table

| Situation | Tool |
|---|---|
| User must see result instantly (<100ms) | Redis Pub/Sub + WebSocket |
| Background work, delay acceptable (2-30s) | RabbitMQ |
| High volume event stream (>50k msg/s), replay needed | Kafka |
| Cache a DB query result | Redis (cache, not pub/sub) |
| Distributed rate limiting / session storage | Redis |
| Task queue with retry and dead-letter | RabbitMQ |
| Audit log stream, analytics pipeline | Kafka |
| Financial settlement, exactly-once required | Kafka |

---

## Redis

### What it is
In-memory data store. Sub-millisecond read/write. Supports pub/sub, sorted sets, streams, distributed locks.

### Use case 1: Real-time push (Pub/Sub + WebSocket)

**When:** user is looking at the screen and must see the update immediately.

```
User A sends chat message
    → API saves to DB
    → Redis PUBLISH channel:chat:room-42 "{message}"   ← instant

WebSocket server (subscribed to channel:chat:room-42)
    → receives from Redis in <1ms
    → pushes to User B's open WebSocket connection
    → User B sees message appear in <10ms
```

Real examples:
- Chat messages
- "User is typing..." indicator
- Live todo board — team member creates a todo, everyone sees it appear
- Live order tracking — "driver is 2km away"
- Notification bell badge — red dot appears immediately

**Why not RabbitMQ here:** RabbitMQ adds 2s+ outbox delay. User would see the message 2 seconds late — unacceptable for chat.

**Why not Kafka here:** Kafka is not designed for pushing to individual WebSocket connections. It is a log, not a push mechanism.

### Use case 2: Caching

**When:** same data is read many times, DB query is slow, slight staleness is acceptable.

```
GetByID(id=42):
    1. Redis GET todo:42          → HIT: return in 0.1ms
    2. MISS: MySQL SELECT         → 5ms
       Redis SET todo:42 TTL=5min
       return
```

Real examples:
- Todo detail page (same todo fetched by many users)
- User profile (read 1000× per second, updated rarely)
- Product catalog on e-commerce (changes daily, read millions/day)
- Config/feature flags (change rarely, read on every request)

**Why not RabbitMQ/Kafka here:** they are message brokers, not stores. You cannot ask a broker "give me the current value of todo:42".

### Use case 3: Distributed rate limiting / session / lock

**When:** app runs on multiple pods and needs shared state.

```
# rate limiting: all pods share one counter
INCR rate:ip:192.168.1.1
EXPIRE rate:ip:192.168.1.1 60

# distributed lock: only one pod processes a job at a time
SET lock:settlement:2026-05-23 "pod-3" EX 30 NX
```

Real examples:
- Rate limiting across 10 API pods (in-memory limiter gives each pod its own quota → 10× the effective limit)
- Session storage for stateful auth
- Preventing duplicate cron job execution across pods
- Inventory reservation hold (lock item for 10 minutes during checkout)

### Redis limitations
- **No durability by default** — data lives in memory. If Redis crashes without persistence configured (`AOF`/`RDB`), data is lost
- **Pub/Sub: no message retention** — if the subscriber (WebSocket server) is down when a message is published, the message is gone forever. Fine for chat (reconnect and reload history from DB), not fine for email notifications
- **Not a queue** — Redis Streams can act like a queue but it is more complex to operate than RabbitMQ

## RabbitMQ

### What it is
Message broker with durable queues, routing (exchange/binding), ACK/NACK, dead-letter queues. Designed for reliable task delivery.

### Use case 1: Async background work (fire and forget from API perspective)

**When:** the API should return immediately; the work happens in the background; the user does not watch it happen.

```
POST /api/v1/todos → 201 (returned immediately)
    ↓ (2 seconds later, outbox relay)
RabbitMQ → consumer → sends welcome email
```

Real examples:
- Send email / SMS notification after user action
- Generate PDF report (takes 5 seconds — do not block the API)
- Resize uploaded image to thumbnails
- Sync data to a third-party CRM after a sale
- Trigger a webhook to a customer's endpoint

**Why not Redis Pub/Sub here:** Redis pub/sub has no durability. If the consumer pod is restarting when the message is published, the message is lost. For email, losing a message = user never gets their receipt.

**Why not Kafka here:** Kafka works but adds operational complexity (ZooKeeper/KRaft, partition management) for a simple task queue. RabbitMQ is simpler to operate at moderate scale.

### Use case 2: Task queue with retry and dead-letter

**When:** a job may fail and needs automatic retry with backoff, and failed jobs need inspection.

```
Consumer processes message → fails (e.g. email service down)
    → NACK → goes to dead-letter queue (DLQ)
    → retry worker picks up from DLQ after 60s
    → if fails 3 times → alert team
```

Real examples:
- Payment webhook delivery (must retry until success)
- Outbound API calls to unreliable third parties
- Data migration jobs that may hit DB locks

### Use case 3: Work distribution across multiple consumers (consumer group)

**When:** one queue has more messages than one consumer can handle → add more consumers.

```
Queue: image-resize (10,000 pending jobs)
    → Consumer pod 1 (processes ~3,333 jobs)
    → Consumer pod 2 (processes ~3,333 jobs)
    → Consumer pod 3 (processes ~3,333 jobs)
```

Horizontal scaling: add a pod → throughput increases. Remove a pod → remaining pods share the load. No code change.

### RabbitMQ limitations
- **Throughput ceiling ~50k msg/s** — sufficient for most apps; not sufficient for high-volume event streaming
- **No message replay** — once a message is ACKed it is gone. Cannot go back and reprocess last 7 days of events
- **Ordering breaks with multiple consumers** — per-queue FIFO only holds with a single consumer. Multiple consumers on the same queue = messages processed out of order
- **No built-in exactly-once** — at-least-once only (use idempotent consumers to handle duplicates)

## Kafka

### What it is
Distributed event log. Messages are written to partitions and retained for days/weeks. Consumers read at their own pace by tracking an offset. Designed for high-throughput, replayable event streams.

### Use case 1: High-volume event streaming

**When:** volume exceeds RabbitMQ's throughput, or you need multiple independent consumer groups reading the same events at different speeds.

```
Payment service publishes 500,000 transactions/minute to topic: payments

Consumer group A (fraud detection):    reads at its own pace, lags 30s — fine
Consumer group B (settlement engine):  reads at its own pace, lags 5s  — fine
Consumer group C (analytics):          reads at its own pace, reads yesterday's data — fine

All three groups read the SAME events independently.
RabbitMQ cannot do this — once a message is consumed it is gone.
```

Real examples:
- Payment transaction stream (fraud, settlement, analytics all consume same events)
- User activity stream (clickstream → recommendation engine + analytics + A/B test logging)
- IoT sensor data (millions of devices publishing metrics)
- Microservices event sourcing at scale

### Use case 2: Event replay

**When:** you need to reprocess historical events — new consumer added, bug fixed, data migration.

```
Bug found in fraud detection logic on 2026-05-01
    → fix the bug
    → reset consumer group offset to 2026-05-01
    → replay all payment events since then
    → fraud scores recalculated correctly
```

RabbitMQ cannot do this — consumed messages are deleted. Kafka retains messages for a configurable period (default 7 days, can be indefinite).

Real examples:
- Replay events to rebuild a read model (CQRS)
- Backfill a new analytics table with historical data
- Recover from a downstream service bug without data loss

### Use case 3: Exactly-once financial processing

**When:** duplicate processing has monetary consequence (double charge, double payout).

```
Kafka producer: idempotent (broker deduplicates retries)
Kafka transaction: atomic publish across multiple topics
Consumer: reads offset + processes + commits offset in one atomic step
```

Real examples:
- Payment deduction (cannot deduct twice)
- Inventory reservation (cannot reserve the same item twice)
- Settlement calculation (cannot include a transaction twice)

### Use case 4: Audit log / event sourcing

**When:** every state change must be permanently recorded and replayable.

```
Topic: todo.events (retention: forever)
    todo.created   id=1  title="Buy milk"   2026-05-23T10:00
    todo.updated   id=1  done=true          2026-05-23T11:00
    todo.deleted   id=1                     2026-05-23T12:00

Replay from beginning → reconstruct full history of every todo
```

### Kafka limitations
- **Operational complexity** — requires ZooKeeper (legacy) or KRaft, partition rebalancing, consumer lag monitoring
- **Not designed for task queues** — no per-message ACK/NACK, no dead-letter queue out of the box
- **Latency higher than RabbitMQ** — optimised for throughput, not individual message latency
- **Overkill for small scale** — if you publish <10k msg/s, RabbitMQ is simpler and sufficient

## Side-by-Side Comparison

| Factor | Redis Pub/Sub | RabbitMQ | Kafka |
|---|---|---|---|
| Primary purpose | Real-time push, cache, lock | Reliable task queue | High-throughput event log |
| Message durability | No (in-memory, lost if no subscriber) | Yes (durable queues) | Yes (disk, configurable retention) |
| Message replay | No | No | Yes |
| Throughput | Very high (in-memory) | ~50k msg/s | >1M msg/s |
| Latency | <1ms | Low (ms) | Higher (batch-optimised) |
| Ordering | No guarantee | Per-queue FIFO (breaks with multiple consumers) | Per-partition (strong) |
| Exactly-once | No | No (at-least-once) | Yes (v0.11+, complex) |
| Consumer groups | No (all subscribers get all messages) | Yes (competing consumers on same queue) | Yes (explicit group.id, partition assignment) |
| Dead-letter queue | No | Yes (built-in) | No (must build manually) |
| Operational complexity | Low | Low-Medium | High |
| Best for | Chat, live updates, cache | Email, PDF, webhooks, task queues | Payments, analytics, event sourcing |

## Real-World Example: Todo App Using All Three

```
User creates a todo
    │
    ├─ Redis Pub/Sub + WebSocket
    │      → all team members see the new todo appear on their board instantly (<10ms)
    │
    ├─ RabbitMQ (outbox pattern, async)
    │      → sends email notification to assignee (2s delay, acceptable)
    │      → generates activity digest for daily summary email
    │
    └─ Kafka (if at scale)
           → appends todo.created event to audit log (retained forever)
           → analytics pipeline counts todos created per user per day
           → replay last 30 days to rebuild reporting dashboard
```

Each tool does one job it is best at. They coexist in the same system.

---

## Interview Answer: "When would you choose each?"

> **Redis:** when the user is watching the screen and needs to see the update in under 100ms, or when I need to cache a frequently-read value to reduce DB load.
>
> **RabbitMQ:** when the work is async (user does not watch it), reliability matters (cannot lose the message), and volume is moderate. Email, PDF generation, webhooks. The outbox pattern gives at-least-once delivery even if the broker goes down temporarily.
>
> **Kafka:** when I need high throughput (>50k msg/s), multiple independent consumer groups reading the same events, message replay for debugging or backfill, or exactly-once semantics for financial transactions. The operational cost is higher so I default to RabbitMQ and migrate to Kafka only when a specific need justifies it.
