## Pillar 1: Caching

> Full implementation detail in [02.redis-caching.md](02.redis-caching.md). This section explains the *performance reasoning*.

### Cache-aside pattern
- `GetByID`: Redis HIT → return (no DB touch, ~0.1ms vs ~5ms DB round-trip)
- `Create/Update/Delete`: DEL cache key after commit (invalidation, not update)

### What caching solves
| Problem | Without cache | With cache |
|---|---|---|
| Hot-path reads (same todo fetched 1000×/s) | 1000 DB queries/s | 1 DB query, 999 Redis hits |
| DB connection exhaustion | Pool saturates under read spike | Reads absorbed by Redis |
| p99 latency spike | Follows DB query time | Flat — cache hits are consistent |

### TTL strategy reasoning
- Short TTL (5 min for single item): stale window is acceptable for todo reads; eviction prevents unbounded memory growth
- Do NOT cache writes: write-through doubles write latency and is complex to keep consistent

## Pillar 2: Database — SQL Pooling + Query Optimization

> Full implementation detail in [06.database-at-scale.md](06.database-at-scale.md). This section explains the *performance reasoning*.

### Connection pool tuning
```go
db.SetMaxOpenConns(25)          // never open more than 25 connections
db.SetMaxIdleConns(10)          // keep 10 warm — no handshake cost on reuse
db.SetConnMaxLifetime(5 * time.Minute)   // recycle before MySQL kills idle conn
db.SetConnMaxIdleTime(1 * time.Minute)   // release idle conns sooner under low load
```

**Why defaults are wrong:**
- Default `MaxOpenConns = 0` (unlimited) → 500 pods × unlimited = MySQL OOM under burst
- Default `MaxIdleConns = 2` → constant connection churn, each new conn costs ~1ms TCP handshake

**Sizing formula:** `MaxOpenConns = (num_cpu_cores × 2) + num_spindle_disks`
For RDS db.t3.medium (2 vCPU, SSD): 25 is a safe starting point per pod.

### Query optimization
- **Explicit column SELECT** — no `SELECT *`: avoids transferring unused columns; allows covering index (index-only scan, no heap fetch)
- **Indexes on filter columns** — `done`, `created_at DESC`, `(entity, entity_id)` on audit_logs
- **Pagination cap** — enforce `Limit ≤ 100` in usecase; unbounded `SELECT` is an OOM vector
- **Context timeout** — `context.WithTimeout(ctx, 3s)` on queries; slow query holds a pool connection and delays all other requests

### EXPLAIN walkthrough (document in wiki)
```sql
EXPLAIN SELECT id, title, done FROM todos WHERE done = false ORDER BY created_at DESC LIMIT 10;
-- type: ref (uses idx_todos_done) → good
-- vs type: ALL → full table scan → add index
```

## Pillar 3: Concurrency

### What the source currently does
- Notification: `go func() { _ = u.notifier.Send(...) }()` — fire-and-forget goroutine (uncontrolled)
- No parallel fan-out anywhere
- No goroutine leak protection

### What to add

#### 3a. `errgroup` for parallel independent calls
When usecase needs to fetch multiple independent resources, run them in parallel:

```go
// Example: enrich todo response with both user info and tag list (two separate DB calls)
import "golang.org/x/sync/errgroup"

g, gctx := errgroup.WithContext(ctx)

var user *model.User
var tags []*model.Tag

g.Go(func() error {
    var err error
    user, err = u.userRepo.GetByID(gctx, todo.UserID)
    return err
})
g.Go(func() error {
    var err error
    tags, err = u.tagRepo.ListByTodo(gctx, todo.ID)
    return err
})

if err := g.Wait(); err != nil {
    return nil, apperror.Internal(err)
}
```
**Benefit:** two 5ms DB calls become 5ms total instead of 10ms.

#### 3b. Worker pool for bounded concurrency
When processing a batch (e.g. bulk send notifications), unbounded goroutines exhaust memory:

```go
// pkg/worker/pool.go
func RunWithPool(ctx context.Context, jobs []func() error, workers int) error {
    sem := make(chan struct{}, workers)
    g, gctx := errgroup.WithContext(ctx)
    for _, job := range jobs {
        job := job
        sem <- struct{}{}
        g.Go(func() error {
            defer func() { <-sem }()
            select {
            case <-gctx.Done():
                return gctx.Err()
            default:
                return job()
            }
        })
    }
    return g.Wait()
}
```
**Use case:** batch-notify 10,000 users — run 50 concurrent goroutines, not 10,000.

#### 3c. Goroutine leak prevention
Current fire-and-forget notification goroutine has no leak protection:
```go
// ❌ current — leaks if ctx is cancelled mid-flight and goroutine blocks forever
go func() { _ = u.notifier.Send(ctx, n) }()

// ✅ fixed — goroutine respects context cancellation
go func() {
    sendCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    if err := u.notifier.Send(sendCtx, n); err != nil {
        slog.ErrorContext(ctx, "notification failed", "error", err)
    }
}()
```
Use `context.Background()` (not the request ctx) so the goroutine is not cancelled when the HTTP response returns.

#### 3d. Graceful shutdown — drain in-flight requests
```go
// cmd/api/cmd.go
quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
<-quit

shutdownCtx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()
if err := server.Shutdown(shutdownCtx); err != nil {
    slog.Error("server shutdown error", "error", err)
}
```
**Why:** Kubernetes sends `SIGTERM` before killing the pod. Without graceful shutdown, in-flight requests get TCP RST → client errors during every rolling deploy.

### Concurrency safety rules (document in wiki)
| Rule | Reason |
|---|---|
| Never share mutable state between goroutines without a mutex or channel | Data race → undefined behavior |
| Always pass a bounded context to goroutines doing I/O | Prevents leaks when parent request is cancelled |
| Use `errgroup` not bare `go` when you need to collect results | `go` has no error return; manual WaitGroup is error-prone |
| Use worker pool (semaphore) for fan-out over user-controlled input | Prevents goroutine explosion on large inputs |

## Pillar 4: How to Measure & Test Performance

Performance work without measurement is guessing. Every optimization must be: **measure → identify bottleneck → fix → measure again**.

### Step 1: Define performance targets first

Before measuring, agree on what "good" means. Without targets, there is no pass/fail.

| Metric | Target example | Why |
|---|---|---|
| p50 response time | < 50ms | Median user experience |
| p99 response time | < 200ms | Worst-case for 99% of users |
| Throughput | > 1000 RPS | Capacity planning |
| Error rate | < 0.1% | Reliability SLA |
| DB query time (p99) | < 20ms | DB not the bottleneck |

**p99 matters more than average** — average hides outliers. A 50ms average with p99=2000ms means 1 in 100 users waits 2 seconds.

---

### Step 2: Benchmark tests (microbenchmark — function level)

Measures a single function in isolation. No network, no DB.

```go
// internal/usecase/todo_usecase_impl_bench_test.go
func BenchmarkTodoUsecase_GetByID(b *testing.B) {
    uc := newUsecase(
        &repoMock.TodoRepositoryMock{
            GetByIDFn: func(_ context.Context, id uint) (*domainModel.Todo, error) {
                return fixedTodo(id), nil
            },
        },
        &repoMock.AuditLogRepositoryMock{},
    )
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        _, _ = uc.GetByID(context.Background(), 1)
    }
}
```

Run:
```bash
go test ./internal/usecase/... -bench=. -benchmem -count=3
```

Output to understand:
```
BenchmarkTodoUsecase_GetByID-8    2847293    421 ns/op    192 B/op    4 allocs/op
```
- `421 ns/op` — nanoseconds per call
- `192 B/op` — bytes allocated per call (high allocs = GC pressure)
- `4 allocs/op` — heap allocations per call

**Regression gate in CI:** compare against baseline using `benchstat`:
```bash
go test -bench=. -count=10 > new.txt
benchstat old.txt new.txt   # shows % change — fail CI if p99 regresses > 20%
```

---

### Step 3: Load testing (macrobenchmark — full HTTP stack)

Measures the real system under realistic traffic. Use **k6** (JavaScript) or **vegeta** (Go).

#### k6 load test script
```javascript
// tests/load/todo_create.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
    stages: [
        { duration: '30s', target: 50 },   // ramp up to 50 VUs
        { duration: '1m',  target: 50 },   // hold at 50 VUs
        { duration: '30s', target: 200 },  // spike to 200 VUs
        { duration: '1m',  target: 200 },  // hold spike
        { duration: '30s', target: 0 },    // ramp down
    ],
    thresholds: {
        'http_req_duration': ['p(99)<200'],   // 99% of requests < 200ms
        'http_req_failed':   ['rate<0.001'],  // error rate < 0.1%
    },
};

export default function () {
    const res = http.post('http://localhost:8080/api/v1/todos',
        JSON.stringify({ title: `Todo ${Math.random()}` }),
        { headers: { 'Content-Type': 'application/json' } },
    );
    check(res, { 'status 201': (r) => r.status === 201 });
    sleep(1);
}
```

Run:
```bash
k6 run tests/load/todo_create.js
```

Key scenarios to test:
- **Normal load** — expected daily traffic (e.g. 50 concurrent users)
- **Spike test** — sudden 4× traffic increase (marketing campaign, etc.)
- **Soak test** — sustained load for 30+ minutes (find memory leaks, connection pool exhaustion)

---

### Step 4: Profiling with pprof (find the bottleneck)

When load test shows high latency, use pprof to find WHERE time is spent.

```bash
# 1. hit the pprof endpoint while load test is running
go tool pprof http://localhost:8080/debug/pprof/profile?seconds=30

# 2. in pprof interactive shell
(pprof) top10          # top 10 functions by CPU time
(pprof) web            # flame graph in browser (needs graphviz)
(pprof) list GetByID   # line-by-line CPU breakdown of a specific function
```

Memory profile (find leak):
```bash
go tool pprof http://localhost:8080/debug/pprof/heap
(pprof) top10 -cum     # cumulative allocations — find who allocates most
```

Goroutine leak check:
```bash
curl http://localhost:8080/debug/pprof/goroutine?debug=2
# count goroutines — should be stable under load, not growing
```

---

### Step 5: Read Prometheus metrics during load test

While k6 is running, watch Prometheus metrics (Task 07) for:

```
# Request rate
rate(http_requests_total[1m])

# p99 latency
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[1m]))

# DB query p99
histogram_quantile(0.99, rate(db_query_duration_seconds_bucket[1m]))

# Active goroutines (spot leaks)
go_goroutines
```

If p99 latency spikes: check DB query histogram first (is the DB slow?), then check goroutine count (leak?), then CPU profile (hot loop?).

---

### Step 6: Interpret and act

| Observation | Likely cause | Action |
|---|---|---|
| p99 latency high, DB query fast | App logic or serialization bottleneck | CPU profile with pprof |
| p99 latency high, DB query slow | Missing index or N+1 query | EXPLAIN query, add index |
| Error rate spikes under load | Connection pool exhausted | Increase `MaxOpenConns`, add cache |
| Memory grows over time (soak) | Goroutine leak or unbounded cache | Goroutine profile, check worker pool bounds |
| Throughput plateaus at low RPS | Single-threaded bottleneck or mutex contention | pprof block/mutex profile |

**Rule: fix the biggest bottleneck first.** Optimizing a 1ms function when DB takes 100ms gives no user-visible improvement.
