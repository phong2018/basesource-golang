# Task 07 — Observability (Metrics + Enhanced Health Check)

## Requirement
CTO must demonstrate: cloud monitoring services, AWS monitoring, understanding of production system health.

## What the source currently has
- `GET /health` — returns `{"status":"ok"}` (no dependency checks)
- `log/slog` structured JSON logging with `request_id`
- No metrics endpoint

## What to add

### 1. Enhanced health check
`GET /health` — shallow (current, just return 200)
`GET /health/ready` — deep readiness probe:
```json
{
  "status": "ok",
  "checks": {
    "database": "ok",
    "cache":    "ok"
  },
  "version": "1.2.3",
  "uptime": "3h24m"
}
```
- DB check: `db.PingContext(ctx)` with 1s timeout
- Cache check: `redis.Ping(ctx)`
- Returns 503 if any check fails (Kubernetes readinessProbe will stop routing)

### 2. Prometheus metrics endpoint
- `GET /metrics` — standard Prometheus format
- Use `prometheus/client_golang`
- Expose:
  - `http_requests_total{method, path, status}` — request counter
  - `http_request_duration_seconds{method, path}` — latency histogram
  - `db_query_duration_seconds{operation}` — DB latency
  - Go runtime metrics (auto-collected by `promhttp`)

### 3. Metrics middleware
- `internal/presentation/http/middleware/metrics.go`
- Intercept every request, record method + path + status + duration

### 4. Request ID propagation
- Current: `request_id` generated in logger middleware
- Add: propagate incoming `X-Request-ID` header if present (for distributed tracing)
- Pass to response header so client can correlate

### 5. AWS CloudWatch integration (document, not implement)
- Add to `wiki/architechture/observability.md`:
  - How to ship slog JSON → CloudWatch Logs (Fluent Bit sidecar on EKS)
  - CloudWatch Container Insights for CPU/memory
  - Custom metric namespace for business KPIs
  - CloudWatch Alarm → SNS → Slack on error rate spike

## Files to create / modify
- `internal/presentation/http/handler/health_handler.go` — add `/health/ready`
- `internal/presentation/http/middleware/metrics.go`
- `internal/presentation/http/server.go` — register `/metrics`, `/health/ready`

## Demonstrates to CTO interviewer
- Distinguishes liveness vs readiness (common Kubernetes interview question)
- Knows Prometheus + Grafana stack (industry standard)
- Structured logging (already done) — mentions correlation via request_id
- Can describe AWS CloudWatch setup for EKS-deployed services
- Understands that `/health` returning 200 without checking DB is useless in prod
