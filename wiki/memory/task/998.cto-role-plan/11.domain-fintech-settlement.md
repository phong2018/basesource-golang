| Challenge | Why it matters |
|---|---|
| Floating point money | `0.1 + 0.2 = 0.30000000000000004` in float64 — wrong for financial calculation |
| Concurrent deduction | Two requests deduct from same wallet simultaneously → balance goes negative |

| Audit trail | Every balance change must be traceable to a transaction |

### 1. Money representation — never use float

```go
// domain/model/money.go
// Value Object: Money represents an amount in a specific currency.
// Uses int64 (minor units) to avoid floating point errors.
// 10000 = 100.00 USD (2 decimal places), 10000 = 100 JPY (0 decimal places)
type Money struct {
    Amount   int64  // minor units (cents, pence, etc.)
    Currency string // ISO 4217: "USD", "VND", "JPY"
}

func (m Money) Add(other Money) (Money, error) {
    if m.Currency != other.Currency {
        return Money{}, errors.New("currency mismatch")
    }
    return Money{Amount: m.Amount + other.Amount, Currency: m.Currency}, nil
}
```

**Why int64 not `decimal` library:** int64 is zero-dependency, fast, and sufficient. Use `shopspring/decimal` only if you need arbitrary precision (e.g. crypto with 18 decimal places).

### 2. Wallet entity with optimistic locking

```go
// domain/model/wallet.go
// Entity: Wallet holds a user's balance for one currency.
type Wallet struct {
    ID        uint   `db:"id"`
    UserID    uint   `db:"user_id"`
    Currency  string `db:"currency"`
    Balance   int64  `db:"balance"`   // minor units
    Version   int    `db:"version"`   // optimistic lock
    UpdatedAt time.Time `db:"updated_at"`
}
```

**Optimistic locking prevents concurrent overdraft:**
```sql
-- repository: UPDATE with version check
UPDATE wallets
SET balance = ?, version = version + 1, updated_at = NOW()
WHERE id = ? AND version = ?   -- fails if another tx already incremented version
```
If `rowsAffected == 0` → version mismatch → retry or return `ErrConcurrentModification`.

### 3. Double-entry bookkeeping (ledger)

Every balance change is recorded as two ledger entries that sum to zero — standard accounting principle:

```go
// domain/model/ledger_entry.go
// Entity: LedgerEntry records one side of a financial transaction.
type LedgerEntry struct {
    ID            uint      `db:"id"`
    TransactionID uint      `db:"transaction_id"`
    WalletID      uint      `db:"wallet_id"`
    Type          string    `db:"type"`    // "debit" | "credit"
    Amount        int64     `db:"amount"`  // always positive
    BalanceAfter  int64     `db:"balance_after"`
    CreatedAt     time.Time `db:"created_at"`
}
```

Transfer $10 from wallet A to wallet B creates two entries:
```
debit  wallet_A  1000 (cents)  balance_after=5000
credit wallet_B  1000 (cents)  balance_after=2000
```
Sum of all entries for a transaction = 0. This is how reconciliation works.

### 5. Reconciliation concept (document, not implement fully)

End-of-day reconciliation verifies no money was created or destroyed:

```sql
-- reconciliation query: opening balance + credits - debits = closing balance
SELECT
    w.id,
    w.balance AS closing_balance,
    snapshot.balance AS opening_balance,
    SUM(CASE WHEN le.type = 'credit' THEN le.amount ELSE 0 END) AS total_credits,
    SUM(CASE WHEN le.type = 'debit'  THEN le.amount ELSE 0 END) AS total_debits,
    snapshot.balance + total_credits - total_debits AS expected_closing
FROM wallets w
JOIN daily_snapshots snapshot ON snapshot.wallet_id = w.id AND snapshot.date = CURDATE() - 1
JOIN ledger_entries le ON le.wallet_id = w.id AND DATE(le.created_at) = CURDATE()
GROUP BY w.id
HAVING closing_balance != expected_closing  -- flag discrepancies
```

Any row returned = reconciliation break → alert, investigate, do not silently pass.

### 6. Settlement flow (explain the concept)

Settlement = aggregating many small transactions into one net transfer between parties (e.g. marketplace paying merchants at end of day):

```
Day's orders for merchant_A:
  order_001: +500,000 VND
  order_002: +300,000 VND
  order_003: -50,000 VND (refund)
  platform_fee: -75,000 VND

Net settlement = 675,000 VND → single bank transfer to merchant_A
```

Key properties:
- Atomic: either all orders are settled or none (use transaction)
- Idempotent: if settlement job crashes and reruns, merchant is not paid twice
- Auditable: settlement record references all included order IDs
