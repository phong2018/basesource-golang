# Task 01 — Security Best Practices

## Requirement
CTO must demonstrate: security best practices across four layers — **Authentication & Authorization**, **Input / Output (XSS, CSRF)**, **Database security**, **Infrastructure & Secrets**.

---

## Layer 1: Authentication & Authorization

### 1a. JWT Authentication Middleware
- `pkg/auth/jwt.go` — sign/verify JWT using `golang-jwt/jwt/v5`
- `internal/presentation/http/middleware/auth.go` — extract + validate Bearer token, inject `user_id` into context
- Protected route group `/api/v1/todos` behind auth middleware
- Config: `JWTSecret` in `config.go`

```go
// middleware/auth.go — fail fast, never leak JWT error detail to client
token, err := jwt.ParseWithClaims(tokenStr, &Claims{}, keyFunc)
if err != nil || !token.Valid {
    return apperror.Unauthorized("invalid token") // never return err.Error() — leaks internals
}
```

### 1b. Authorization — resource ownership check
Not just "is the user logged in" but "does this user own this todo":
```go
// usecase: after fetching todo, verify ownership
if todo.UserID != input.CallerUserID {
    return nil, apperror.Forbidden("access denied")
}
```
- Return `403 Forbidden`, not `404` — but in some cases returning `404` is intentional (don't reveal resource existence to unauthorized callers)

### 1c. Rate Limiting (brute-force protection)
- `internal/presentation/http/middleware/rate_limit.go`
- Per-IP token bucket using `golang.org/x/time/rate`
- Stricter limit on auth endpoints (`/login`, `/refresh`) than general API
- Return `429 Too Many Requests` with `Retry-After` header

## Layer 2: Input / Output Security (XSS, CSRF)

### 2a. XSS (Cross-Site Scripting)

**This is a Go API — XSS risk is in the client, not the server. But the server must not help an attacker.**

What the server must do:
- **Never reflect user input unescaped in HTML responses** — this API returns JSON, so risk is low
- **Set `Content-Type: application/json`** explicitly — prevents browser from MIME-sniffing a JSON response as HTML
- **Set `X-Content-Type-Options: nosniff`** — prevents IE/legacy browsers from MIME-sniffing
- **Sanitize stored HTML** — if any field accepts rich text (e.g. todo description with HTML), sanitize on write using `microcosm-cc/bluemonday` before storing:

```go
// usecase: sanitize before save, not on read
p := bluemonday.StrictPolicy()  // strips all HTML tags
input.Description = p.Sanitize(input.Description)
```

**Why sanitize on write, not read:** stored data is already clean; no risk of forgetting to sanitize at a new read path added later.

### 2b. CSRF (Cross-Site Request Forgery)

**CSRF applies when the API uses cookie-based sessions. JWT in Authorization header is naturally CSRF-safe.**

Current design uses JWT Bearer token → CSRF is not a risk because:
- Browser does NOT auto-attach `Authorization: Bearer <token>` to cross-origin requests
- Only `Cookie` headers are auto-attached (the CSRF attack vector)

**Document the rule:** if we ever add cookie-based auth (e.g. for SSR pages), add CSRF token middleware immediately:
```go
// internal/presentation/http/middleware/csrf.go (future)
e.Use(middleware.CSRFWithConfig(middleware.CSRFConfig{
    TokenLookup:    "header:X-CSRF-Token",
    CookieSameSite: http.SameSiteStrictMode,
    CookieSecure:   true,
    CookieHTTPOnly: true,
}))
```

**`SameSite=Strict`** on session cookies already blocks CSRF for same-origin policies on modern browsers.

### 2c. Request Validation — defense in depth

Validation must happen at **two layers** with different responsibilities:

#### Presentation layer — validate request shape (HTTP concern)
Validates that the request is well-formed before it reaches business logic.

```go
// internal/presentation/http/handler/todo_handler.go
func (h *todoHandler) Create(c echo.Context) error {
    var req dto.CreateTodoRequest
    if err := c.Bind(&req); err != nil {
        return apperror.BadRequest("invalid request body")
    }
    if err := c.Validate(&req); err != nil {
        return err  // validator returns 400 with field errors
    }
    ...
}
```

What to validate here:
- **Required fields** — `title` must not be empty
- **Type correctness** — `page` and `limit` must be positive integers
- **Max length on all string fields** — without this, a 100MB title exhausts memory:
```go
// internal/presentation/http/dto (request structs — NOT usecase DTOs)
type CreateTodoRequest struct {
    Title       string  `json:"title"       validate:"required,min=1,max=255"`
    Description *string `json:"description" validate:"omitempty,max=2000"`
}

type ListTodoRequest struct {
    Page  int `query:"page"  validate:"required,min=1"`
    Limit int `query:"limit" validate:"required,min=1,max=100"`
    Done  *bool `query:"done"`
}
```
- **Enum/allowed values** — if `status` field only allows `"open"|"done"`, reject anything else at this layer
- **Reject unexpected fields** — Echo's `Bind` silently ignores unknown fields (safe by default) — document this behavior

#### Usecase layer — validate business rules (domain concern)
Validates that the operation makes sense in the domain context, after the request shape is confirmed valid.

```go
// internal/usecase/todo_usecase_impl.go
func (u *todoUsecase) Create(ctx context.Context, input dto.CreateTodoInput) (*dto.TodoOutput, error) {
    // business rule: title must be unique per user
    existing, _ := u.repo.FindByTitle(ctx, input.Title, input.UserID)
    if existing != nil {
        return nil, apperror.BadRequest("todo with this title already exists")
    }
    // business rule: max 1000 todos per user
    count, _ := u.repo.CountByUser(ctx, input.UserID)
    if count >= domainModel.MaxTodosPerUser {
        return nil, apperror.BadRequest("todo limit reached")
    }
    ...
}
```

#### Custom validator setup (wire into Echo)
```go
// internal/presentation/http/validator/validator.go
type RequestValidator struct {
    validator *validator.Validate
}

func NewRequestValidator() *RequestValidator {
    v := validator.New()
    // register custom tag for future use (e.g. `validate:"iso4217"` for currency codes)
    return &RequestValidator{validator: v}
}

func (rv *RequestValidator) Validate(i any) error {
    if err := rv.validator.Struct(i); err != nil {
        // map validation errors to structured 400 response with field-level detail
        var ve validator.ValidationErrors
        if errors.As(err, &ve) {
            fields := make([]string, len(ve))
            for i, e := range ve {
                fields[i] = fmt.Sprintf("%s: %s", e.Field(), e.Tag())
            }
            return apperror.BadRequestWithFields("validation failed", fields)
        }
        return apperror.BadRequest("invalid request")
    }
    return nil
}
```

#### Validation layer responsibility table
| Concern | Layer | Example |
|---|---|---|
| Field required / type / length / format | Presentation | `required`, `max=255`, `email` |
| Allowed enum values | Presentation | `oneof=open done` |
| Cross-field rules | Presentation | `gtfield=StartDate` (end > start) |
| Business uniqueness | Usecase | title already exists for this user |
| Domain limits | Usecase | max 1000 todos per user |
| Ownership / authorization | Usecase | caller owns this todo |

### 2d. CORS
- `labstack/echo/v4/middleware.CORS()` with explicit allowlist — never `AllowOrigins: ["*"]` in production
- Configured from env `CORS_ALLOWED_ORIGINS=https://app.example.com`
- Only allow methods and headers the API actually uses

```go
middleware.CORSWithConfig(middleware.CORSConfig{

    AllowHeaders: []string{"Authorization", "Content-Type", "X-Request-ID"},
})
```

## Layer 3: Database Security

### 3a. SQL Injection prevention — already done, document why
Current: sqlx with parameterized queries (placeholders `?`).

```go
// ✅ safe — value is bound, never interpolated into query string
db.GetContext(ctx, &todo, "SELECT id, title FROM todos WHERE id = ?", id)

// ❌ never do this — SQL injection
db.GetContext(ctx, &todo, fmt.Sprintf("SELECT * FROM todos WHERE id = %d", id))
```

Document the rule explicitly: **never use `fmt.Sprintf` or string concatenation to build SQL.**

### 3b. Principle of Least Privilege — DB user permissions
The app's DB user should only have the permissions it needs:

```sql
-- migrations run as admin (goose) — full DDL
-- app runtime user — DML only, no DROP/ALTER
CREATE USER 'appuser'@'%' IDENTIFIED BY '...';
GRANT SELECT, INSERT, UPDATE, DELETE ON appdb.* TO 'appuser'@'%';
-- NOT: GRANT ALL PRIVILEGES
```

- `config.go`: separate `DATABASE_DSN` (app runtime) from `DATABASE_ADMIN_DSN` (migrations only)
- `cmd/migrate/cmd.go` uses admin DSN; `cmd/api/cmd.go` uses app DSN

### 3c. Sensitive data — never log DB values
```go
// ❌ logs the user's raw input — may contain PII
slog.ErrorContext(ctx, "query failed", "input", input)

// ✅ log only safe metadata
slog.ErrorContext(ctx, "query failed", "todo_id", input.ID, "error", err)
```

Use `pkg/helper/mask.go` (already exists) to mask PII fields before logging.

### 3d. DB connection over TLS (production)
- DSN must include `tls=true` or `tls=custom` when connecting to RDS
- Add to `.env.example` comment: `# append ?tls=true for RDS / Cloud SQL`
- Without TLS, credentials and query results travel in plaintext on the network

### 3e. Sensitive fields — never return in API response
Example: if `users` table is added later, never return `password_hash`, internal flags, or audit fields in `TodoOutput` / `UserOutput`.
- Map entity → DTO in usecase: explicitly choose which fields to expose
- Current `TodoOutput` already does this correctly (no `db:` tagged fields leak out)

## Layer 4: Infrastructure & Secrets

### 4a. No hardcoded secrets in defaults
```go
// config.go — fail fast in production
if cfg.AppEnv == "production" && cfg.JWTSecret == "" {
    return nil, errors.New("JWT_SECRET must be set in production")
}
```
Applies to: `JWT_SECRET`, `DATABASE_DSN`, `AWS_SECRET_ACCESS_KEY`, `NOTIFICATION_API_KEY`.

### 4b. Security response headers
`internal/presentation/http/middleware/security_headers.go`:

| Header | Value | Protects against |
|---|---|---|
| `X-Content-Type-Options` | `nosniff` | MIME-type sniffing → XSS |
| `X-Frame-Options` | `DENY` | Clickjacking |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | SSL stripping (HTTPS only) |
| `Content-Security-Policy` | `default-src 'none'` | XSS (for any HTML responses) |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Referrer info leakage |

### 4c. Dependency scanning (CI)
Add to GitHub Actions CI pipeline (Task 04):
```yaml
- name: Vulnerability scan
  uses: golang/govulncheck-action@v1   # official Go vuln db
```
Catches known CVEs in `go.sum` dependencies before they reach production.

## Demonstrates to CTO interviewer
- **Request validation:** two-layer separation — presentation validates shape, usecase validates business rules; knows the difference
- **Auth:** JWT Bearer is naturally CSRF-safe — knows the difference, doesn't add CSRF blindly
- **XSS:** understands server's role (sanitize on write, correct headers) even for JSON APIs
- **CSRF:** knows when it applies (cookies) and when it doesn't (Bearer token) — not cargo-cult security
- **Database:** parameterized queries + least-privilege DB user + TLS in transit
- **Secrets:** fails fast in production, never ships with empty defaults
- **Defense-in-depth:** multiple layers — validation, auth, rate limit, headers, DB permissions, dep scanning
- **PII hygiene:** masks sensitive data in logs, doesn't leak entity fields in API responses
