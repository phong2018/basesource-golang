## Architecture Overview

```
Todo Usecase (Create/Update/Delete)
    │
    │ inside WithinTransaction:
    │   1. INSERT todos
    │   2. INSERT audit_logs
    │   3. INSERT outbox_events  ← same tx, atomic
    │
    ▼
Outbox Relay Worker (background goroutine in cmd/worker)
    │
    │ polls outbox_events WHERE status = 'pending'
    │ publishes to RabbitMQ exchange
    │ UPDATE status = 'published'
    │
    ▼
[Exchange: todo.events — topic type]
    ├──▶ Queue: todo.notification  → NotificationConsumer (logs event)
    └──▶ Queue: todo.audit         → AuditConsumer        (logs event)
```

**Why outbox, not direct publish after tx.Commit():**
- App crashes between `tx.Commit()` and `broker.Publish()` → event lost forever
- Outbox: INSERT in same tx → if crash, row stays `pending` → relay retries on restart

```go
    body, _ := json.Marshal(event)
    return p.ch.PublishWithContext(ctx,
        p.cfg.Exchange,    // exchange
        event.EventType,   // routing key: todo.created / todo.updated / todo.deleted

        false,             // immediate

            Body:         body,
        },
```

Declares queue, binds to exchange, starts consuming:
```go
func (c *consumer) Start(ctx context.Context, queueName string, routingKey string, handler func([]byte) error) error {
    // declare durable queue
    q, _ := c.ch.QueueDeclare(queueName, true, false, false, false, nil)
    // bind queue to exchange with routing key
    c.ch.QueueBind(q.Name, routingKey, c.cfg.Exchange, false, nil)
    // prefetch = 1: fair dispatch
    c.ch.Qos(c.cfg.PrefetchCount, 0, false)

    msgs, _ := c.ch.Consume(q.Name, "", false, false, false, false, nil)
    for {
        select {
        case <-ctx.Done():
            return nil
        case msg, ok := <-msgs:
            if !ok { return nil }
            if err := handler(msg.Body); err != nil {
                msg.Nack(false, false)  // → dead-letter queue, no requeue
            } else {
                msg.Ack(false)
            }
        }
    }
}
```

### Step 8 — Infrastructure: outbox relay worker

**File: `internal/infrastructure/messaging/outbox_relay.go`**

Polls `outbox_events WHERE status='pending'` and publishes:
```go
type OutboxRelay struct {
    outboxRepo domainRepo.IOutboxRepository
    publisher  domainSvc.IEventPublisher
    interval   time.Duration
}

func (r *OutboxRelay) Start(ctx context.Context) {
    ticker := time.NewTicker(r.interval)  // e.g. 2 seconds
    defer ticker.Stop()
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            r.process(ctx)
        }
    }
}

func (r *OutboxRelay) process(ctx context.Context) {
    events, err := r.outboxRepo.ListPending(ctx, 100)
    if err != nil {
        slog.ErrorContext(ctx, "outbox relay: list pending failed", "error", err)
        return
    }
    for _, event := range events {
        if err := r.publisher.Publish(ctx, event); err != nil {
            slog.ErrorContext(ctx, "outbox relay: publish failed", "event_id", event.EventID, "error", err)
            _ = r.outboxRepo.MarkFailed(ctx, event.ID)
            continue
        }
        _ = r.outboxRepo.MarkPublished(ctx, event.ID)
    }
}
```

## Concepts: Publisher and Consumer

Before reading the scenarios, understand these two roles:

### Publisher (who sends messages)
The **publisher** is the component that puts a message into RabbitMQ.

In this source:
- The **outbox relay** (`outbox_relay.go`) is the publisher
- It polls `outbox_events` table every 2 seconds for rows with `status=pending`
- For each pending row it calls `publisher.Publish(event)` which sends the message to RabbitMQ exchange `todo.events`
- After a successful publish it updates the row to `status=published`

The **usecase is NOT the publisher** — the usecase only writes to the `outbox_events` table. The relay worker does the actual publishing. This is the outbox pattern.

```
curl POST /api/v1/todos
    → usecase.Create()
        → INSERT todos           ┐
        → INSERT audit_logs      ├ one DB transaction
        → INSERT outbox_events   ┘  status=pending

(2 seconds later, relay worker wakes up)
    → SELECT outbox_events WHERE status=pending
    → rabbitmq_publisher.Publish(event)   ← THIS is the publisher
    → UPDATE outbox_events SET status=published
```

### Consumer (who receives messages)
The **consumer** is the component that reads messages from RabbitMQ and does something with them.

In this source:
- The **worker** (`cmd/worker/cmd.go`) starts the consumer
- The consumer connects to queue `todo.notification` (bound to exchange `todo.events`)
- For each message received it calls a **handler function** — in this source the handler just logs the event
- After the handler succeeds it sends **ACK** back to RabbitMQ ("I processed this, remove it from the queue")
- If the handler fails it sends **NACK** ("I failed, send this to the dead-letter queue")

```
RabbitMQ exchange: todo.events
    │
    │ routes by routing key (todo.created / todo.updated / todo.deleted)
    ▼
Queue: todo.notification
    │
    ▼
Consumer (cmd/worker)
    → handler(message body)
        → slog.Info("event received", ...)   ← in real system: send email, update search index, etc.
    → ACK (tell RabbitMQ: done, remove this message)
```

### How they relate

```
[API server]              [RabbitMQ]               [Worker]
     │                       │                        │
usecase writes         exchange routes           consumer reads
outbox_events    →     todo.events          →    todo.notification
(DB row)               (broker)                  (handler logs it)
     │                       ↑                        │
     │               outbox relay                     │
     │               publishes every 2s               │
     └───────────────────────┘                        │
                                                      ACK
```

The publisher and consumer **never talk to each other directly**. They only know about RabbitMQ. This is the decoupling benefit — you can replace, scale, or add consumers without touching the publisher, and vice versa.

**Purpose:** verify the full flow works end-to-end.

**Setup:**
```bash
docker compose -f docker/docker-compose.yaml up -d   # starts MySQL + RabbitMQ
```

**Terminal 1 — start API:**

**Terminal 2 — start worker (this process contains BOTH the publisher/relay AND the consumer):**
```bash
go run main.go worker
# Inside worker two goroutines start:
#   goroutine 1: outbox_relay  → PUBLISHER — polls DB, sends to RabbitMQ
#   goroutine 2: event_consumer → CONSUMER — reads from RabbitMQ queue, logs event
```

**Terminal 3 — create a todo (triggers the whole chain):**
```bash
curl -X POST http://localhost:8080/api/v1/todos \
  -H "Content-Type: application/json" \
  -d '{"title":"Buy milk"}'
```

**What happens step by step:**
```
1. curl hits API server (Terminal 1)
2. usecase.Create() opens a DB transaction:
      INSERT todos           → row created
      INSERT audit_logs      → audit row created
      INSERT outbox_events   → status=pending  ← not yet in RabbitMQ
   tx.Commit()
3. API returns 201 immediately — RabbitMQ not involved yet

4. (2 seconds later) outbox relay wakes up:
      SELECT outbox_events WHERE status=pending  → finds the row
      rabbitmq_publisher.Publish(event)           ← PUBLISHER sends to RabbitMQ
      UPDATE outbox_events SET status=published

5. RabbitMQ routes the message from exchange todo.events → queue todo.notification

6. Consumer (goroutine 2 in worker) receives the message:
      handler(body) → logs it                     ← CONSUMER processes it
      ACK → RabbitMQ removes message from queue
```

In Terminal 1 (API):

In Terminal 2 (worker) — first the publisher relay logs, then the consumer logs:

**What this proves:** API call → INSERT todos + audit_logs + outbox_events in one transaction → relay (publisher) picks up pending event → sends to RabbitMQ → consumer receives and processes it. The API server and worker are completely separate processes — they only share MySQL and RabbitMQ.

### Scenario 2 — Outbox guarantees: publisher crash does not lose events

**Purpose:** show why we use the outbox pattern instead of publishing directly after `tx.Commit()`. If the **publisher** (outbox relay / worker) crashes before it can send to RabbitMQ, the event is NOT lost — it stays `pending` in MySQL and is retried when the publisher restarts.

**Steps:**

1. Stop the worker — **publisher is now down**, consumer is also down:
```bash
# Ctrl+C in Terminal 2
```

2. Create a todo via API — API server has NO knowledge of the worker:
```bash
curl -X POST http://localhost:8080/api/v1/todos \
  -H "Content-Type: application/json" \
  -d '{"title":"Task while worker is down"}'
# → 201 Created ← API still works, usecase only writes to MySQL
```

3. What happened inside the usecase (publisher NOT involved yet):
```
usecase.Create():
  tx.Commit():
    INSERT todos           → done
    INSERT audit_logs      → done
    INSERT outbox_events   → status=pending  ← saved in MySQL, waiting for publisher
  return 201
```

4. Check DB — event is `pending`, publisher has not sent it to RabbitMQ yet:
```sql
SELECT event_type, status FROM outbox_events ORDER BY id DESC LIMIT 1;
-- status: pending  ← safely stored, not lost
```

5. Restart the worker — **publisher (outbox relay) comes back online**:
```bash
go run main.go worker
```

6. Relay (publisher) polls DB immediately, finds the pending row, publishes to RabbitMQ:
```
# publisher logs:
level=INFO msg="outbox relay: published event" event_type=todo.created event_id=abc-123
# consumer logs (receives from RabbitMQ):
level=INFO msg="event received" event_type=todo.created aggregate_id=2
```

7. Check DB — publisher marked the row published:
```sql
SELECT status, published_at FROM outbox_events ORDER BY id DESC LIMIT 1;
-- status: published ← relay caught up after restart
```

**What this proves:** outbox pattern gives at-least-once delivery. The publisher (relay) and consumer are in the worker process — they can go down and come back without losing any events, because events are durably stored in MySQL first. Compare to direct publish: if the process crashes between `tx.Commit()` and `broker.Publish()`, the event is lost permanently with no way to recover.

### Scenario 3 — Pub/Sub: multiple independent consumers receive the same event

**Purpose:** show that adding a second consumer does not require any change to the publisher. This is the core value of pub/sub over direct HTTP calls.

**Key concept here:**
- The **publisher** (outbox relay) sends one message to the exchange `todo.events`
- The **exchange** fans the message out to every queue bound to it
- Each queue has its own **consumer** — they all receive the same message independently
- Adding a new consumer = add a new queue binding. Zero change to the publisher.

```
Publisher (outbox relay)
    │
    │ sends ONE message with routing key: todo.created
    ▼
Exchange: todo.events (topic)
    │
    ├──▶ Queue: todo.notification  →  Consumer 1 (notification worker) receives it
    └──▶ Queue: todo.audit         →  Consumer 2 (audit worker)        receives it

Both consumers get the SAME message. Publisher sent it once.
```

**Setup:** worker starts two consumers bound to the same exchange:
```bash
go run main.go worker
# worker internally starts:
#   publisher:  outbox relay → sends to exchange todo.events
#   consumer 1: queue=todo.notification, binding=todo.*  → receives all todo events
#   consumer 2: queue=todo.audit,        binding=todo.*  → receives all todo events (same copy)
```

Create a todo:
```bash
curl -X POST http://localhost:8080/api/v1/todos \
  -H "Content-Type: application/json" \
  -d '{"title":"Multi-consumer test"}'
```

**What you should see in Terminal 2 (publisher relay logs first, then both consumers log):**
```
level=INFO msg="outbox relay: published event"      event_type=todo.created   ← publisher sent it once
level=INFO msg="[notification] event received"      event_type=todo.created   ← consumer 1 got it
level=INFO msg="[audit] event received"             event_type=todo.created   ← consumer 2 got it too
```

In RabbitMQ UI (http://localhost:15672):
- Queues → `todo.notification`: 1 message in, 1 out
- Queues → `todo.audit`: 1 message in, 1 out
- Same event delivered to both queues from one publish

**What this proves:** publisher sent one event, two consumers each received their own copy independently. In a real system: consumer 1 sends an email, consumer 2 updates a search index — they do completely different things from the same event, with no knowledge of each other. Adding consumer 3 (e.g. analytics) requires zero changes to the publisher.

### Scenario 4 — Consumer group: multiple worker instances share the load

**Purpose:** show horizontal scaling — run 3 worker instances (same consumer group), messages are distributed across them. Each message is processed by **exactly one** worker, not all three.

**Key concept here:**
- Multiple workers connect to the **same queue** `todo.notification`
- RabbitMQ round-robins messages across all connected consumers
- This is a **consumer group** — all workers do the same job, but share the workload
- Adding more workers = more throughput, no code change needed

```
Queue: todo.notification (9 pending messages)
    ├──▶ Worker pod 1 (consumer)  → processes messages 1, 4, 7
    ├──▶ Worker pod 2 (consumer)  → processes messages 2, 5, 8
    └──▶ Worker pod 3 (consumer)  → processes messages 3, 6, 9

Each message processed ONCE. Not by all three workers.
```

**Note:** the **publisher** (outbox relay) only needs to run on ONE worker instance. Running it on all 3 would cause duplicate publishes. In this demo, only Terminal 2 runs the relay; Terminals 3 and 4 run consumer-only workers.

**Setup:** open 3 terminals, each running a worker:
```bash
# Terminal 2
go run main.go worker
# Terminal 3
go run main.go worker
# Terminal 4
go run main.go worker
```

Create 9 todos rapidly:

### Scenario 6 — RabbitMQ goes down: publisher cannot send, but API still works, events catch up

**Purpose:** show fault isolation — RabbitMQ outage affects only the **publisher** (relay cannot send) and **consumer** (cannot receive). The API and the `outbox_events` table are completely unaffected. Events catch up automatically when RabbitMQ recovers.

```
Normal flow:
  API → MySQL (outbox) → relay (publisher) → RabbitMQ → consumer

RabbitMQ down:
  API → MySQL (outbox) → relay tries to publish → FAILS, row stays pending
  Consumer: receives nothing (broker is down)

After RabbitMQ recovers:
  relay (publisher) retries → publishes all pending rows
  consumer: starts receiving again
```

**Steps:**

1. Stop RabbitMQ:
```bash
docker compose -f docker/docker-compose.yaml stop rabbitmq
```

2. Create todos via API — should still return 201 (transaction commits to MySQL, outbox row saved):
```bash
curl -X POST http://localhost:8080/api/v1/todos \
  -H "Content-Type: application/json" \
  -d '{"title":"Created during RabbitMQ outage"}'
# → 201 Created ← API not affected
```

3. Worker logs will show relay errors (cannot connect to broker) but continues retrying:
```
level=ERROR msg="outbox relay: publish failed" error="dial tcp: connection refused"
```

4. Restart RabbitMQ:
```bash
docker compose -f docker/docker-compose.yaml start rabbitmq
```

5. Within 2 seconds, relay picks up all pending events and publishes them:
```
level=INFO msg="event received" event_type=todo.created aggregate_id=...
```

**What this proves:** the outbox pattern decouples the API from RabbitMQ. Users can still create todos even when the message broker is down. Events are durably stored in MySQL and published as soon as the broker recovers. This is the key operational advantage over direct HTTP call to a notification service (which would cause 500 errors during an outage).

## Architecture rules respected

- `IOutboxRepository` in `domain/repository/` — not in infrastructure
- `IEventPublisher` in `domain/service/` — not in infrastructure
- Outbox INSERT happens inside `WithinTransaction` — atomic with todo + audit log
- Publisher is called from the relay worker (infrastructure), NOT from usecase
- Usecase only depends on `domain` interfaces — no `amqp` import in usecase layer
- `cmd/worker` uses container for wiring — same pattern as `cmd/api`
