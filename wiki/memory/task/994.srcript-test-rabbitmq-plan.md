## Test 05 — Queue durability (restart worker, message survives)

**What it proves:** RabbitMQ queue is declared `durable=true` and messages are published with `DeliveryMode=Persistent`. A notification published while the worker is alive survives a worker restart and is delivered after reconnect.

**Flow:**
1. POST a todo → outbox relay publishes notification to RabbitMQ → worker processes it → assert `notification task received` in log.
2. Kill worker (`pkill -TERM`), wait for it to stop.
3. POST a second todo → outbox relay publishes another notification to RabbitMQ.
   - Worker is dead, so message sits in the queue unacknowledged (actually undelivered — no consumer).
4. Verify via RabbitMQ management HTTP API that the queue has 1 ready message:
   ```bash
   curl -s -u guest:guest http://localhost:15672/api/queues/%2F/todo.notifications \
     | python3 -c "import sys,json; print(json.load(sys.stdin)['messages_ready'])"
   # expected: 1
   ```
5. Restart worker, wait for `rabbitmq notification consumer starting` in new log.
6. Assert `notification task received` for the second todo appears in new worker log within 10s.

**Pass condition:** message queued while worker was offline is delivered after restart.

## Test 06 — Queue while worker offline (batch drain)

**What it proves:** Multiple messages queued while the worker is down are all drained in order when it restarts.

**Flow:**
1. Kill worker.
2. POST 3 todos in sequence (ids N, N+1, N+2) → 3 notifications published to RabbitMQ.
3. Verify queue depth = 3 via management API.
4. Restart worker.
5. Wait up to 15s for all 3 `notification task received` entries in new log.
6. Assert exactly 3 new notifications appeared (count before restart vs after).

**Pass condition:** all 3 messages drained after worker restart, none dropped.

## Test 07 — NACK on handler error (no infinite loop)

**What it proves:** When `HandleNotificationTask` returns an error, the consumer calls `d.Nack(false, false)` (no requeue). The message is dropped (or sent to DLX if configured) rather than retried forever.

**Implementation note:** This test requires temporarily making the handler fail. The cleanest approach for a bash script is to launch a separate binary with a modified env/flag, or verify the NACK behavior via the management API message count.

**Practical approach (no code change needed):**
1. Use RabbitMQ management API to manually publish a malformed message (invalid JSON) to `todo.notifications` queue directly:
   ```bash
   curl -s -u guest:guest -X POST http://localhost:15672/api/exchanges/%2F/amq.default/publish \
     -H "Content-Type: application/json" \
     -d '{"properties":{"delivery_mode":2},"routing_key":"todo.notifications","payload":"not-json","payload_encoding":"string"}'
   ```
2. Wait 3s.
3. Assert via management API that `messages_ready` and `messages_unacknowledged` are both 0 — message was NACK'd and dropped, not requeued.
4. Assert worker is still running (process alive) — handler error did not crash the worker.
5. Assert subsequent valid notifications still work: POST a todo, verify `notification task received` appears.

**Pass condition:** malformed message is dropped (not requeued), worker continues processing valid messages.

## Test 08 — Competing consumers (two workers share the queue)

**What it proves:** Two worker instances consume from the same `todo.notifications` queue. RabbitMQ round-robins messages between them. Total notifications processed across both workers equals total posted todos.

**Flow:**
1. Start a second worker instance writing to `/tmp/worker2.log`:
   ```bash
   /tmp/basesource worker >> /tmp/worker2.log 2>&1 &
   ```
2. Wait for second worker's `rabbitmq notification consumer starting`.
3. POST 4 todos in quick succession.
4. Wait 10s for all notifications to drain.
5. Count `notification task received` in `/tmp/worker.log` + `/tmp/worker2.log` combined → must equal 4.
6. Optionally assert both workers got at least 1 message each (RabbitMQ distributes fairly with prefetch=1).
7. Kill second worker.

**Pass condition:** total notifications across both workers = 4, no message delivered twice (sum = exactly 4).

## Shared helpers to add to each script

```bash
rmq_queue_depth() {
  local queue=$1
  curl -s -u guest:guest \
    "http://localhost:15672/api/queues/%2F/${queue}" \
    | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('messages_ready',0))"
}

rmq_publish_raw() {
  local queue=$1 payload=$2
  curl -s -u guest:guest -X POST \
    "http://localhost:15672/api/exchanges/%2F/amq.default/publish" \
    -H "Content-Type: application/json" \
    -d "{\"properties\":{\"delivery_mode\":2},\"routing_key\":\"${queue}\",\"payload\":\"${payload}\",\"payload_encoding\":\"string\"}"
}
```
