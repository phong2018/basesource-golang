# Plan: Kubernetes Integration Test Suite

Duplicate `994.script-test-kafka` and `994.script-test-rabbitmq` into a new
test suite that targets the **kind cluster** instead of docker-compose.

Target directory: `wiki/memory/task/994.script-test-k8s/`

---

## What changes when moving from docker-compose to Kubernetes

| Concern | docker-compose | Kubernetes |
|---|---|---|
| API endpoint | `http://localhost:8080` | `kubectl port-forward svc/basesource-api-svc 18080:80` ã `http://localhost:18080` |
| Worker logs | `/tmp/worker.log` (local file) | `kubectl -n basesource logs -l app=basesource-worker` |
| DB queries | `docker compose exec -T db mysql ...` | `kubectl -n basesource exec -T <mysql-pod> -- mysql ...` |
| Kafka CLI | `docker compose exec -T kafka /opt/kafka/bin/...` | `kubectl -n basesource exec -T <kafka-pod> -- /opt/kafka/bin/...` |
| RabbitMQ management API | `http://localhost:15672` | `kubectl port-forward svc/rabbitmq-service 15672:15672` |
| Stop a service | `docker compose stop rabbitmq` | `kubectl -n basesource scale deployment rabbitmq --replicas=0` |
| Start a service | `docker compose start rabbitmq` | `kubectl -n basesource scale deployment rabbitmq --replicas=1` |
| Reset / wipe data | truncate via docker exec | truncate via kubectl exec |
| Worker restart | `pkill` + re-run binary | `kubectl -n basesource rollout restart deployment basesource-worker` |
| Consumer group cleanup | `kafka-consumer-groups.sh` via docker exec | same tool via kubectl exec |
| Build binary | `go build` locally | `docker build` + `kind load docker-image` + rollout restart |

**There is no local binary in K8s tests.** All processes run as pods.
The test scripts only drive the cluster from the outside (HTTP + kubectl).

## lib.sh ã shared kubectl helpers

Every test script sources this file. It hides the kubectl verbosity.

```bash
NS=basesource

# Dynamic pod name lookup (called fresh each time ã pods can be restarted)
mysql_pod()  { kubectl -n $NS get pod -l app=mysql  -o jsonpath='{.items[0].metadata.name}'; }
kafka_pod()  { kubectl -n $NS get pod -l app=kafka  -o jsonpath='{.items[0].metadata.name}'; }
rmq_pod()    { kubectl -n $NS get pod -l app=rabbitmq -o jsonpath='{.items[0].metadata.name}'; }

# Run SQL against MySQL pod
db_exec() { kubectl -n $NS exec -T "$(mysql_pod)" -- mysql -u appuser -papppass appdb -sNe "$1" 2>/dev/null; }

# Run a Kafka CLI command inside the kafka pod
kafka_exec() { kubectl -n $NS exec -T "$(kafka_pod)" -- "$@" 2>/dev/null; }

# Query a single outbox_delivery status
db_delivery_status() {
  local agg=$1 event_type=$2 dest=$3
  db_exec "SELECT d.status FROM outbox_events e
           JOIN outbox_deliveries d ON d.outbox_event_id = e.id
           WHERE e.aggregate_id='$agg' AND e.event_type='$event_type'
           AND d.destination='$dest'
           ORDER BY d.id DESC LIMIT 1;"
}

# Wait until a delivery reaches expected status
wait_for_delivery() { ... }

# Read combined worker logs (from both worker pods)
worker_logs() { kubectl -n $NS logs -l app=basesource-worker --tail=500 2>/dev/null; }

# Scale a deployment and wait
scale_deployment() {
  local name=$1 replicas=$2
  kubectl -n $NS scale deployment "$name" --replicas="$replicas"
  if [ "$replicas" -gt 0 ]; then
    kubectl -n $NS wait --for=condition=ready pod -l app="$name" --timeout=60s
  else
    # wait until all pods are gone
    kubectl -n $NS wait --for=delete pod -l app="$name" --timeout=60s 2>/dev/null || true
  fi
}

# Restart worker pods and wait for ready
restart_worker() {
  kubectl -n $NS rollout restart deployment basesource-worker
  kubectl -n $NS rollout status deployment basesource-worker --timeout=60s
}

# Port-forward helpers (start in background, cleanup on exit)
start_port_forward() { ... }
stop_port_forward()  { ... }
```

## reset.sh

```
1. Truncate DB: outbox_deliveries, outbox_events, audit_logs, todos
2. Delete Kafka consumer group todo-worker
3. Delete + recreate Kafka topic todo-events
4. Rollout restart basesource-worker (clears in-memory state)
5. Wait for 2 worker pods to be Ready
6. Verify consumer group has 1 active member (one partition ã one worker handles it)
```

No binary build step ã the image is already in kind.  
To test a new code change, rebuild outside the test suite:
```bash
docker build -f docker/Dockerfile -t basesource:local .
kind load docker-image basesource:local --name basesource
kubectl -n basesource rollout restart deployment basesource-api basesource-worker
```

### test_04_replay.sh
Mirrors `994.script-test-kafka/test_04_replay.sh`.

Steps:
1. Count messages in Kafka topic via `kafka_exec kafka-console-consumer.sh --from-beginning --max-messages 100 --timeout-ms 5000`
2. Delete consumer group `todo-worker-replay` (via `kafka_exec`)
3. Apply a temporary Deployment `basesource-worker-replay` with env override
   `KAFKA_GROUP_ID=todo-worker-replay`.

   > **Why not a Job?** `app worker` is a long-running daemon ã it never exits.
   > A Job expects the container to exit 0 when done. A Job running a worker would loop
   > forever, never reach `Completed`, and the test would timeout.
   > A temporary Deployment is the correct approach: apply it, read its logs, then delete it.

   The test generates the manifest inline with `kubectl apply -f -` (heredoc):
   ```bash
   kubectl apply -f - <<EOF
   apiVersion: apps/v1
   kind: Deployment
   metadata:
     name: basesource-worker-replay
     namespace: basesource
   spec:
     replicas: 1
     selector:
       matchLabels:
         app: basesource-worker-replay
     template:
       metadata:
         labels:
           app: basesource-worker-replay
       spec:
         securityContext:
           runAsNonRoot: true
           runAsUser: 65532
         containers:
           - name: worker
             image: basesource:local
             imagePullPolicy: Never
             args: ["worker"]
             env:
               - name: KAFKA_GROUP_ID
                 value: todo-worker-replay
             envFrom:
               - configMapRef:
                   name: basesource-config
               - secretRef:
                   name: basesource-secret
   EOF
   kubectl -n basesource wait --for=condition=ready pod -l app=basesource-worker-replay --timeout=30s
   ```

4. Wait for replayed domain event count ãË topic message count (from replay deployment logs):
   ```bash
   kubectl -n basesource logs -l app=basesource-worker-replay -f
   ```
5. Assert LAG=0 for `todo-worker-replay` group via `kafka_exec`
6. Delete the replay deployment:
   ```bash
   kubectl -n basesource delete deployment basesource-worker-replay
   ```

## Key differences from docker-compose tests

### No `/tmp/worker.log`
Worker logs are read via `kubectl logs`, not a local file. The helper `worker_logs()`
aggregates logs from all worker pods. This means:
- Log lines are interleaved from 2 pods ã patterns must not rely on ordering.
- Grep must search the combined output, not a file.
- For test_08, logs must be read per-pod separately.

### Worker restart is a rollout, not a `pkill`
```bash
# docker-compose:
pkill -TERM -f "$BINARY worker"
"$BINARY" worker >> "$WORKER_LOG" 2>&1 &

# Kubernetes:
restart_worker()  # = kubectl rollout restart + rollout status
```

### Stopping a service is a scale, not a compose stop
```bash
# docker-compose:
docker compose stop rabbitmq

# Kubernetes:
kubectl -n basesource scale deployment rabbitmq --replicas=0
kubectl -n basesource wait --for=delete pod -l app=rabbitmq --timeout=60s
```

### Consumer group replay needs a temporary Job
There is no way to `export KAFKA_GROUP_ID=... && kubectl exec` an existing worker pod.
The env var must be injected at pod creation time. A temporary Job is the cleanest approach.

### Port-forward must be managed in run_all.sh, not reset.sh
The API and RabbitMQ management API are only reachable via `kubectl port-forward`.

Port-forwards must be started in `run_all.sh` ã **not** in `reset.sh`.
If started in `reset.sh`, the `trap EXIT` fires when reset.sh exits and kills both
port-forwards before any test script runs. Every curl call in every test would fail.

`run_all.sh` starts both port-forwards before calling reset.sh and traps EXIT at the
suite level:

```bash
# run_all.sh ã start port-forwards once for the entire suite
kubectl -n basesource port-forward svc/basesource-api-svc 18080:80 &
PF_API_PID=$!
kubectl -n basesource port-forward svc/rabbitmq-service 15672:15672 &
PF_RMQ_PID=$!
trap "kill $PF_API_PID $PF_RMQ_PID 2>/dev/null || true" EXIT

sleep 2   # let port-forwards bind

bash "$DIR/reset.sh"
# ... run tests
```

`reset.sh` and each test script inherit the open ports ã they just use `http://localhost:18080`
and `http://localhost:15672` directly without managing the port-forward lifecycle.
