## Concepts

### Message Broker

A message broker is a middleware that decouples a **producer** (the code that sends a message) from a **consumer** (the code that receives and processes it). Instead of calling another service directly (tight coupling), the producer drops a message into the broker and moves on. The consumer picks it up independently.

Benefits:
- Producer does not wait for the consumer to finish (async)
- Consumer can be slow, crash, or restart without losing messages
- Multiple consumers can read the same message independently

---

### RabbitMQ Concepts

#### Exchange

An exchange receives messages from producers and routes them to queues based on rules. The producer never writes directly to a queue ‚Äî it publishes to an exchange with a **routing key**.

| Exchange type | Routing rule |
|---|---|
| `direct` | Route to queue whose binding key exactly matches the routing key |
| `topic` | Route using wildcard patterns (`*` = one word, `#` = zero or more words) |
| `fanout` | Broadcast to all bound queues, ignoring the routing key |
| `default` (empty name) | Route directly to the queue whose name matches the routing key |

In this project: `todo.events` is a **topic** exchange. Notification tasks use the **default** exchange (routing key = queue name).

#### Queue

A queue is a buffer that stores messages until a consumer reads them. Properties:

- **Durable** ‚Äî survives broker restart (messages written to disk)
- **Exclusive** ‚Äî only one consumer, deleted when consumer disconnects
- **Auto-delete** ‚Äî deleted when last consumer unsubscribes

In this project: `todo.notifications` is a durable queue.

#### Binding

A binding connects an exchange to a queue with a routing key pattern. When a message arrives at the exchange, RabbitMQ checks all bindings to decide which queues receive it.

#### Consumer (AMQP)

A consumer subscribes to a queue. Each message is delivered once to one consumer (competing consumers pattern). After processing:

- **ACK** ‚Äî message is removed from the queue (success)
- **NACK / Reject** ‚Äî message is returned to the queue or moved to dead-letter queue (failure)

#### Prefetch Count

Limits how many unacknowledged messages the broker pushes to a single consumer at once. Setting `prefetch=1` means the broker waits for an ACK before sending the next message ‚Äî prevents one slow consumer from hoarding the queue.

In this project: `RabbitMQPrefetchCount` is set in config.

#### Dead-Letter Queue (DLQ)

When a message is rejected (NACK without requeue) or expires, RabbitMQ can route it to a dead-letter exchange/queue instead of dropping it. Used for debugging failed messages.

#### Acknowledgement Model

```
Broker ‚Üí Consumer: delivers message
Consumer: processes message
Consumer ‚Üí Broker: ACK (success) or NACK (failure)
Broker: removes (ACK) or requeues/dead-letters (NACK)
```

If the consumer crashes before sending ACK, the broker redelivers the message to another consumer.

### Kafka Concepts

#### Topic

A topic is a named, append-only log of messages. Producers write to a topic; consumers read from it. Unlike RabbitMQ queues, messages are **not deleted after being consumed** ‚Äî they are retained for a configured retention period (default 7 days).

In this project: `todo-events` topic stores all domain events.

#### Partition

Each topic is split into one or more partitions. A partition is an ordered, immutable sequence of messages. Partitions enable:

- **Parallelism** ‚Äî multiple consumers in a group each read from different partitions simultaneously
- **Ordering** ‚Äî within a single partition, messages are strictly ordered by offset

In this project: `KAFKA_NUM_PARTITIONS: 3` ‚Äî the topic has 3 partitions.

#### Partition Key

When a producer writes a message with a key, Kafka hashes the key to decide which partition receives the message. The same key always goes to the same partition.

In this project: `aggregate_id` (todo ID) is the key. All events for the same todo land in the same partition ‚Üí strict ordering per entity is guaranteed.

```go
kafka.Message{
    Key:   []byte(event.AggregateID), // e.g. "42" ‚Üí always partition 1
    Value: body,
}
```

#### Offset

Each message in a partition has a sequential integer offset (0, 1, 2, ‚Ä¶). Consumers track their position by committing offsets. On restart, the consumer resumes from the last committed offset.

#### Consumer Group

A consumer group is a set of consumers that cooperate to consume a topic. Kafka assigns each partition to exactly one consumer in the group. This means:

- 3 partitions + 3 consumers ‚Üí each consumer owns 1 partition (maximum parallelism)
- 3 partitions + 1 consumer ‚Üí one consumer reads all 3 partitions
- 3 partitions + 5 consumers ‚Üí 2 consumers are idle (more consumers than partitions = waste)

In this project: group ID is `todo-worker`.

#### StartOffset

When a consumer group has no committed offset (first run, or group was deleted/reset), Kafka must decide where to start reading:

- `LastOffset` (default) ‚Äî start from the end, skip all existing messages
- `FirstOffset` ‚Äî start from the beginning, read all historical messages

In this project: `StartOffset: kafka.FirstOffset` so a fresh worker replays all past events.

#### Manual vs Auto Commit

- **Auto commit** ‚Äî Kafka commits the offset on a timer. Risk: message is committed before processing finishes; if the consumer crashes mid-process, the message is lost.
- **Manual commit** ‚Äî the consumer explicitly calls `CommitMessages` after successfully processing. Safer: if the consumer crashes, the message is redelivered.

In this project: manual commit is used (`FetchMessage` ‚Üí process ‚Üí `CommitMessages`).

#### Replication Factor

Each partition can be replicated across multiple brokers. `KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1` means no replication (single broker setup). In production, set to 3 for fault tolerance.

#### KRaft Mode

Traditional Kafka required ZooKeeper for cluster metadata and leader election. **KRaft** (Kafka Raft) embeds the controller directly in the Kafka broker ‚Äî no ZooKeeper needed.

In this project: `KAFKA_PROCESS_ROLES: broker,controller` ‚Äî single node running both roles.

#### RequiredAcks

Controls how many broker replicas must acknowledge a write before the producer considers it successful:

- `RequireNone` ‚Äî fire-and-forget, no acknowledgement
- `RequireOne` ‚Äî only the leader broker acknowledges
- `RequireAll` ‚Äî all in-sync replicas acknowledge (strongest durability guarantee)

In this project: `RequiredAcks: kafka.RequireAll`.

### Outbox Pattern

The outbox pattern solves the **dual-write problem**: how do you write to the database AND publish a message atomically?

**The problem:**
```
// Not atomic ‚Äî if Kafka publish fails, DB is already committed
db.Insert(todo)
kafka.Publish(event)  // ‚Üê could fail ‚Üí event lost
```

**The solution:**
```
// Atomic ‚Äî both succeed or both fail
tx.Insert(todo)
tx.Insert(outbox_event, status=pending)  // same transaction
tx.Commit()

// Separate relay process
poll outbox WHERE status=pending
kafka.Publish(event)
UPDATE outbox SET status=published
```

The database commit is the single source of truth. The relay retries until the message is published. No message is ever lost.

In this project: `OutboxRelay` polls every 2 seconds, publishes to Kafka, then marks `status=published`.

### Comparison: RabbitMQ vs Kafka

| | RabbitMQ | Kafka |
|---|---|---|
| **Model** | Message queue (push) | Distributed log (pull) |
| **Message retention** | Deleted after ACK | Retained by time/size policy |
| **Ordering** | Per queue (single consumer) | Per partition (by key) |
| **Replay** | Not possible after ACK | Yes ‚Äî reset consumer offset |
| **Consumer model** | Competing consumers (one gets each message) | Consumer groups (each group gets all messages) |
| **Best for** | Task queues, job dispatch, notifications | Event streaming, audit log, analytics, event sourcing |
| **Throughput** | Moderate | Very high |
| **Routing** | Exchange + binding rules (flexible) | Topic + partition key (simple) |

### When to use each in this project

| Use case | Broker | Reason |
|---|---|---|
| Send email / push notification | RabbitMQ | Task must happen once; no need to replay |
| Domain event audit log | Kafka | Must be durable and replayable |
| Analytics pipeline | Kafka | Multiple independent consumers can read the same event |
| Scheduled job dispatch | RabbitMQ | Work queue pattern, competing workers |
| Event sourcing / CQRS | Kafka | Replay from offset 0 to rebuild state |

## Test Case 7: Kafka Consumer Group Replay

Simulates a new downstream consumer or a reprocessing scenario.

```bash
# Kill worker
pkill -f basesource-996

# Reset consumer group offsets to earliest
docker compose -f docker/docker-compose.yaml exec kafka \
  /opt/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group todo-worker \
  --topic todo-events \
  --reset-offsets --to-earliest --execute
```

Expected:
```
GROUP        TOPIC        PARTITION  NEW-OFFSET
todo-worker  todo-events  0          0
todo-worker  todo-events  1          0
todo-worker  todo-events  2          0
```

```bash
# Restart worker
/tmp/basesource-996 worker > /tmp/996-worker.log 2>&1 &
sleep 5
cat /tmp/996-worker.log
```

Expected: all past `domain event received` entries replayed ‚Äî one per historical outbox event.

## Troubleshooting

### `notification task received` never appears

- Check RabbitMQ is healthy: `docker compose ps rabbitmq`
- Check exchange exists: `rabbitmqctl list_exchanges`
- Check queue binding: Management UI ‚Üí Queues ‚Üí `todo.notifications` ‚Üí Bindings

### `domain event received` never appears after first run

Consumer group already has committed offsets. Reset:
```bash
pkill -f basesource-996
docker compose -f docker/docker-compose.yaml exec kafka \
  /opt/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group todo-worker --topic todo-events \
  --reset-offsets --to-earliest --execute
```

### Offset reset fails ‚Äî "group is PreparingRebalance"

Worker is still running. Kill first:
```bash
pkill -f basesource-996
# then run reset command
```

### outbox_events stuck at `status=pending`

Kafka is unreachable. The relay retries automatically every 2s when Kafka recovers.
```bash
docker compose -f docker/docker-compose.yaml ps kafka
docker compose -f docker/docker-compose.yaml restart kafka
```
