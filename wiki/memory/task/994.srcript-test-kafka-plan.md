# 994 — Bash Script Test Suite for Kafka/RabbitMQ Real Integration

## Goal

Automate every manual step from `996.apply-Kafka-guide-test.md` as runnable bash
scripts so the full end-to-end test can be re-run with a single command on a clean
slate, with clear PASS/FAIL output for each assertion.

---

## Findings from manual run (input to script design)

| # | Finding | Mitigation in scripts |
|---|---|---|
| 1 | `status=failed` rows when Kafka broker is restarting | reset.sh truncates outbox_events; health-check waits before tests |
| 2 | Consumer only drains one partition after offset reset | reset.sh resets all 3 partitions; assertions use a poll loop not fixed sleep |
| 3 | `todo.updated` can land `failed` during broker churn | Each test creates its own todo; prior state does not leak |
| 4 | `domain event received` only appears for one partition at a time | `wait_for_log` polls with configurable timeout instead of `sleep 3` |
| 5 | Two worker PIDs hold group open → `PreparingRebalance` on offset reset | reset.sh kills all `basesource-*` and waits for group to reach `Empty` state |

## File layout

```
scripts/
  reset.sh           # Step 0 — wipe DB data + Kafka offsets, rebuild binary, start infra
  test_01_create.sh  # Test 1 — POST todo, assert outbox published + Kafka + RabbitMQ notification
  test_02_update.sh  # Test 2 — PUT todo, assert todo.updated in Kafka, no notification
  test_03_delete.sh  # Test 3 — DELETE todo, assert todo.deleted in Kafka, no notification
  test_04_replay.sh  # Test 4 — reset offsets, restart worker, assert all past events replayed
  run_all.sh         # Orchestrator: runs reset.sh then all test_*.sh; exits 1 on first failure
```

All scripts source a small inline library (`pass`, `fail`, `assert_contains`,
`wait_for_log`, `wait_for_db_status`).

---

## reset.sh — what it does

1. Kill any running `basesource-*` processes (ensures consumer group goes inactive)
2. Wait until `todo-worker` group state is `Empty` or not found (up to 30s)
3. `docker compose -f docker/docker-compose.yaml up -d db rabbitmq kafka`
4. Wait for all three services to be `healthy`
5. Truncate `todos`, `audit_logs`, `outbox_events` via mysql
6. Reset Kafka consumer group `todo-worker` offsets to `--to-earliest`
7. `go build -o /tmp/basesource ./...`
8. Start API (`/tmp/basesource api`) and worker (`/tmp/basesource worker`),
   redirect stdout+stderr to `/tmp/api.log` and `/tmp/worker.log`
9. Wait for `/tmp/worker.log` to contain `"outbox relay starting (Kafka)"` (up to 15s)

## test_01_create.sh — Create flow

```
GIVEN  DB is clean, worker is running
WHEN   POST /api/v1/todos {"title":"Test-01 Kafka create"}
THEN   HTTP 201, response body contains "id"
AND    outbox_events row for that aggregate_id → status=published  (within 8s)
AND    /tmp/worker.log contains "domain event received" event_type=todo.created
AND    /tmp/worker.log contains "notification task received"
AND    kafka-console-consumer --from-beginning returns ≥1 message with AggregateID
```

`TODO_ID` is extracted with `jq -r .id` from the curl response.

---

## test_02_update.sh — Update flow

```
GIVEN  Creates its own todo {"title":"Test-02 Kafka update"}
WHEN   PUT /api/v1/todos/$TODO_ID {"title":"Test-02 Updated","done":true}
THEN   HTTP 200, response body contains "done":true
AND    outbox_events row for todo.updated → status=published  (within 8s)
AND    /tmp/worker.log contains "domain event received" event_type=todo.updated
AND    /tmp/worker.log does NOT gain a new "notification task received" line
```

## test_03_delete.sh — Delete flow

```
GIVEN  Creates its own todo {"title":"Test-03 Kafka delete"}
WHEN   DELETE /api/v1/todos/$TODO_ID
THEN   HTTP 204
AND    outbox_events row for todo.deleted → status=published  (within 8s)
AND    /tmp/worker.log contains "domain event received" event_type=todo.deleted
AND    /tmp/worker.log does NOT gain a new "notification task received" line
```

## test_04_replay.sh — Consumer group replay

```
GIVEN  At least the 3 todos from tests 01–03 have Kafka messages
WHEN   pkill worker
       reset todo-worker offsets to --to-earliest
       restart worker
THEN   kafka-consumer-groups --describe shows all partition LAG=0 (within 30s)
AND    count of "domain event received" lines in /tmp/worker.log after restart
       >= count of messages in Kafka topic (verified via kafka-console-consumer)
```

The count check polls for up to 30s at 1s intervals to account for partition fan-out lag.

## run_all.sh — orchestrator

```bash
#!/usr/bin/env bash
set -euo pipefail
DIR="$(cd "$(dirname "$0")" && pwd)"
bash "$DIR/reset.sh"
bash "$DIR/test_01_create.sh"
bash "$DIR/test_02_update.sh"
bash "$DIR/test_03_delete.sh"
bash "$DIR/test_04_replay.sh"
echo ""
echo "========================================="
echo "  ALL TESTS PASSED"
echo "========================================="
```

`set -euo pipefail` exits immediately on any test failure so the first broken
assertion surfaces clearly.

## Assertion helpers (inlined in each script)

```bash
pass()  { echo "  ✅  $*"; }
fail()  { echo "  ❌  $*"; exit 1; }

assert_contains() {
  local label=$1 haystack=$2 needle=$3
  if echo "$haystack" | grep -q "$needle"
  then pass "$label"
  else fail "$label — expected to find: $needle"
  fi
}

# Poll a log file until pattern appears or timeout
wait_for_log() {
  local file=$1 pattern=$2 timeout=${3:-10}
  local i=0
  while ! grep -q "$pattern" "$file" 2>/dev/null; do
    [ $i -ge $timeout ] && fail "timeout (${timeout}s) waiting for '$pattern' in $file"
    sleep 1; i=$((i+1))
  done
  pass "log contains: $pattern"
}

# Poll DB until outbox row for aggregate_id reaches expected status
wait_for_db_status() {
  local agg=$1 expected=$2 timeout=${3:-8}
  local i=0
  while true; do
    local got
    got=$(docker compose -f docker/docker-compose.yaml exec -T db \
      mysql -u appuser -papppass appdb -sNe \
      "SELECT status FROM outbox_events WHERE aggregate_id='$agg' AND event_type LIKE '%' ORDER BY id DESC LIMIT 1;" 2>/dev/null || true)
    [ "$got" = "$expected" ] && pass "outbox[$agg] status=$expected" && return
    [ $i -ge $timeout ] && fail "outbox[$agg] status: want $expected, got '$got'"
    sleep 1; i=$((i+1))
  done
}
```

## Known edge cases to handle in scripts

- `kafka-console-consumer.sh` always exits non-zero (TimeoutException) — capture
  output with `|| true`, then grep the captured variable
- `todo-worker` group shows `PreparingRebalance` briefly after kill —
  reset.sh must loop until `--describe` returns no CONSUMER-ID column (group inactive)
- `outbox_events` rows with `status=failed` from a prior crashed run must be
  cleared by the truncate in reset.sh so they don't pollute assertions
- Log file carries output from multiple worker restarts — `test_04_replay.sh`
  must count only lines appended *after* the restart timestamp, so it uses
  `wc -l` before/after to measure the delta
- `wait_for_db_status` must match by `event_type` when a todo has multiple
  outbox rows (created + updated + deleted all share the same aggregate_id)
