# 02 — Kubernetes Deploy & Scale

---

## Deploy a new version

```bash
# 1. Build new image
docker build -f docker/Dockerfile -t basesource:v1.2.3 .

# 2. Load into kind cluster (local) — skip in production, push to registry instead
kind load docker-image basesource:v1.2.3 --name basesource

# 3. Update deployment image
kubectl set image deployment/basesource-api api=basesource:v1.2.3 -n basesource
kubectl set image deployment/basesource-worker worker=basesource:v1.2.3 -n basesource

# 4. Watch rollout
kubectl rollout status deployment/basesource-api -n basesource
kubectl rollout status deployment/basesource-worker -n basesource
```

### How to read rollout output

```
Waiting for deployment "basesource-api" rollout to finish: 1 out of 2 new replicas have been updated...
Waiting for deployment "basesource-api" rollout to finish: 1 old replicas are pending termination...
deployment "basesource-api" successfully rolled out
```

- K8s spins up 1 new pod (maxSurge:1), waits for readiness, then kills 1 old pod
- If readiness probe fails → rollout pauses, old pods keep serving traffic → safe

### Rollback if something is wrong

```bash
kubectl rollout undo deployment/basesource-api -n basesource
kubectl rollout undo deployment/basesource-worker -n basesource

# Check rollout history
kubectl rollout history deployment/basesource-api -n basesource
```

## Run database migration after deploy

```bash
# Delete old job (K8s doesn't allow re-applying a completed Job)
kubectl delete job basesource-migrate -n basesource

# Apply and wait
kubectl apply -f k8s/migrate-job.yaml
kubectl wait --for=condition=complete job/basesource-migrate -n basesource --timeout=60s

# View migration output
kubectl logs -n basesource job/basesource-migrate
```

## Manual scaling

```bash
# Scale API to 4 replicas (HPA will override back after cooldown)
kubectl scale deployment basesource-api -n basesource --replicas=4

# Watch pods come up
kubectl get pods -n basesource -l app=basesource-api -w
```

### When to scale manually

- Pre-scale before a known traffic spike (marketing campaign, product launch)
- After HPA fails to scale fast enough
- Debug: reduce to 1 replica to simplify log reading

---

## Watch HPA scale in real time

```bash
# Watch HPA every 5 seconds
watch -n 5 kubectl get hpa -n basesource
```

### How to read

```
NAME                 TARGETS      REPLICAS
basesource-api-hpa   cpu: 85%/70%  3         ← scaling up (85 > 70)
basesource-api-hpa   cpu: 45%/70%  3         ← cooling down, will scale back to 2
basesource-api-hpa   cpu: 20%/70%  2         ← scaled back to min
```

- Scale-up is fast (immediate when threshold crossed)
- Scale-down has 5-minute cooldown to prevent flapping

## Describe a deployment for full details

```bash
kubectl describe deployment basesource-api -n basesource
```

Key sections to look at:

| Section | What to check |
|---|---|
| `Replicas` | desired vs available |
| `StrategyType` | should be `RollingUpdate` |
| `Conditions` | `Available True` and `Progressing True` |
| `Events` | recent scaling events, failed probes |

## Zero-downtime deploy — how it works

```
API: maxUnavailable=0, maxSurge=1

Before:   [pod-A] [pod-B]
Step 1:   [pod-A] [pod-B] [pod-C-new]  ← new pod starts, readiness probe runs
Step 2:   [pod-A] [pod-B-terminating] [pod-C-new]  ← old pod gets SIGTERM
Step 3:   [pod-A] [pod-C-new]
Step 4:   [pod-A-terminating] [pod-C-new] [pod-D-new]
Step 5:   [pod-C-new] [pod-D-new]  ← done
```

Traffic is always routed only to pods that passed the readiness probe.
Old pod drains in-flight requests during terminationGracePeriodSeconds (30s).
