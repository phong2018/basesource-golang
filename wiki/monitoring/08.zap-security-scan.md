# 08 — ZAP Security Scan

OWASP ZAP scans the API for security vulnerabilities. Run after every significant feature
deployment, especially changes to auth, input handling, or new endpoints.

---

## Run the scan

```bash
cd /Users/phongle/sources/freelancer/basesource/basesource-golang/zap/scripts
./zap_scan.sh
```

Prerequisites:
- API running at `localhost:8080` (local binary or port-forward)
- Docker running (ZAP runs in a container)

---

## What the scan does (3 phases)

### Phase 1 — Baseline passive scan (unauthenticated)

No attack payloads. ZAP spiders the API and checks:
- Missing security headers (X-Frame-Options, CSP, HSTS)
- Information disclosure (server version, stack traces)
- Session tokens in URLs
- Insecure cookies

Takes ~30 seconds.

### Phase 2 — API scan (authenticated, active)

Uses `zap/openapi.yaml` to find every endpoint. ZAP attacks each one with:
- SQL injection payloads (`'`, `" OR 1=1`, etc.)
- XSS payloads
- Path traversal
- Server-side template injection

Takes ~3-5 minutes.

### Phase 3 — Manual access control checks

Script-driven checks for business logic flaws:
- No token → 401?
- Tampered JWT → 401?
- Cross-ownership (admin accessing user's todo) → 404?
- User calling admin endpoints → 403?

## Reading the scan output

```
PASS: XPath Injection [90021]         ← not vulnerable
WARN-NEW: SQL Injection [40018] x 5   ← potential issue, investigate
FAIL-NEW: 0                           ← no hard failures
```

| Status | Meaning |
|---|---|
| `PASS` | ZAP tested this and found no issue |
| `WARN-NEW` | Possible vulnerability — review the URLs listed |
| `FAIL-NEW` | Definite vulnerability — must fix before deploy |
| `INFO` | Informational, not a risk |

**WARN vs FAIL:** WARN means ZAP saw a suspicious response (e.g. 500 error) but isn't certain it's exploitable. FAIL means ZAP confirmed the attack worked.

---

## Reading WARN: SQL Injection findings

```
WARN-NEW: SQL Injection - MySQL [40018] x 5
  http://host.docker.internal:8080/api/v1/my/todos?condition=condition (500)
  http://host.docker.internal:8080/api/v1/my/todos/10/field?field=field (500)
```

These 500 errors on `condition` and `field` params are **intentional** in this project
(planted vulnerabilities for demo purposes). In a real system, any 500 from a SQL
injection probe is a critical finding.

To verify if it's real:
```bash
# Does this return data it shouldn't?
curl "http://localhost:8080/api/v1/my/todos?condition=1%3D1" \
  -H "Authorization: Bearer $USER_TOKEN"

# Does this crash the server?
curl "http://localhost:8080/api/v1/my/todos?condition=%27" \
  -H "Authorization: Bearer $USER_TOKEN"
```


Each finding includes:

## When to run

| Event | Run scan? |
|---|---|
| New auth endpoint added | Yes — full scan |
| New CRUD endpoint | Yes — full scan |
| Config-only change | No |
| Dependency upgrade | Yes — passive scan minimum |
| Weekly routine | Yes — full scan |

## Add a new manual check

Edit `zap/scripts/zap_scan.sh`, add after the existing checks:

```bash
# New check: unauthenticated DELETE should return 401
SC=$(curl -s -o /dev/null -w "%{http_code}" -X DELETE "$API/api/v1/my/todos/1")
check "Unauthenticated DELETE → 401" "401" "$SC"
```
