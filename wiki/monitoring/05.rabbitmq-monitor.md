# 05 — RabbitMQ Monitor

---

## Management UI (easiest way)

```bash
kubectl port-forward -n basesource pod/rabbitmq-0 15672:15672
```

Open: http://localhost:15672
Login: `guest` / `guest`

The UI shows queues, message rates, consumers, memory, disk in real time.

---

## CLI commands inside the pod

```bash
# Health check
kubectl exec -n basesource rabbitmq-0 -- rabbitmq-diagnostics ping

# List queues with message count and consumers
kubectl exec -n basesource rabbitmq-0 -- \
  rabbitmqctl list_queues name messages consumers

# List exchanges
kubectl exec -n basesource rabbitmq-0 -- rabbitmqctl list_exchanges

# List bindings (exchange → queue routing)
kubectl exec -n basesource rabbitmq-0 -- rabbitmqctl list_bindings
```

## Reading queue status

```bash
kubectl exec -n basesource rabbitmq-0 -- \
  rabbitmqctl list_queues name messages consumers 2>/dev/null
```

### How to read

```
name                  messages  consumers
todo.notifications    0         2          ← healthy: consumed as fast as produced
todo.notifications    150       2          ← backlog building — consumers slow
todo.notifications    5000      0          ← NO consumers — worker is down
```

| messages | consumers | Status                                    |
| -------- | --------- | ----------------------------------------- |
| ~0       | 1+        | Healthy                                   |
| Growing  | 1+        | Consumer is slow — check worker logs     |
| Any      | 0         | Workers disconnected — check worker pods |
| 0        | 0         | Queue idle — fine if no traffic          |

## Message rates (via management HTTP API)

`rabbitmqctl list_queues` does not support `message_stats.*` columns.
Use the management API instead (port-forward required):

```bash
kubectl port-forward -n basesource pod/rabbitmq-0 15672:15672

# Get queue stats with rates in JSON
curl -s -u guest:guest \
  "http://localhost:15672/api/queues/%2F/todo.notifications" | \
  python3 -c "
import sys, json
q = json.load(sys.stdin)
ms = q.get('message_stats', {})
print('publish rate:', ms.get('publish_details', {}).get('rate', 0), '/s')
print('deliver rate:', ms.get('deliver_details', {}).get('rate', 0), '/s')
print('messages ready:', q.get('messages_ready', 0))
"
```

### How to read

```
publish rate: 2.3 /s
deliver rate: 2.3 /s    ← matched — healthy
messages ready: 0

publish rate: 5.0 /s
deliver rate: 0.5 /s    ← delivering slower than publishing — backlog growing
messages ready: 150
```

## Check node resource usage

```bash
kubectl exec -n basesource rabbitmq-0 -- \
  rabbitmqctl status 2>/dev/null | grep -E "memory|disk|sockets|processes"
```

### How to read

```
{memory,[{total,{bytes,89478144}}...]}     ← ~85MB used
{disk_free_limit,{absolute,"50MB"}}        ← RabbitMQ will block when disk < 50MB
{disk_free,{absolute,8123456789}}          ← ~8GB free — OK
```

**Alert when:**

- Memory > 80% of pod limit (256Mi)
- `disk_free` < 500MB — RabbitMQ will stop accepting new messages

## Durable queue check (after pod restart)

RabbitMQ StatefulSet + PVC means queues and messages survive pod restarts.
Verify after any restart:

```bash
kubectl exec -n basesource rabbitmq-0 -- \
  rabbitmqctl list_queues name durable messages 2>/dev/null
```

```
name                durable  messages
todo.notifications  true     0        ← durable=true means it survives restart
```

## RabbitMQ logs

```bash
kubectl logs -n basesource rabbitmq-0 -f | grep -E "error|warning|accepting|started"
```

Healthy startup:

```
Starting RabbitMQ 3.13.x ...
completed with 4 plugins
started TCP listener on [::]:5672
```

Problem signs:

```
Error: disk_free_limit exceeded
Error: cannot connect to node rabbit@...   ← node name resolution failed
```
