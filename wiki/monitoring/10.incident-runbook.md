# 10 — Incident Runbook

Quick decision tree for when something breaks in the simulated production environment.

---

## Pod in CrashLoopBackOff

```bash
# Step 1 — identify which pod
kubectl get pods -n basesource | grep CrashLoop

# Step 2 — read the crash logs (last run)
kubectl logs -n basesource <pod-name> --previous

# Step 3 — describe for events
kubectl describe pod -n basesource <pod-name> | tail -20
```

### Common causes and fixes

| Exit code / log | Cause | Fix |
|---|---|---|
| `Access denied for user 'appuser'` | Wrong DB password in secret | Check `secret.local.yaml`, reapply, restart pod |
| `rabbitmq dial: Exception 403` | Wrong RabbitMQ credentials or user doesn't exist | Check RabbitMQ users; reapply secret |
| `Error 1146: Table doesn't exist` | Migrations not run | Delete + re-apply `migrate-job.yaml` |
| `signal: killed` | OOM — memory limit exceeded | Increase memory limit in deployment yaml |
| `connection refused` | Dependency not ready | Check mysql-0/kafka-0/rabbitmq-0 are Running |

## Pod OOMKilled

```bash
kubectl describe pod -n basesource <pod-name> | grep -A3 "OOMKilled\|Last State"
```

```
Last State: Terminated
  Reason: OOMKilled
  Exit Code: 137
```

Fix:
```bash
# Increase memory limit in the deployment
kubectl edit deployment basesource-api -n basesource
# Change: limits.memory: 256Mi → 512Mi
```

## Kafka consumer lag spike

```bash
kubectl exec -n basesource kafka-0 -- \
  /opt/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 --describe --group todo-worker 2>/dev/null
```

**LAG > 500 and growing:**

1. Check worker pods are running:
   ```bash
   kubectl get pods -n basesource -l app=basesource-worker
   ```
2. Check worker logs for errors:
   ```bash
   kubectl logs -n basesource -l app=basesource-worker --tail=50
   ```
3. If pods are crashing → fix crash first (see CrashLoopBackOff above)
4. If pods are running but lag grows → scale workers:
   ```bash
   kubectl scale deployment basesource-worker -n basesource --replicas=4
   ```
   Note: max effective Kafka consumers = number of partitions (3). Scaling beyond 3 workers doesn't help Kafka consumption but helps RabbitMQ and outbox relay.

## RabbitMQ queue backlog

```bash
kubectl exec -n basesource rabbitmq-0 -- \
  rabbitmqctl list_queues name messages consumers 2>/dev/null
```

**messages > 1000 and consumers = 0:**
→ Workers are disconnected. Restart worker deployment:
```bash
kubectl rollout restart deployment/basesource-worker -n basesource
```

**messages growing and consumers > 0:**
→ Workers are too slow. Scale up:
```bash
kubectl scale deployment basesource-worker -n basesource --replicas=4
```

## MySQL disk full

```bash
kubectl exec -n basesource mysql-0 -- df -h /var/lib/mysql
```

**Use% > 85%:**

Option 1 — Archive old outbox records (safe):
```bash
kubectl exec -n basesource mysql-0 -- mysql -u appuser -papppass appdb -e "
DELETE FROM outbox_deliveries
WHERE status = 'delivered'
  AND created_at < NOW() - INTERVAL 30 DAY;
OPTIMIZE TABLE outbox_deliveries;"
```

Option 2 — Expand PVC (requires StorageClass that supports expansion):
```bash
kubectl patch pvc mysql-data-mysql-0 -n basesource \
  -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'
```

## HPA not scaling

```bash
kubectl describe hpa basesource-api-hpa -n basesource | grep -A5 "Conditions\|Events"
```

Common causes:
- `metrics-server` not installed → `kubectl top pods` fails
- CPU requests not set on pod → HPA has no baseline to calculate from
- `minReplicas = maxReplicas` → autoscaling disabled

## Rolling update stuck

```bash
kubectl rollout status deployment/basesource-api -n basesource
# Hangs at: "Waiting for rollout to finish: 1 old replicas are pending termination"
```

Cause: new pods fail readiness probe — old pods not replaced.

```bash
# Check new pod status
kubectl get pods -n basesource -l app=basesource-api
kubectl logs -n basesource <new-crashing-pod>

# If new image is bad — rollback immediately
kubectl rollout undo deployment/basesource-api -n basesource
```

## Fluent Bit not collecting logs

```bash
kubectl get pod -n basesource -l app=fluent-bit
kubectl logs -n basesource -l app=fluent-bit | grep -v "^{\"date\""
```

Common causes:
- Log path filter too narrow — no files match `*basesource*.log`
- `Mem_Buf_Limit` exceeded — increase limit in ConfigMap
- Container runtime uses a different log format (cri-o vs docker)

Fix:
```bash
kubectl edit configmap fluent-bit-config -n basesource
kubectl rollout restart daemonset/fluent-bit -n basesource
```

## Quick triage checklist (copy-paste this when incident starts)

```bash
# 1. Pod status
kubectl get pods -n basesource

# 2. Recent events
kubectl get events -n basesource --sort-by='.lastTimestamp' | tail -20

# 3. HPA status
kubectl get hpa -n basesource

# 4. Outbox backlog
kubectl exec -n basesource mysql-0 -- mysql -u appuser -papppass appdb \
  -e "SELECT destination, status, COUNT(*) FROM outbox_deliveries GROUP BY 1,2;" 2>/dev/null

# 5. Kafka lag
kubectl exec -n basesource kafka-0 -- \
  /opt/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 --describe --group todo-worker 2>/dev/null

# 6. RabbitMQ queues
kubectl exec -n basesource rabbitmq-0 -- \
  rabbitmqctl list_queues name messages consumers 2>/dev/null

# 7. Disk usage
kubectl exec -n basesource mysql-0 -- df -h /var/lib/mysql
kubectl exec -n basesource kafka-0 -- df -h /var/lib/kafka/data
```
