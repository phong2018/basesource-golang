# 09 — K6 Performance Test

K6 load tests measure how the API behaves under sustained traffic.
Run before a production release or after infra changes.

---

## Run the tests

```bash
cd wiki/memory/task/993.performance-test/scripts

# 1. Smoke test — 1 user, 30s — verify basic functionality
k6 run smoke_test.js

# 2. Load test — ramp up to 50 users over 5 minutes — normal load
k6 run load_test.js

# 3. Stress test — ramp up to 200 users — find breaking point
k6 run stress_test.js

# 4. Run all in sequence
./run_all.sh
```

Prerequisites: `k6` installed (`brew install k6`), API running at `localhost:8080`.

### How to read

| Metric | What it means |
|---|---|
| `checks 100%` | All assertions passed |
| `http_req_duration avg=23ms` | Average response time |
| `p(95)=89ms` | 95% of requests completed in under 89ms — the key SLA metric |
| `http_req_failed 0%` | No errors — good |
| `iterations 1.67/s` | Throughput — requests per second |

**Smoke test goal:** p(95) < 500ms, 0% errors.

### How to read

| Metric | Green | Yellow | Red |
|---|---|---|---|
| `checks` | > 99% | 95–99% | < 95% |
| `p(95) duration` | < 500ms | 500ms–1s | > 1s |
| `http_req_failed` | < 1% | 1–5% | > 5% |
| `avg duration` | < 200ms | 200–500ms | > 500ms |

### Correlate with K8s during the test

```bash
# In a separate terminal — watch HPA scaling during load test
watch -n 3 kubectl get hpa -n basesource

# Watch pods scale up
watch -n 3 kubectl get pods -n basesource -l app=basesource-api
```

You should see replicas increase from 2 → 3 → 4 as CPU climbs past 70%.

## Reading stress test output

```
http_req_duration..............: avg=892ms  p(95)=4201ms  p(99)=8934ms
http_req_failed................: 18.30%
```

Stress test intentionally pushes past limits. Look for:

1. **At what VU count did p(95) exceed 1s?** — that's your capacity threshold
2. **At what VU count did error rate spike?** — that's your breaking point
3. **Did the system recover** after load dropped? — check if pods went back to min replicas

## Interpret HPA behaviour during test

```
Before test:  cpu: 1%/70%   REPLICAS: 2
During load:  cpu: 78%/70%  REPLICAS: 3   ← HPA scaled up
Peak load:    cpu: 85%/70%  REPLICAS: 5
After test:   cpu: 5%/70%   REPLICAS: 2   ← scale back after cooldown (5 min)
```

If REPLICAS hits MAXPODS (10) and p(95) is still high:
→ The bottleneck is not CPU — it's likely MySQL or the app code.

---

## Key questions to answer after each run

1. What is the **maximum RPS** the system handles at p(95) < 500ms?
2. At what **VU count** do errors start appearing?
3. Does **HPA scale correctly** under load?
4. Does **MySQL become the bottleneck** (slow outbox queries)?
5. Does the system **recover cleanly** after load drops?
